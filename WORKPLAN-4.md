# WORKPLAN 4차 — 아키텍처 정렬·인프라

랄프 루프가 이 파일을 진행 상태 저장소로 사용한다. 각 단계를 완료할 때마다 해당 체크박스를 `[x]`로 갱신하고 파일을 저장한다.

## 진행 방법

- 작업은 위에서부터 순서대로, 한 번에 하나만 진행한다. 이 차수는 순서 의존이 있다: 아웃박스 릴레이가 app-batch에 살아야 하므로 1 → 2 → 3 순서를 바꾸지 않는다.
- 한 이터레이션은 한 작업 안에서만 진행한다. 한 작업의 연속된 단계 여러 개를 끝내도 되지만, 작업 경계를 넘어 다음 작업을 시작하지 않는다.
- 각 작업은 아래 5단계를 순서대로 거친다.

| 단계 | 내용 | 산출물·검증 |
|---|---|---|
| 1. 설계 | `docs/architecture.md`, `docs/coding-conventions.md`, `docs/entity-persistence.md` 중 해당 문서를 로딩한다. 가정·해석 후보·트레이드오프를 명시하고, 완료 기준을 검증 가능한 테스트 목록으로 구체화한다. | `worklog/4-<번호>-<작업명>.md`에 설계 기록 |
| 2. 설계 리뷰 | 설계를 `AGENTS.md` 작업 원칙(단순함 우선, 외과적 수정, 목표 주도)과 `docs/` 규칙에 대조해 리뷰하고 지적 사항을 설계에 반영한다. | worklog 파일에 리뷰 결과·반영 내역 추가 |
| 3. 구현 | 이전(移轉) 작업은 "전후로 테스트가 통과하는지 보장한다"를 기준으로 진행한다. 신규 동작은 TDD로 진행한다. | 테스트 전부 통과 |
| 4. 코드 리뷰 | 변경 diff를 검토한다. 바뀐 모든 줄이 이 작업으로 추적되는지, `docs/` 규칙 위반이 없는지, 이전 후 잔재(고아 설정·데드 코드)가 없는지 확인하고 수정한다. | worklog 파일에 리뷰 결과 추가 |
| 5. 게이트·커밋 | `./gradlew build` 통과를 확인하고 작업 단위로 커밋한다(구조 이전은 `refactor: ...`, 신규 동작은 `feat: ...`). | build 성공, 커밋 완료 |

- 막히면 worklog 파일에 시도한 것과 막힌 지점을 기록하고 이터레이션을 종료한다. 같은 단계에서 3회 연속 실패하면 작업 상태에 `중단`을 표기하고 사유를 기록한 뒤 다음 작업으로 넘어간다. 단, 작업 3(아웃박스)은 작업 1의 완료를 전제하므로 작업 1이 중단되면 작업 3도 중단 처리한다.
- 설계 단계에서 결정이 필요한 항목은 worklog에 선택지와 결정·근거를 남긴다. 사용자에게 되묻지 않고 더 단순한 쪽을 택한다.

## 작업 목록

### 1. app-batch 분리

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 스케줄러(미결 주문 스윕 등)를 `app-batch` 앱으로 이전해 `docs/architecture.md`의 앱 4종 구조와 실제를 일치시킨다.
- 배경: `PendingOrderSweepFacade`·`SchedulingConfig`·스케줄러 락이 app-api에서 돌고 있다.
- 범위: `module-apps/app-batch` 모듈 신설(`convention.app-module`), 스윕 잡·스케줄링 설정·락 이전, 배치 전용 읽기가 필요하면 `infrastructure/reader` 격리 구역 사용. app-api에서 스케줄링 잔재를 제거한다. 배포 편의 옵션(프로파일 등)은 넣지 않는다 — 요청된 분리만 한다.
- 완료 기준:
  - 이전된 스윕 관련 테스트가 app-batch에서 전부 통과.
  - app-api에 `@EnableScheduling`·스케줄러 관련 코드가 남지 않음.
  - 아키텍처 테스트(모듈 경계) 통과.
- 참고: `docs/architecture.md` app-batch 절, `PendingOrderSweepFacade`, `SchedulingConfig`, `SchedulerLockTest`.

### 2. app-admin 분리

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: admin 컨트롤러 전부를 `app-admin` 앱으로 이전한다.
- 배경: 어드민 표면(`web/v1/admin/**`)이 app-api 안에 있다. 문서는 별도 앱과 `infrastructure/query` 격리 구역을 정의한다.
- 범위: `module-apps/app-admin` 모듈 신설, admin 슬라이스(컨트롤러·request/response·테스트) 이전, 관리자 전용 SecurityConfig·시딩 이전. 공유 응답 DTO는 "본인/공개 슬라이스 소유, 어드민 참조" 규칙을 적용해 정리한다. 크로스 도메인 조회가 필요한 곳만 `infrastructure/query`를 도입한다 — 기존 Reader 경유로 충분하면 그대로 둔다.
- 완료 기준:
  - 이전된 어드민 테스트 전부 app-admin에서 통과.
  - app-api에 admin 표면(패키지·경로·시딩)이 남지 않음.
  - 아키텍처 테스트(어드민 표면 4중 규칙 포함) 통과.
- 참고: `docs/architecture.md` app-admin 절·앱 모듈 공통, `AdminSeedConfig`, `web/auth/Admin`.

### 3. 트랜잭션 아웃박스

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 도메인 이벤트 발행을 아웃박스 경유로 전환해 커밋과 발행의 원자성을 보장한다.
- 배경: 이벤트가 인프로세스로 발행되어 커밋-발행 사이 유실 가능성이 있다.
- 범위: 아웃박스 저장은 common-messaging/infra-messaging 축에, 릴레이(폴링 잡)는 app-batch에 둔다. 기존 발행 경로(주문 결제 완료 등)를 아웃박스 경유로 전환하고 소비 멱등성을 유지한다. 외부 브로커(Kafka 등) 도입은 범위 제외 — 전달 방식은 유지하고 영속 단계만 앞에 세운다.
- 완료 기준:
  - 트랜잭션 롤백 시 이벤트가 발행되지 않는 테스트 통과.
  - 커밋 후 릴레이가 발행하고 소비되는 테스트 통과.
  - 릴레이 중복 실행 시 중복 부작용이 없는(멱등 소비) 테스트 통과.
- 참고: `docs/architecture.md` 트랜잭션 경계·도메인 이벤트 절, `OrderPaidListener`, common-messaging, 작업 1의 app-batch.
