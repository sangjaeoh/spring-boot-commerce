# 커머스 요구사항 (Requirements)

Spring Boot 기반 커머스 최소 구현. 회원이 상품을 담고 주문·결제하는 핵심 거래 흐름의 요구사항(*무엇을 만드는가*)을 정의한다. 실제 구현 모델(도메인·엔티티·필드·정책·불변식)은 [`DOMAIN_MODEL.md`](./DOMAIN_MODEL.md)가 소유한다.

- **대상 사용자**: 구매 회원 · 서비스 운영 관리자
- **기준 방향**: 모듈러 모놀리스 — 7개 도메인을 한 앱에 두되 도메인 간은 ID 값 참조 + 앱 계층(파사드) 조율로만 잇는다(물리 FK·크로스 도메인 컴파일 의존 없음). 결제는 벤더 중립 포트로 격리한 **동기 승인 fake PG**(실패·응답 유실을 결정론적으로 시뮬레이션)이며, 응답 유실로 미확정에 남은 결제는 **주기 리컨실·웹훅 확정 경로**가 PG 거래 상태 조회로 확정한다. 도메인 이벤트는 **in-process**(무손실 보장 아님). 인증은 이메일+패스워드 로그인·JWT 액세스 토큰 발급, 구매자 셀프서비스 표면의 토큰 주체 강제·소유권 검사, 역할(BUYER/ADMIN) 기반 관리자 오퍼레이션 가드까지 포함한다.

## 범위

- **포함**: 회원(가입·정지·탈퇴·역할), 인증(가입 패스워드 자격증명·이메일+패스워드 로그인·JWT 액세스 토큰 발급·구매자 셀프서비스의 토큰 주체 강제·소유권 검사·관리자 오퍼레이션 역할 가드), 상품 카탈로그·변형(SKU)·옵션, 재고(차감·복원·재입고), 장바구니, 쿠폰(발급·발급 한도·할인·사용·발급분 관리자 무효화), 주문(체크아웃·결제 상태·이행 상태·출고 시 택배사·운송장 번호 기록·취소·배송 완료 주문의 전체 반품 환불), 결제(승인·실패·환불·미확정 결제 리컨실·웹훅 확정), 체크아웃·취소·반품 환불 크로스 도메인 조율.
- **제외**: 소셜 로그인·리프레시 토큰·비밀번호 재설정·이메일 인증, 세분화된 퍼미션·권한 매트릭스(역할은 BUYER/ADMIN 이분법), 관리자 계정 관리 API·역할 변경 오퍼레이션(관리자는 설정 기반 기동 시딩으로만 생성), 관리자 전용 앱 분리, 실 PG 연동(fake PG 교체), 배송 추적·택배사 API 연동·운송장 수정(출고·배송완료 상태와 출고 시 택배사·운송장 번호 기록 자체는 포함), 부분 취소·부분 환불·부분 선택 주문, 분할 배송·부분 이행, 부분 반품·교환·반품 요청/승인 워크플로·반품 회수 배송 추적(배송 완료 주문의 전체 반품 환불 자체는 포함), 사용된(USED) 쿠폰의 소급 회수·주문 금액 재계산(무효화는 미사용 발급분만 대상), 회원 주소록, 상인용 SKU 코드, 사용자 개시 미결제 주문 취소 경로.

## 용어

| 용어 | 정의 |
|---|---|
| 변형(SKU) | 상품의 판매·재고 단위. 판매가·옵션 조합을 소유 |
| 옵션 시그니처 / 라벨 | 변형 옵션 조합의 정규화 유니크 키 / 사람이 읽는 표시 |
| 자격 활성 회원 | 담기·주문·발급 자격이 있는 회원(활성 상태이고 미탈퇴) |
| 발급분 | 회원에게 발급된 쿠폰 1건 |
| 사용 기한 | 발급분의 사용 만료(발급시각 + 정책 사용창). 정책의 발급 가능 기간과 별개 |
| 이행(Fulfillment) | 주문의 물리 배송 진행 축(시작 전·준비·출고·배송완료·보류) |
| 스냅샷 | 주문 시점 값을 복사해 이후 원본 변경과 무관하게 보존 |
| 체크아웃 | 장바구니를 주문·결제로 전환하는 앱 계층 흐름 |
| 오버셀 | 재고보다 많이 판매되는 것 |
| 사가 앵커 | 체크아웃의 모든 부작용이 걸리는 미결제(생성 직후) 주문 |

## 기능 요구사항

### 회원

- 이메일·이름·패스워드로 가입하고, 가입 즉시 담기·주문할 수 있다(별도 활성화 절차 없음).
- 활성 회원 사이 이메일은 유일하되, 탈퇴한 이메일은 재가입할 수 있다.
- 표시 이름을 변경할 수 있다(이메일은 불변).
- 탈퇴(사유 기록)를 제공하고, 탈퇴 회원은 기본 조회에서 제외한다.
- 관리자가 회원을 정지·해제(사유 기록)할 수 있다. 정지는 탈퇴와 독립 축이라 정지된 채 탈퇴·탈퇴 없이 정지가 모두 가능하다.
- 담기·주문·쿠폰 발급은 자격 활성(활성 ∧ 미탈퇴) 회원만 가능하다.
- 정지·탈퇴는 회원의 장바구니·발급 쿠폰·미결제 주문을 연쇄 정리하지 않고, 다음 담기·체크아웃 자격 검증에서만 차단한다.
- 미배송 결제완료 주문이 있는 회원의 탈퇴는 거부한다.

### 인증

- 가입 시 패스워드를 설정한다. 패스워드는 8자 이상 72바이트(bcrypt 입력 한계) 이하다.
- 회원은 역할(`BUYER`·`ADMIN`) 하나를 갖는다. 가입은 항상 구매자를 만들고, 관리자는 설정(환경변수 `AUTH_ADMIN_EMAIL`·`AUTH_ADMIN_PASSWORD`·`AUTH_ADMIN_NAME`) 기반 기동 시딩으로만 생성한다(같은 이메일의 활성 회원이 있으면 건너뛰는 멱등 시딩, 관리자 계정 관리 API 없음).
- 이메일+패스워드 로그인이 성공하면 JWT 액세스 토큰을 발급한다. 토큰은 주체(회원 ID)와 역할을 클레임으로 싣는다. 역할 변경은 새 토큰 발급부터 반영된다(가드는 토큰 클레임만 검사).
- 로그인 실패(미존재 이메일·탈퇴 회원·패스워드 불일치)는 원인을 구분하지 않고 동일하게 거부한다(계정 존재 노출 방지).
- 정지 회원의 로그인은 허용한다 — 정지 차단은 담기·주문 자격 게이트가 담당한다(정지·탈퇴 비연쇄 원칙과 동일 축).
- 엔드포인트 표면은 아래 셋으로 분류한다.

  | 분류 | 표면 | 강제 |
  |---|---|---|
  | 공개 | 가입, 로그인, 카탈로그 목록·상세 조회 | 무인증 허용(비로그인 쇼핑) |
  | 본인(셀프서비스) | 장바구니 조회·담기·수량 변경·제거·비우기, 체크아웃, 주문 목록·상세·취소·결제 조회, 쿠폰 발급(셀프 클레임), 발급분 단건·목록 조회, 회원 본인 조회·이름 변경·탈퇴 | 회원을 토큰 주체에서 도출(클라이언트는 memberId를 보내지 않음). 미인증 401 |
  | 관리자 | 회원 지정 조회·정지·해제, 상품 등록·추가 변형 등록·노출·숨김·편집·논리삭제, 변형 가격 변경·활성/비활성/은퇴, 재고 재입고·품절/재개·단종, 쿠폰 정책 생성·중지·재개·발급분 무효화, 주문 출고·배송완료·이행 보류/해제·반품 환불 | 관리자 토큰만 허용. 미인증 401, 관리자가 아닌 주체 403 |
  | 시스템(웹훅) | PG 결제 확정 통지 수신 | 토큰이 아니라 PG 공유 시크릿의 HMAC-SHA256 본문 서명(`X-Webhook-Signature`)을 상수 시간 비교로 검증. 불일치·부재 401 |

- 쿠폰 발급은 회원 셀프 클레임이다 — 대상 회원을 토큰 주체에서 도출하며, 관리자가 특정 회원에게 발급하는 형상은 두지 않았다.
- 타인 소유 리소스(주문·발급분) 접근은 존재 누출 방지로 미존재(404) 취급한다.

### 상품 (카탈로그)

- 상품을 등록하고(최초 숨김), 노출·숨김을 전환할 수 있다.
- 상품명·설명을 편집할 수 있으며, 이미 생성된 주문 내역에는 영향이 없다.
- 상품을 논리삭제할 수 있다.
- 카탈로그 노출은 판매중 ∧ 미삭제 ∧ 활성 변형 1개 이상일 때다.
- 활성 변형이 모두 소진돼도 상품은 계속 노출되며 품절로 표시된다.
- 상품 대표가는 활성 변형의 현재가를 반영한다.

### 상품 변형 (SKU)

- 변형을 판매가·옵션 조합으로 등록할 수 있다(최초 비활성). 판매가는 1원 이상이다(무료 변형 미지원).
- 변형 판매가를 변경할 수 있으며, 이미 생성된 주문 내역에는 영향이 없다.
- 변형을 활성·비활성·은퇴할 수 있고, 은퇴는 되돌릴 수 없는 종료다.
- 한 상품·한 옵션 조합에 비은퇴 변형은 최대 1개이며, 은퇴한 조합은 새 변형으로 재등록할 수 있다.
- 옵션 조합은 생성 시 확정·불변이며, 조합 변경은 새 변형이다.

### 재고

- 재고는 변형당 하나이며 초기 수량으로 시작한다.
- 재고를 차감(부족 시 거부)·복원(취소·결제 실패 보상)·재입고할 수 있다.
- 복원은 재고 상태(품절·단종)와 무관하게 항상 성공한다(취소·환불이 재고 상태에 막히지 않도록). 재입고는 단종 재고에는 거부한다.
- 운영자가 수동 품절·판매 재개, 단종(종료)할 수 있다.
- 동시 주문에도 재고보다 많이 팔리지 않는다(오버셀 없음).
- 소진(수량 0)·수동 품절·단종은 회복 경로가 달라(재입고·판매 재개·불가) 구분한다.

### 장바구니

- 회원당 장바구니 1개를 두고, 최초 담기 시 생성한다(회원 가입과 결합하지 않는다).
- 자격 활성 회원이 활성 변형(그 상품이 노출·미삭제)을 담을 수 있고, 동일 변형 재담기는 수량을 합산한다.
- 수량을 1 이상으로 변경할 수 있다 — 증량은 담기와 동일 자격 게이트, 감량·유지는 게이트 없이 허용한다(정지·탈퇴 회원, 비활성·삭제 라인도 정리 가능). 0으로 낮춰 제거하지는 않는다(제거는 별도 기능).
- 라인 개별 제거·일괄 제거·전체 비우기를 제공한다.
- 재고는 담기 시점에 검증하지 않고 체크아웃 시점에 검증한다. 총액은 체크아웃 시 변형 현재가로 계산해 항상 현재가를 반영한다.
- 결제 완료 시 주문된 변형 라인을 장바구니에서 제거한다.

### 쿠폰

- 쿠폰 정책을 할인(정액/정률)·최소 주문 금액·발급 가능 기간·사용창·발급 한도(선택)로 생성한다. 한도가 없으면 무제한 발급이다.
- 정책의 발급 가능·중지를 전환할 수 있다.
- 회원에게 발급한다 — 정책이 발급 가능·발급 기간 내·대상 회원이 자격 활성·회원당 1회·발급 한도 미소진일 때만. 발급 시 사용 기한을 확정(스냅샷)한다.
- 발급 한도가 소진되면 발급을 거부하고, 동시 발급 경합에서도 한도를 초과하지 않는다.
- 관리자가 미사용(ISSUED) 발급분을 사유와 함께 무효화(REVOKED)할 수 있다. 무효화된 발급분은 사용이 거부된다.
- 사용된(USED) 발급분은 무효화를 거부한다 — 사용된 쿠폰의 소급 회수·주문 금액 재계산은 범위 밖이다.
- 할인액을 계산한다 — 정액은 주문금액 상한, 정률은 퍼센트·상한(있으면)·주문금액으로 제한. 결과는 항상 주문금액 이하다.
- 적용은 (할인 전) 주문금액이 최소 주문 금액 이상·본인 소유·미사용·사용 기한 내·산출 할인 > 0일 때만 한다.
- 주문 생성 후 사용을 확정하고, 주문 취소·결제 실패 시 복원한다.
- 발급 후 정책이 중지돼도 이미 발급된 쿠폰은 계속 사용할 수 있다(중지는 신규 발급만 막는다).
- 만료는 사용 시점 판정으로 거부한다(별도 만료 상태·배치 없음).
- 동시 두 주문이 같은 발급분을 사용하면 한쪽만 성공한다.

### 주문

- 체크아웃으로 장바구니 전체를 주문으로 전환한다(부분 주문 미지원).
- 라인(변형·상품명·옵션·단가)과 배송지를 주문 시점 값으로 스냅샷 보관해, 이후 상품·변형·회원 변경이 내역을 바꾸지 않는다.
- 단가·쿠폰 할인·결제 대상 금액은 서버가 계산한다(클라이언트 전달값 불신). 결제 대상 금액 = 상품 라인 합계 − 쿠폰 할인 + 배송비이며 항상 0 이상이다.
- 배송비(0 가능)는 배송 정책이 범위 밖이라 체크아웃 요청이 제공하는 스냅샷을 사용한다.
- 결제 상태(미결제·완료·취소·환불)를 관리한다.
- 이행 상태(준비·출고·배송완료, 보류·해제)를 결제 상태와 독립 축으로 관리한다. 이행은 결제 전 '시작 전'이며, 결제 완료가 이행을 '준비'로 진입시킨다.
- 모든 이행 전진은 결제 완료 주문에서만 유효하다(취소·미결제 주문은 출고 불가).
- 출고 시 택배사·운송장 번호를 필수로 기록하고 주문 상세에서 조회할 수 있다(출고 전에는 값이 없다). 택배사는 자유 문자열이며 택배사 API 연동·배송 추적·운송장 수정은 범위 밖이다.
- 결제완료 주문은 사용자가 취소할 수 있다(환불 수반). 출고 이후(출고·배송완료)는 취소를 거부한다.
- 배송 완료 주문은 관리자가 전체 반품 환불(사유 기록)할 수 있다. 환불은 주문 전체·전액 1회이며 절차는 관리자 단일 액션이다(요청/승인 워크플로·회수 추적 없음). 출고됐으나 배송 완료 전인 주문은 취소도 환불도 안 되는 의도된 공백이다(배송 완료 확정 후 환불).
- 미결제 주문의 취소는 내부 보상 전이로만 일어난다 — 체크아웃 실패 시 동기 보상, 또는 payment 행 없이 유예가 지난 잔여를 주기 PENDING 스윕이 종결한다(이 범위에서 사용자 개시 미결제 취소는 없다).
- 사람이 읽는 주문번호를 부여한다.
- 생성 후 라인 구성은 불변이며, 변경은 취소 후 재주문이다.

### 결제

- 주문당 결제 1건을 두고, 결제 금액은 주문의 결제 대상 금액과 같다.
- 결제를 승인·실패(사유 기록)·취소(환불)할 수 있다.
- 결제 대상 금액이 0원(전액 할인이고 배송비도 0)이면 PG를 생략하고 자동 승인한다.
- 승인/실패가 동기 확정되는 수단만 지원한다(카드·간편결제·계좌이체). 가상계좌 등 입금 대기·비동기 통지 수단은 범위 밖이다.
- 승인 거래와 취소(환불) 거래를 각각 식별·추적할 수 있다.
- 이미 결제된 주문의 재결제 요청은 거부한다.
- 승인 응답이 유실돼 미확정(요청 상태)으로 남은 결제는 유예 후 주기 리컨실과 웹훅 통지가 PG 거래 상태 조회로 확정한다 — 승인이면 주문 결제완료, 거절·청구 미도달이면 실패 기록과 보상(재고·쿠폰 복원·주문 취소), 주문이 이미 취소된 지연 승인이면 고아 청구 환불(환불 선행 후 승인·취소를 한 커밋으로 기록해 중단 시 재시도 가능하게 유지). 결제 상태는 기록됐는데 주문측 종결이 남은 잔여(승인 기록 후 결제완료 전 중단, 거절 기록 후 보상 취소 실패)는 PENDING 주문 스윕이 발견해 같은 확정 경로가 종결한다. 확정은 반복 실행·중복 전달에 멱등하다.
- 웹훅 통지는 서명 검증을 거치되 페이로드의 결과를 신뢰하지 않는다 — 통지는 결제를 지목하는 트리거이고 확정 근거는 항상 PG 상태 조회다.

### 체크아웃·취소·반품 환불 (크로스 도메인)

- 체크아웃은 검증 → 주문 생성(미결제) → 재고 차감 → 쿠폰 확정 → 결제 → (성공) 결제완료·장바구니 정리 / (실패) 동기 보상 순으로 조율한다.
- 어느 라인이든 주문 불가·재고 부족이면 전체 체크아웃을 거부한다(부분 주문 없음). 문제 라인은 자동 제거하지 않는다.
- 결제 성공 전 모든 실패(재고·쿠폰·결제)는 그 자리에서 동기 보상(복원·주문 취소)한다.
- 결제 후 취소는 환불을 선행하고, 성공 시 재고·쿠폰을 복원한다. 환불이 실패하면 주문은 결제완료로 남고 복원하지 않는다(재시도).
- 반품 환불은 취소와 같은 순서(환불 선행 → 성공 시 재고·쿠폰 복원)로 조율하되, 재고 복원은 반품 상품의 회수·재판매를 가정하고 쿠폰 복원은 사용 기한 경과분이면 명목 복원이다(사용 시점 검증이 거부, 기한 연장·재발급 없음).
- 상품 등록 → 첫 변형·재고 시딩 → 변형 활성화 → 상품 노출을 순차 조율한다.
- 취소·실패·반품 보상에서 재고·쿠폰 복원은 주문의 1회성 전이가 게이트해 중복 복원이 없다. 이 1회성 전이는 주문·결제의 낙관락이 직렬화해 동시 중복 취소·환불이 겹쳐도 한쪽만 커밋된다(진 쪽은 충돌 거부, 클라이언트 재시도). 전이 커밋과 복원 완료 사이 중단의 복원 유실은 잔여로 수용한다(운영 대사로 회복 — 이중 복원(오버셀)보다 복원 유실(팬텀 품절)이 안전). 이중 환불은 결제·주문의 1회성 전이가 구조적으로 거부한다.

## 비기능 요구사항

- **정합성**: 오버셀·쿠폰 이중 사용이 없고, 취소 시 재고·쿠폰이 정확히 한 번 복원되며, 동기 경로의 결제 이후 상태는 즉시 확정된다("결제됐는데 미결제" 창 없음). 응답 유실·프로세스 중단으로 미확정 결제가 남으면 결제 리컨실이 유예 + 스윕 주기 안에 PG 상태 조회로 수렴시킨다(돈과 주문 상태가 어긋난 채 방치되지 않음). 결제 요청 이전 중단으로 payment 행 없이 남은 PENDING 주문은 PG 조회 대상이 없어 주문 기준 PENDING 스윕이 유예 후 보상 종결(재고·쿠폰 복원·주문 취소)로 수렴시킨다. 결제는 종결 기록됐는데 주문측 종결이 남은 잔여(승인 × 미결제, 실패 × 미결제)는 PENDING 스윕이 발견해 결제 리컨실 확정 경로가 수렴시킨다(승인은 결제완료 완결, 실패는 보상 종결). 스윕·리컨실 보상의 재고 복원은 주문 단위 차감 완료 마커(체크아웃이 전 라인 차감 뒤 기록)가 게이트한다 — 증거 없는 잔여(차감 전·차감 중 중단)는 복원을 생략해 과복원(재고 증식→오버셀) 없이 과소복원(팬텀 품절)으로 남고 운영 대사로 회복한다.
- **멱등(더블서밋 방어)**: unsafe 요청에 `Idempotency-Key` 헤더를 실으면 같은 키의 TTL 창 내 재요청을 HTTP 경계(멱등 필터)가 409로 거부한다 — 헤더 opt-in의 best-effort 방어이며 응답 재생은 없다.
- **이력 보존**: 주문은 라인·배송지·상품명·단가를 스냅샷으로 보관해 상품·변형·회원 변경과 무관하게 내역이 불변이다.
- **모듈성·확장성**: 도메인 간 물리 FK·객체 연관·컴파일 의존이 없고(크로스 도메인은 파사드가 조율), 외부 PG는 벤더 중립 포트로 격리해 교체 가능하다.
- **테스트 용이성**: 결제 PG는 트리거 금액으로 실패·응답 유실을 결정론적으로 재현하는 fake, 이벤트는 in-process라 오프라인으로 거래 흐름을 검증한다.
- **동시성·재고 차감**: 동시 주문 경합(재고 차감·쿠폰 사용)과 동시 쿠폰 발급 경합(발급 한도), 동시 취소·환불 경합(복원 이중 실행)에서도 정합성이 깨지지 않는다. 재고는 주문 시 즉시 차감하며 예약(홀드) 모델이 아니다.
- **보안·자격**: 담기·주문·발급은 자격 활성 회원만, 주문·발급분 접근은 본인 소유만 허용한다(미소유는 존재 누출 방지로 미존재 취급). 구매자 셀프서비스는 토큰 주체로만 회원을 식별하고, 관리자 오퍼레이션은 관리자 역할 토큰만 허용한다. 패스워드는 bcrypt 해시로만 저장하고 평문·복호화 가능한 형태를 저장하지 않는다. 로그인은 클라이언트 IP당 창 한도를 넘는 시도를 429로 거부해 무제한 브루트포스·크리덴셜 스터핑을 막고, 모든 응답에 시큐리티 헤더 최소셋(nosniff·no-store)을 싣는다(키 설계·창·한도·보류 헤더 근거는 제약·전제).

## 외부 연동

- **결제 게이트웨이**: 벤더 중립 포트(`PaymentGateway`, approve·cancel·inquire[거래 상태 조회]). 구현은 외부 어댑터이며 개발은 실패·응답 유실을 시뮬레이션하는 fake. 상태 조회 키는 가맹점 참조(결제 ID)다 — 응답 유실 시 호출자가 PG 거래 ID를 모르기 때문.

## 제약·전제

- 결제 게이트웨이는 동기 승인 fake다 — 거래를 인메모리 보관하고 트리거 금액으로 거절·응답 유실을 결정론적으로 시뮬레이션한다. 실 PG 연동은 범위 밖이며, 프로세스 재시작 시 PG측 거래가 소실되는 것(실 PG는 지속)과 다중 인스턴스에서 승인 노드 밖의 상태 조회가 미도달로 판정되는 것(실 PG는 공유 시스템이라 없음 — 확정 경로가 다른 노드의 유실 승인을 청구 미도달 실패로 종결할 수 있다)은 연습 한계로 수용한다.
- 미확정 결제 확정은 app-api 내 `@Scheduled` 리컨실 스윕이 담당하고, 웹훅 수신은 같은 확정 경로를 공유한다. 확정 경로는 생성 후 유예(`payment.reconciliation.stale-after`)가 지난 결제만 손대 동기 체크아웃과 경합하지 않는다 — 유예 전 웹훅 통지는 무시되고 PG 재전달·주기 스윕이 수렴을 보장한다.
- 결제 요청 이전 중단으로 payment 행 없이 남은 PENDING 주문은 별도 `@Scheduled` 주문 기준 PENDING 스윕이 담당한다. 생성 후 유예(`order.reconciliation.stale-after`, 결제 리컨실 유예 이상)가 지난 주문 중 payment 행이 없는 주문만 직접 보상 종결하고, REQUESTED 행이 있으면 미확정 결제 리컨실 관할로 두어 손대지 않으며(이중 개입 차단), 종결 기록된(APPROVED·FAILED) 행이 남긴 잔여는 결제 리컨실 확정 경로에 위임한다(결제 리컨실 스윕은 REQUESTED만 선별해 이를 보지 못하므로 발견은 이 스윕이 소유한다).
- 스케줄 스윕(결제 리컨실·PENDING 스윕)은 다중 인스턴스 배포를 전제하고 ShedLock 분산 락(Redis)으로 노드 간 겹치는 동시 실행을 배제한다 — 멱등·레이트리밋 저장소의 Redis 채택과 같은 다중 인스턴스 근거·기성 인프라다(단일 인스턴스 전제 선언은 그 채택 근거와 모순이라 기각했다). 확정 경로가 멱등이라 중복 실행도 정합성은 깨지지 않으므로 락은 정합성 장치가 아니라 동시 실행이 내는 PG 조회 중복·경합 경고 노이즈 제거다. 노드별 위상이 어긋난 주기 실행은 락으로 줄지 않되 이미 종결된 대상을 재조회하지 않는 빈 스캔이라 수용했다(`lockAtLeastFor`로 주기당 1회로 더 줄이는 것은 이 비용에 과한 장치라 기각). 락 경합은 조용한 건너뜀이고 Redis 장애 시엔 주기마다 오류 로그와 함께 스윕이 걸러지되 다음 주기가 재시도한다 — Redis는 이미 돈의 경로(멱등)의 경성 의존이라 새 장애 축이 아니다. 비정상 종료(SIGKILL 등)로 남은 락은 `lockAtMostFor`(10분) 만료로 회수되며 그동안 전 노드가 해당 스윕을 건너뛴다. 웹훅 확정 경로는 락 밖이라 스윕과의 동시 확정은 남는다(멱등이 흡수).
- 웹훅 인증은 PG와 공유한 시크릿(환경변수 `PAYMENT_WEBHOOK_SECRET`)의 HMAC-SHA256 본문 서명이다(토큰 인증 표면 밖). 아웃박스(내구성 메시징)는 계속 범위 밖이다.
- 도메인 이벤트는 in-process이며 무손실을 보장하지 않는다(유실 시 장바구니가 남는 것 외 영향 없음).
- 통화는 KRW 단일로 가정한다.
- 크로스 트랜잭션 부분 실패(보상 중 크래시)의 무손실 복구는 범위 밖이다.
- 멱등 필터의 키 저장소는 Redis다(`SET NX` 원자 선점, 만료는 TTL) — 프로세스 재시작·다중 인스턴스에서도 중복 차단이 유지된다. 재시작 내구성은 Redis AOF 활성화를 전제한다(compose가 켠다). Redis 장애 시 멱등 검사 대상 요청은 오류로 거부한다(fail-closed) — 중복 차단 없는 통과는 이중 결제 위험이라 가용성보다 정합성을 택했다. Postgres 저장은 신규 인프라가 없는 대안이었으나 TTL 만료 처리를 직접 구현해야 해 기각했다(만료가 규칙인 dedup 락은 네이티브 TTL이 맞다).
- 로그인·가입 레이트리밋 저장소도 Redis다(멱등 저장소와 같은 다중 인스턴스 근거·기성 인프라). 키는 클라이언트 IP(소켓 원격 주소), 창당 한도는 IP당 10회/5분 고정창(원자 `INCR`+`PEXPIRE` Lua)이고 초과 시도는 로그인·가입 표면에 붙는 같은 필터가 429 problem+json으로 거부한다 — 로그인 무제한 시도(브루트포스·크리덴셜 스터핑)와 가입 무제한 시도(활성 이메일을 바꿔가며 회원 존재를 열거)를 원천에서 캡한다. 두 표면은 같은 저장소를 공유하되 표면별 키 접두사(`login:`·`signup:`)로 카운터를 분리해 한 표면의 시도가 다른 표면 한도를 소모하지 않는다. 프록시 헤더(`X-Forwarded-For`)는 알려진 프록시가 없어 신뢰하지 않는다(위조 가능). 이메일 기준·조합 키는 본문 파싱이 필요하고 이메일 회전으로 우회되어 현 단계에서 IP 단일 키를 택했다 — 공유 NAT 부수 차단·분산 출처 우회는 한계로 수용하고 프런트엔드·프록시 도입 시 재결정한다. Redis 장애 시 로그인·가입을 통과시키지 않고 503으로 거부한다(fail-closed) — 멱등 저장소의 fail-closed와 같은 정합성 우선이며, Redis가 이미 돈의 경로(멱등)의 경성 의존이라 로그인·가입 불능이 기존 가용성 수준을 더 낮추지 않는다. 계정 잠금·captcha·2FA는 범위 밖이다(레이트리밋만).
- 시큐리티 응답 헤더는 자체 필터가 모든 응답에 최소셋을 싣는다 — `X-Content-Type-Options: nosniff`(MIME 스니핑 차단)·`Cache-Control: no-store`(토큰 실은 로그인 응답 포함 캐싱 차단). HSTS는 TLS 종단이 앱 밖이라, CSP·`X-Frame-Options` 계열은 브라우저 문서를 제공하지 않는 JSON API라, Referrer·Permissions-Policy는 브라우저 클라이언트가 없어 보류한다. 캐시 전략 도입 시 캐시 가능한 조회를 no-store에서 옵트아웃한다.
- CORS는 미설정을 유지한다 = 브라우저 교차 출처 요청 불허가 기본이다. 브라우저 클라이언트가 없어 허용 출처를 정할 근거가 없고, 프런트엔드 도입 시 허용 출처·메서드·헤더·자격 동반을 재결정한다.
- 인증은 얇은 자체 토큰 발급·검증 코드로 구현한다. Spring Security 전면 도입은 이 범위(로그인 엔드포인트 하나·토큰 검증 필터 하나)에 프레임워크 결합 비용이 과대해 기각했다. 패스워드 해시는 spring-security-crypto(bcrypt)만 단독 사용한다.
- JWT는 HS256 단일 대칭 키로 서명하고 키는 환경변수(`AUTH_JWT_SECRET`)로 주입한다(키 로테이션·JWKS·리프레시 토큰은 범위 밖).
- 관리자 가드는 토큰의 역할 클레임만 검사하고 요청마다 회원 저장소를 조회하지 않는다. 역할 저장소 조회는 변경 즉시 반영이 장점이나 가드가 도메인에 결합되고 요청마다 조회 비용이 들어 기각했다 — 역할 변경 수단이 시딩뿐이라 반영 지연은 토큰 TTL로 바운드된다.
- 관리자 시딩은 마이그레이션이 아니라 설정 기반 기동 시딩이다. 마이그레이션 시딩은 자격증명 해시가 스키마 이력에 고정되고 환경별 자격증명 분리가 불가능해 기각했다.
- 관리자 운영 조회(주문 상태 목록·회원 이메일 검색·쿠폰 정책 목록·정책별 발급분 목록·재고 현황·숨김 포함 상품 목록)는 관리자 쓰기 오퍼레이션과 같은 app-api에 `@AdminOnly` 표면으로 두고 도메인 Reader·Info 경계를 지킨다. 관리자 전용 앱 분리(제외 선언 유지)는 범위 밖이다 — 이 조회들이 전부 단일 도메인이라 크로스 스키마 격리 구역이 불필요하고, 미확정 결제 리컨실 스윕이 이미 app-api에 있는 것과 일관된다.
