# WORKPLAN 5차 — 운영 공백·조회 계층·이벤트 확장

랄프 루프가 이 파일을 진행 상태 저장소로 사용한다. 각 단계를 완료할 때마다 해당 체크박스를 `[x]`로 갱신하고 파일을 저장한다.

## 진행 방법

- 작업은 위에서부터 순서대로, 한 번에 하나만 진행한다.
- 한 이터레이션은 한 작업 안에서만 진행한다. 한 작업의 연속된 단계 여러 개를 끝내도 되지만, 작업 경계를 넘어 다음 작업을 시작하지 않는다.
- 각 작업은 아래 5단계를 순서대로 거친다.

| 단계 | 내용 | 산출물·검증 |
|---|---|---|
| 1. 설계 | `docs/architecture.md`, `docs/coding-conventions.md`, `docs/entity-persistence.md`, `docs/testing.md` 중 해당 문서를 로딩한다. 가정·해석 후보·트레이드오프를 명시하고, 완료 기준을 검증 가능한 테스트 목록으로 구체화한다. | `worklog/5-<번호>-<작업명>.md`에 설계 기록 |
| 2. 설계 리뷰 | 설계를 `AGENTS.md` 작업 원칙(단순함 우선, 외과적 수정, 목표 주도)과 `docs/` 규칙에 대조해 리뷰하고 지적 사항을 설계에 반영한다. "시니어 엔지니어가 이걸 과설계라 할까?"를 자문한다. | worklog 파일에 리뷰 결과·반영 내역 추가 |
| 3. 구현 | TDD로 진행한다(`superpowers:test-driven-development` 스킬 로드). 완료 기준의 테스트를 먼저 작성해 실패를 확인하고, 통과하는 최소 코드를 구현한다. 기존 코드 스타일·패키지 구조를 따른다. | 테스트 전부 통과 |
| 4. 코드 리뷰 | 변경 diff를 검토한다(`superpowers:requesting-code-review` 스킬 로드). 바뀐 모든 줄이 이 작업으로 추적되는지, `docs/` 규칙 위반이 없는지, 고아 import·데드 코드가 없는지 확인하고 수정한다. | worklog 파일에 리뷰 결과 추가 |
| 5. 게이트·커밋 | `./gradlew build` 통과를 확인하고(`superpowers:verification-before-completion` 스킬 로드) 작업 단위로 커밋한다(`feat: ...`). | build 성공, 커밋 완료 |

- 막히면 worklog 파일에 시도한 것과 막힌 지점을 기록하고 이터레이션을 종료한다. 같은 단계에서 3회 연속 실패하면 작업 상태에 `중단`을 표기하고 사유를 기록한 뒤 다음 작업으로 넘어간다.
- 설계 단계에서 결정이 필요한 항목은 worklog에 선택지와 결정·근거를 남긴다. 사용자에게 되묻지 않고 더 단순한 쪽을 택한다.

- 참고: `app-admin .../inquiry/InquiryAdminController.java`, domain-inquiry `InquiryReader.getProductPage`, `OrderAdminController`의 상태 필터 목록(선례).

### 3. 이메일 발송 표면 레이트리밋

- 진행: [x] 설계 / [x] 설계 리뷰 / [x] 구현 / [x] 코드 리뷰 / [x] 게이트·커밋
- 목표: 무인증으로 이메일 발송을 유발하는 표면의 무제한 호출을 차단한다.
- 배경: `RateLimitConfig`는 로그인·가입 두 경로만 보호한다. `POST /api/v1/auth/password-reset-request`는 무인증 이메일 발송 유발 표면인데 무제한 호출이 가능해 이메일 폭탄·회원 존재 열거에 노출된다.
- 범위: 기존 `LoginRateLimitFilter`에 신규 scope(`reset:`)를 등록한다. `email-verification` 등 다른 무인증 POST 표면의 포함 여부는 설계에서 판정한다. 한도·창은 기존값(10회/5분)을 기본으로 하되 설계에서 재검토한다.
- 완료 기준:
  - 한도 초과 시 429 거부 테스트 통과.
  - 표면별 카운터 분리 테스트 통과(로그인 시도가 재설정 한도를 소모하지 않음).
  - Redis 장애 시 fail-closed 계약 테스트 통과(기존 계약과 동일).
- 참고: `app-api .../config/RateLimitConfig.java`, common-web `LoginRateLimitFilter`, `worklog/3-2-비밀번호-재설정-이메일-인증.md`.

### 4. query 모듈 첫 도입 — 관리자 주문 검색

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 회원 이메일 축과 주문 상태 축이 걸친 관리자 주문 검색을 제공하고, query 모듈 유형의 첫 실물을 만든다.
- 배경: `docs/architecture.md`가 query-{name} 계층을 정의했지만 실물이 0개다. 관리자 주문 목록은 상태 필터만 가능해 회원 이메일로 주문을 찾을 수 없다 — 축이 두 도메인(member·order)에 걸쳐 파사드 합성이 성립하지 않는 조회다.
- 범위: `module-query/query-{name}` 모듈 신설(이름은 설계에서 결정) — settings 등록, provided 인터페이스 + Info record, QueryDSL 또는 `EntityManager` 크로스 스키마 조회. `convention.query-module` 컨벤션 플러그인이 미구현이므로 함께 구현하고(`build-logic`), 아키텍처 테스트가 query 모듈 규칙(도메인 상태 변경 금지·시그니처 엔티티 비노출·의존 방향)을 커버하도록 확장한다. 표면은 `GET /api/v1/admin/orders` 확장 vs 별도 검색 경로 중 설계에서 결정한다.
- 완료 기준:
  - 이메일+상태 복합 조건 페이지 검색 테스트 통과.
  - 새 모듈이 아키텍처 테스트(모듈 경계·의존 방향·query 모듈 규칙)를 통과.
  - 기존 상태 단독 필터 목록의 동작이 유지되는 테스트 통과.
- 참고: `docs/architecture.md` query 모듈 구조·크로스 도메인 조회, `build-logic/src/main/kotlin/`(기존 플러그인 선례), `ModuleDependencyRules.kt`.

### 5. 재입고 알림

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋

- 배경: 이벤트가 `OrderPaid` 1종뿐이라 아웃박스·dead letter 인프라가 과소 사용된다. 위시리스트에 재방문 유도 수단이 없다.
- 범위: `module-events/event-stock` 신설과 `StockRestocked` 발행(품절→판매 가능 전이에서만 — 발행 조건 상세는 설계에서 결정). 소비 주체는 설계에서 판정하되 단순함 우선이면 domain-wishlist가 소비해 대상 회원을 도출하고 메일을 보낸다. 변형 이벤트→상품 단위 위시리스트 매핑(variantId→productId) 방법, 메일 계약의 common 승격(현재 domain-member 소유 `LoggingMailGateway` — 두 번째 소비자 등장으로 승격 조건 충족)은 설계에서 결정한다.
- 완료 기준:
  - 품절→판매 가능 전이에서만 이벤트가 발행되는 테스트 통과(단순 증량은 미발행).

  - 중복 전달에 멱등한 소비 테스트 통과.
  - 아키텍처 테스트(이벤트 모듈 의존 제한·소비 커버리지) 통과.
- 참고: `module-events/event-order/OrderPaid.java`, domain-cart `OrderPaidListener`, domain-member `adapter/integration/LoggingMailGateway`, `docs/architecture.md` 이벤트 모듈 구조·크로스 도메인 상호작용.

### 6. 부분 취소

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋

- 배경: 취소·환불이 주문 전체 단위만 지원된다. 부분 취소는 할인 안분·라인 상태·부분 복원이 필요해 기존 제외 항목 중 학습 가치가 가장 크다.
- 범위: `OrderLine` 단위 취소 상태 축 도입, 부분 환불 산식(쿠폰 할인의 라인 금액 비례 안분을 기본으로 하되 끝전 처리 규칙 포함 — 설계에서 확정), 부분 재고 복원, 결제 부분 취소(fake PG의 부분 금액 취소 확장), 마지막 잔여 라인 취소 시 전체 취소로 수렴하는 규칙. 쿠폰 발급분 복원은 전량 취소로 수렴할 때만 한다(부분 취소 중에는 사용 유지 — 근거를 설계에 기록). order 스키마 마이그레이션 새 버전을 포함한다. 표면은 `POST /api/v1/orders/{orderId}/lines/{lineId}/cancel` 계열로 하되 설계에서 확정한다.
- 완료 기준:
  - 부분 취소 후 잔여 결제 금액·라인 상태가 정확한 테스트 통과.
  - 안분 할인 합계가 총할인과 일치하는(끝전 포함) 테스트 통과.
  - 부분 취소 라인의 재고만 복원되는 테스트 통과.
  - 동시 부분 취소 경합에서 한쪽만 커밋되는 테스트 통과.
  - 마지막 라인 취소 시 주문이 전체 취소 상태로 수렴하는 테스트 통과.
- 참고: domain-order `Order`·`OrderLine`, `OrderCancellationFacade`, domain-payment `PaymentProcessor.cancel`, `worklog/3-1-교환-반품-요청.md`(부분 반품 기각 근거 — 이번에 해소).
