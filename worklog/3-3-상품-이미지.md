# 작업 3-3 — 상품 이미지

## 설계

### 가정·해석

- 이미지 바이트는 스토리지가, 메타(키·URL·정렬 순서)는 DB가 소유한다. 조회 응답은 DB의 URL만 실으므로 읽기 경로는 스토리지 포트를 모른다.
- URL은 업로드 시점에 포트가 반환한 값을 영속한다 — 읽기마다 포트로 URL을 파생하면 모든 조회 파사드에 포트 의존이 번진다.
- 대표 이미지는 정렬 순서 최솟값(첫 번째)이다 — 별도 primary 플래그는 "정확히 하나" 불변식 관리(지정·해제·삭제 시 승계)를 강제해 과설계다.
- 정렬 순서는 업로드 순(max+1)이고 재정렬 API는 범위에 없다.
- 파일 서빙(정적 리소스 핸들러)은 범위 밖 — 응답의 URL은 개발용 로컬 구현이 반환한 경로 문자열이다.
- 원자성 트레이드오프: 파일 저장 후 DB 영속이 실패하면 고아 파일이 남는다(역순은 깨진 URL을 노출하므로 더 나쁨). 삭제는 DB 행을 먼저 지우고 파일을 지운다 — 파일 삭제 실패도 고아 파일뿐이다. 고아 파일 정리는 범위 밖.

## 설계 리뷰

- 단순함: primary 플래그·재정렬 API·소프트삭제·이미지 개수 상한·관리자 이미지 목록·정적 파일 서빙·고아 파일 정리를 전부 기각했다. 새 표면은 포트 1·구현 1·엔티티 1·서비스 3·컨트롤러 1(핸들러 2)·에러코드 2다. "시니어가 과설계라 할까?" — 남은 것이 완료 기준 3건과 1:1이다.
- 외과적 수정: 기존 파일 변경은 ProductDetailFacade·ProductCatalogFacade·ProductView·ProductSummaryView·응답 record 2·ProductErrorCode·application.yml·settings·app-api build.gradle.kts·ProductPersistenceTest로 한정. 기존 상품 등록·변형 경로는 건드리지 않는다.
- 규칙 대조:
  - 애그리거트 간 ID 참조·전용 리포지토리(루트당 하나)·파생 쿼리 네이밍 — ProductVariant 선례 그대로.
  - 취소 불가 부작용(파일 쓰기)은 트랜잭션 밖, 영속만 TransactionTemplate — 트랜잭션 경계 규칙·PaymentProcessor 선례. 호출자(컨트롤러)는 트랜잭션 밖 ✓.
  - 하드삭제 명시 동사 `purge` — coding-conventions 네이밍 규칙.
  - 포트 소유(도메인)·구현(external)·벤더 중립 명명 — architecture.md. external 의존 제한은 컨벤션 플러그인이 컴파일 강제.
  - admin 4중 요건·@Operation/@ApiResponse/@Schema — 아키텍처 테스트 게이트 대상.
- 지적·반영 1: 초안은 업로드 응답에 url을 함께 실었다 — "명령은 ID 등 최소 결과만"에 맞춰 imageId만 반환으로 축소(URL은 공개 상세로 확인).
- 지적·반영 2: 초안의 `deleteByIdAndProductId` 단독 벌크 삭제는 삭제 여부(404 판정)와 storageKey 확보가 안 된다 — 조회 후 삭제(관리 엔티티 삭제)로 교체.
- 트레이드오프: 파일 저장↔영속 비원자(고아 파일 수용, 위 가정 참조). 멀티파트 상한(10MB)과 도메인 상한(5MB)의 이중 존재 — 프레임워크 상한은 방어선, 정책은 도메인 소유로 역할이 다르다.
