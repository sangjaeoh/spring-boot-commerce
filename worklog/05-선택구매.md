# 작업 5 — 선택구매(장바구니 부분 체크아웃)

## 설계

### 가정·해석

- "선택구매" = 체크아웃 요청에 장바구니 변형 ID 목록을 실어, 그 라인들만 주문으로 전환하는 것. 생략하면 기존 전체 체크아웃과 동일(후방 호환).
- "주문된 항목만 장바구니에서 제거"는 이미 충족돼 있다 — 장바구니 비움은 `OrderPaid` 이벤트의 `orderedVariantIds` 기준 선택 제거(`OrderPaidListener` → `CartModifier.removeItems`)라서, 스냅샷에 담긴 라인만 제거된다. 추가 구현 불필요(확인만 테스트로 고정).
- 선택 목록은 집합 의미다. 중복 ID는 무해하게 흡수한다(장바구니 라인은 변형별 유일).
- 명시적 빈 목록(`[]`)은 "아무것도 사지 않겠다"라 의미가 없어 400으로 거부한다(`@Size(min = 1)`). 전체 구매는 생략(null)으로 표현한다.
- 작업 3의 미리보기는 "추후 선택구매 필터와 호환"을 범위로 명시했으므로, `CheckoutPreviewRequest`에도 같은 선택 필터를 추가한다(주문서 화면 정합).

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 요청 확장 | `CheckoutRequest`·`CheckoutPreviewRequest`에 첫 컴포넌트로 `@Nullable @Size(min=1) List<@NotNull UUID> variantIds` 추가 | "무엇을 살지"가 논리적 선두(`DirectOrderRequest.items` 선례). 별도 엔드포인트는 계약 분산이라 기각(WORKPLAN이 요청 확장으로 명시) |
| 파사드 | 신규 `checkout(memberId, variantIds, ...)`에 로직, 기존 5-인자 `checkout(...)`은 null 위임 오버로드로 유지 | 기존 호출처 40곳(테스트 11파일)이 전체 체크아웃 계약을 그대로 표현. 시그니처 일괄 변경은 무관 테스트 churn만 유발 |
| 선택 필터 | 사적 `selectItems(items, variantIds)` — null이면 전체, 선택 중 장바구니에 없는 ID가 있으면 거부 | 검증·필터를 한 곳에. 존재 검증은 유일 라인 보장(장바구니 변형별 유일) 위에서 `매칭 수 < 선택 수` 비교로 충분 |
| 에러 계약 | 신규 `ApiErrorCode.CART_ITEM_NOT_FOUND("API_CART_ITEM_NOT_FOUND", 409)` | 요청 형식은 유효하나 장바구니 상태와 불일치 — `EMPTY_CART`와 같은 409 계열. `NOT_ORDERABLE` 재사용은 원인 구분이 흐려져 기각 |
| preview | `preview(memberId, variantIds, shippingFee, issuedCouponId)`로 시그니처 변경(호출처 컨트롤러뿐) | 오버로드 불필요 |

### 완료 기준(테스트 목록)

`OrderControllerTest`에 추가:

1. 부분 체크아웃 — A·B를 담고 B만 선택 체크아웃 201. 주문 금액이 B 기준이고, 장바구니에 A만 남는다.
2. 장바구니에 없는 variantId 선택 — 409 `API_CART_ITEM_NOT_FOUND`.
3. 부분 미리보기 — B만 선택한 미리보기 금액이 B 기준으로 계산된다.
4. 기존 전체 체크아웃 회귀 — 기존 체크아웃·보상·동시성 테스트 전부 통과(변경 없는 5-인자 경로).

### 구현 계획

```text
1. OrderControllerTest에 실패 테스트 3건 추가 + CheckoutRequest·CheckoutPreviewRequest 확장에 따른
   기존 생성 호출처(테스트 16곳 + 프리뷰 몇 곳) 갱신 → 검증: 컴파일 실패(red) 후 갱신
2. ApiErrorCode.CART_ITEM_NOT_FOUND + CheckoutFacade.selectItems·오버로드·preview 확장 → 검증: 컴파일
3. OrderController 두 핸들러에서 variantIds 전달 → 검증: 신규 테스트 통과(green)
4. 체크아웃 회귀 스위트 + ./gradlew build → 검증: 전체 게이트 통과
```

## 설계 리뷰

- 단순함: 신규 로직은 `selectItems` 하나. 나머지는 파라미터 전달. 새 타입 없음(에러코드 1개).
- 외과적 수정: DTO 컴포넌트 추가에 따른 테스트 호출처 갱신은 기계적 파급이며 전부 이 작업으로 추적된다. 위임 오버로드로 파사드 호출처 40곳은 무변경.
- 규칙 대조:
  - 요청 DTO Bean Validation(`@Size(min=1)`·원소 `@NotNull`) — 컬렉션 요청 선례 동일.
  - 에러코드 형식·409 매핑 — `ApiErrorCode` 기존 항목과 동일 골격.
  - 파사드 무트랜잭션 조율 유지.
- 트레이드오프: 선택을 변형 ID로 지정하므로 같은 변형을 다른 수량으로 부분 구매하는 것(수량 분할)은 지원하지 않는다 — 장바구니 라인 단위 선택이 국내 커머스 표준 UX이고, 수량 분할은 요구에 없다.
- 지적·반영: 초안은 파사드 시그니처 일괄 변경이었으나 호출처 40곳 churn 확인 후 위임 오버로드로 전환.

## 코드 리뷰

- 추적성: 변경 12파일 — 프로덕션 6파일은 이 작업의 직접 변경, 테스트 6파일 중 5파일은 DTO 컴포넌트 추가에 따른 기계적 null 인자 삽입(20곳)이며 전부 이 작업으로 추적된다.
- 규칙 대조 결과:
  - 파사드 위임 오버로드로 기존 40개 호출처의 전체 체크아웃 계약 무변경 — 회귀는 기존 스위트가 고정.
  - `selectItems`의 존재 검증은 장바구니 라인 유일성(변형별) 위에서 매칭 수 비교 — 이유를 라인 주석으로 명시.
  - `Set.copyOf`가 중복 선택을 집합 의미로 흡수(설계 결정과 일치).
  - 신규 에러코드 409 매핑·형식은 기존 항목과 동일 골격.
  - 미선택 라인 장바구니 잔존은 기존 `OrderPaid` 이벤트 경로(`orderedVariantIds` 선택 제거)가 담당 — 신규 코드 없음, 테스트로 고정.
- 고아 검사: `Set` import 신규 사용. 데드 코드 없음.
- 지적 없음. 수정 사항 없이 통과.
