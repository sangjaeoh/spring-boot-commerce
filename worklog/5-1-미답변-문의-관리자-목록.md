# 작업 5-1: 미답변 문의 관리자 목록

## 설계

### 현황

- `InquiryAdminController`(app-admin)는 답변 등록(`POST /{inquiryId}/answer`)만 소유한다. 관리자가 답할 문의를 찾는 조회 표면이 없다.
- 조회 provided는 `InquiryReader.getProductPage(productId, pageable)` 하나 — 상품별 공개 목록뿐이다.
- `Inquiry.answer`는 `@Nullable String` — 미답변 = `answer IS NULL`. 별도 상태 enum 없다.
- `InquiryRepository`는 파생 쿼리 1개(`findByProductIdOrderByIdDesc`). UUIDv7 ID 내림차순이 최신순 정렬을 겸한다.
- 선례: `OrderAdminController.getOrdersByStatus` — `@RequestParam` enum 필터 + `PaginationRequest`(1-based) → `PageRequest`(0-based) → `PaginationResponse` 보정. 응답은 어드민 소유 사본 record.

### 결정

- 표면: 기존 `InquiryAdminController`에 `GET /api/v1/admin/inquiries` 핸들러 추가. 별도 컨트롤러를 만들지 않는다.
- 기본 필터(선택지: 미답변 전용 vs 전체+필터): `@RequestParam(defaultValue = "false") boolean answered` 단일 파라미터를 택한다.
  - 무파라미터 호출이 미답변 목록(운영 기본 목적)에 착지하고, `answered=true`로 답변 완료분도 조회 가능 — 범위의 "답변 여부 필터"를 충족한다.
  - "전체(무필터)" 뷰는 넣지 않는다. 동적 쿼리(조건 유무 분기)가 필요해지는데 요구된 운영 목적(답변 대기 발견)에 불필요 — 단순한 쪽을 택했다.
- provided 확장(선택지: `InquiryReader` 확장 vs 신규 계약): 기존 `InquiryReader`에 `getAnswerStatusPage(boolean answered, Pageable)` 추가. 같은 애그리거트 조회에 신규 계약은 과설계다.
- 영속(선택지: 파생 쿼리 2개 vs QueryDSL): 파생 쿼리 2개 — `findByAnswerIsNullOrderByIdDesc`, `findByAnswerIsNotNullOrderByIdDesc`. 정적 단순 조건이라 파생 쿼리 규칙에 맞고, 런타임 조건 조합이 아니므로 QueryDSL 도입 근거가 없다. Reader가 boolean으로 분기한다.
- 비밀글: `InquiryInfo`는 항상 전문을 싣고 마스킹은 노출 계층 소유(기존 계약). 어드민 응답은 마스킹 없이 전문을 싣는다 — 별도 로직 불필요.
- 응답: app-admin 소유 `InquiryPageResponse`/`InquiryResponse` 신설(`web/v1/admin/inquiry/response`). app-api의 `InquiryResponse`는 뷰어 기반 마스킹 형상이라 참조 재사용이 성립하지 않는다(형상이 다름 — memberId·productId 포함, 마스킹 없음). 클래스명은 Order 선례(어드민 패키지의 동명 record)를 따라 Admin 접두 없이 짓는다.

### 가정·트레이드오프

- 관리자 목록의 정렬은 최신순 고정(기존 목록들과 동일). 미답변 운영은 오래된 것 우선이 유리할 수 있으나 정렬 파라미터는 요청 밖 — 넣지 않는다.
- 상품별·회원별 축 필터도 요청 밖 — 넣지 않는다.

### 완료 기준(검증 가능한 테스트 목록 — TDD)

영속 슬라이스 `InquiryPersistenceTest`(도메인 모듈):

1. 미답변 필터(`answered=false`)는 미답변 문의만 최신순 페이지로 반환한다.
2. 답변 완료 문의는 미답변 필터에서 제외되고, `answered=true` 필터에 포함된다.

웹 통합 `InquiryAdminControllerTest`(app-admin):

3. 관리자 목록 조회는 200이고 미답변 문의가 전문(비밀글 포함)·페이지 메타로 실린다.
4. 답변 완료 문의는 미답변 목록에서 제외되고 `answered=true` 필터 목록에 실린다(파라미터 바인딩·defaultValue 배선 검증).
5. 일반 회원의 목록 조회는 403으로 거부된다.

## 설계 리뷰

AGENTS.md 작업 원칙·docs 규칙 대조:

- (확인) 단순함 — 신규 모듈·enum·계약 없음. 파생 쿼리 2개 + 기존 컨트롤러 핸들러 1개 추가가 전부. 동적 쿼리·정렬 파라미터·추가 축 필터를 모두 배제했다.
- (확인) 인가 축 커버리지 — 미인증 401은 `SecurityFilterChainTest`가 어드민 URL 네임스페이스 전역으로 이미 소유한다(테스트 레벨 규칙: 같은 행동을 상위 레벨에서 반복하지 않는다). 컨트롤러 테스트는 선례대로 403 + 성공 경로만 둔다 — 시나리오 4가 정합.
- (확인) 페이지네이션 축 — 시나리오 3의 웹 통합 단언에 1-based 응답 보정(`$.page.number=1`)을 포함한다. `OrderAdminControllerTest` 목록 테스트 선례와 동일한 형식.
- (반영) 응답 클래스명 — 설계 초안의 `InquiryAdminPageResponse`를 Order 선례(`OrderPageResponse` — 어드민 패키지 동명 record, Admin 접두 없음)에 맞춰 `InquiryPageResponse`/`InquiryResponse`로 수정했다.
- (확인) 파생 쿼리 이름 — `Inquiry`는 `deletedAt` 없는 엔티티라 소프트삭제 네이밍(`DeletedAtIsNull`) 대상이 아니다. base finder 직접 호출 금지 예외에 해당한다.
