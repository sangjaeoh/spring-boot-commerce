# 작업 4 — 바로구매

## 설계

### 가정·해석

- "바로구매" = 장바구니를 읽지도 쓰지도 않고, 요청 바디의 `[{variantId, quantity}]` 라인으로 주문·결제를 생성하는 것.
- "재고 차감·쿠폰·결제·보상 의미가 체크아웃과 동일"(WORKPLAN 범위)의 가장 강한 보장은 문서화가 아니라 같은 코드 경로 실행이다. 체크아웃의 부작용 파이프라인(주문 생성 → 재고 차감 → 쿠폰 사용 → 결제 승인 → 결제 완료, 각 단계의 동기 보상 포함)을 사적 메서드로 추출해 두 흐름이 공유한다.
- 요청에 같은 variant가 중복 등장하는 것은 허용한다. 주문 라인이 2개 생기고 재고도 라인별로 차감·복원돼 수학적으로 정확하다. 병합은 조용한 마법이라 하지 않고, 거부는 새 에러코드가 필요해 표면만 늘린다.
- 멱등성은 기존 `IdempotencyFilter`(전역, `Idempotency-Key` 헤더 기반 unsafe 메서드 대상)가 자동 적용된다. 추가 배선 불필요.

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 엔드포인트 | `POST /api/v1/orders/direct`, 201 | 기존 액션형 하위 경로(`/preview`, `/{id}/cancel`) 스타일. `POST /api/v1/orders`에 items 유무로 분기하는 안은 계약이 흐려져 기각(WORKPLAN 배경) |
| 파사드 | `CheckoutFacade.orderDirect(memberId, lines, address, shippingFee, issuedCouponId, method)` | 부작용 파이프라인 공유가 목적이라 같은 파사드. 새 파사드는 보상 로직 중복 또는 대규모 추출 필요 |
| 공유 리팩터 | 체크아웃 3~8단계(금액 산출→주문→차감→쿠폰→결제→확정)를 사적 `placeAndPay(...)`로 추출, 라인 게이트는 `buildOrderableSnapshots`를 라인 단위 `orderableSnapshotOf(variantId, quantity)`로 일반화 | 보상 의미 동일성의 구조적 보장. 작업 3의 글루 7줄과 달리 여기는 보상 포함 ~40줄이라 중복 시 드리프트 위험이 실질적 |
| 파사드 입력 | `com.commerce.api.facade.DirectOrderLine(variantId, quantity)` record, compact ctor에서 `quantity < 1`이면 IAE | 파사드가 web DTO를 받는 방향 역전 회피(기존: 컨트롤러가 도메인 타입으로 변환해 전달). 수량 하한은 web Bean Validation이 1차 차단하므로 IAE는 호출자 버그 선행조건 규칙에 부합 |
| 요청 DTO | `DirectOrderRequest(items @NotEmpty @Valid, shippingAddress, shippingFee, issuedCouponId, method)` + `DirectOrderItemRequest(variantId @NotNull, quantity @NotNull @Positive)` | `CheckoutRequest`·`AddCartItemRequest` 검증 선례 동일. `toLines()` 변환은 `toAddress()` 선례 |
| 응답 DTO | `DirectOrderResponse(orderId)` — `CheckoutResponse` 미러 | 재사용하면 OpenAPI 설명이 체크아웃으로 오기됨. 3줄짜리 명시 계약 선택 |
| 에러 계약 | 400(요청 무효·수단 누락) / 401 / 402(결제 거절) / 409(주문 불가·재고 부족·쿠폰 부적격·자격 없음·동시성 충돌) | 체크아웃 계약에서 빈 장바구니(409 EMPTY_CART)만 빠짐 — 빈 items는 400 `@NotEmpty`가 차단 |

### 완료 기준(테스트 목록)

`OrderControllerTest`에 추가:

1. 성공·장바구니 불변 — 장바구니에 다른 상품을 담아둔 채 바로구매 201. 주문 PAID·payAmount 정확, 대상 재고 차감, 장바구니는 담아둔 그대로.
2. 존재하지 않는 variant — 409 `API_NOT_ORDERABLE`.
3. 재고 부족 — 409 `API_INSUFFICIENT_STOCK`.
4. 결제 거절(트리거 금액 999) — 402 `API_PAYMENT_DECLINED`, 재고 원복(보상 동작 증거).
5. 미인증 — 401 `UNAUTHENTICATED`.

### 구현 계획

```text
1. OrderControllerTest에 실패 테스트 5건 추가 → 검증: 컴파일 실패(red)
2. DirectOrderLine + CheckoutFacade 리팩터(placeAndPay 추출·라인 게이트 일반화)·orderDirect 추가
   → 검증: 기존 CheckoutFacadeTest·CheckoutFaultCompensationTest 등 회귀 통과
3. DirectOrderRequest·DirectOrderItemRequest·DirectOrderResponse + OrderController 핸들러
   → 검증: 신규 테스트 통과(green)
4. ./gradlew build → 검증: 전체 게이트 통과
```

## 설계 리뷰

- 단순함: 신규 표면은 파사드 메서드 1개·레코드 4개·핸들러 1개. 파이프라인은 재사용.
- 외과적 수정 검토: `checkout` 본문을 리팩터하지만, 이는 이 작업의 요구("보상 의미 동일")를 구조적으로 만족시키는 유일한 방법이며, 기존 보상 계약은 기존 테스트(CheckoutFaultCompensationTest·CheckoutCompensationTest·CheckoutConcurrencyTest)가 회귀로 고정한다. 동작 변경 없는 등가 추출임을 회귀 통과로 검증.
- 규칙 대조:
  - facade 무트랜잭션 조율 — 유지(추출은 메서드 분리일 뿐).
  - 값 객체 record 단수 명사(`DirectOrderLine`) — 클래스 접미사 표 부합. web DTO ↔ 파사드 입력의 변환 방향(바깥→안) 준수.
  - 핸들러 OpenAPI 명시·Bean Validation — 계획 반영.
- 트레이드오프: `CheckoutFacade`가 두 진입점(checkout·orderDirect)을 소유해 커진다. 분리 파사드 대비 중복 제거를 우선 — 클래스 요약으로 책임을 명시.
- 지적·반영: 초안은 `buildOrderableSnapshots`에 공용 라인 record를 추가하는 안이었으나, 라인 단위 메서드 일반화가 새 타입 없이 같은 효과를 내 채택.

## 코드 리뷰

- 추적성: 변경 3파일 + 신규 5파일 전부 이 작업 범위.
- 리팩터 검증: `placeAndPay` 추출·`orderableSnapshotOf` 일반화는 동작 변경 없는 등가 추출 — `CheckoutFacadeTest`·`CheckoutFaultCompensationTest`·`CheckoutCompensationTest`·`CheckoutConcurrencyTest` 회귀 전부 통과로 확인.
- 규칙 대조 결과:
  - `DirectOrderLine`은 단수 명사 record 값 객체, compact ctor 검증(IAE는 호출자 버그 선행조건 규칙 부합 — web은 Bean Validation이 1차 차단).
  - 요청 DTO 컬렉션은 `@NotEmpty @Valid List<@NotNull ...>` — `ProductRegistrationRequest` 선례 동일(요청 DTO는 방어적 복사 없이 바인딩 단일 소유, 기존 스타일 준수).
  - `toLines()` 변환은 `toAddress()`·`toProductOptions()` 선례의 바깥→안 방향.
  - 핸들러 OpenAPI 계약: 체크아웃과 동일 골격에서 빈 장바구니만 제외(빈 items는 400).
- 멱등성: 전역 `IdempotencyFilter`가 헤더 기반으로 자동 적용 — 경로 배선 불필요 확인.
- 고아 검사: `buildOrderableSnapshots` 축약 후에도 `ProductInfo`·`ProductVariantInfo` 등 import 사용 유지. 데드 코드 없음.
- 지적 없음. 수정 사항 없이 통과.
