# 작업 3-2 — 비밀번호 재설정·이메일 인증

## 설계

### 가정·해석

- 재설정 토큰이 곧 자격 증명이다 — 토큰 검증과 새 비밀번호 설정은 한 엔드포인트에서 일어나고, 현재 비밀번호 대조는 없다(잊었으니 재설정한다).
- 위조·만료·재사용 토큰은 구분하지 않는 동일 401 거부다 — 리프레시 토큰(작업 2-1)과 같은 존재 노출 방지 계보. 만료 = Redis 키 부재 = 위조와 동일 경로이므로 만료의 저장소 수준 검증은 스토어 테스트가 소유한다.
- 재설정 요청은 이메일 존재와 무관하게 항상 204다 — 계정 존재를 노출하지 않는다. 미존재·탈퇴 이메일이면 토큰 발급·메일 발송 없이 통과한다.
- 이메일 인증은 "기록"이다 — 인증 시각(emailVerifiedAt)을 남기고 본인 조회에 노출한다. 인증 여부로 다른 기능을 게이트하지 않는다.
- 테스트에서 토큰은 메일 게이트웨이 mock/spy로 캡처한다 — 개발용 구현이 로그 발송이므로 토큰의 유일한 관측 지점이 포트 호출이다.

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 미인증 이메일 로그인 차단 | 차단하지 않음 | 기존 가입 회원 소급 문제 — 차단하려면 전원 잠금이거나 "인증됨" 백필 조작이 필요하다. 완료 기준에도 차단 검증이 없다. 차단은 나중에 로그인 경로에 정책 한 줄로 추가 가능한 확장 |
| 메일 포트 | domain-member `port/MailGateway`: `sendPasswordResetMail(to, token)`·`sendVerificationMail(to, token)` | 도메인 비즈니스(자격 회복·이메일 소유 검증)와 직결된 기능이라 도메인이 포트 소유(architecture.md infra vs external). 접미사는 외부 연동 포트 표(`Gateway`). 벤더 중립. 제목·본문 조립은 구현 소유 — 도메인은 의도만 표현 |
| external-mail | `module-external/external-mail`, `com.commerce.external.mail.LoggingMailGateway`(package-private `@Component`) — 로그로 발송을 대체 | WORKPLAN 범위 그대로(SMTP 제외). external-payment 절차 답습: settings 등록·`convention.external-module`(targetDomain=domain-member)·app-api `runtimeOnly` 배선 |
| 토큰 저장 포트 | common-auth `token/OneTimeTokenStore`: `save(namespace, token, subject, ttl)`·`consume(namespace, token): Optional<String>`. infra-redis `RedisOneTimeTokenStore`(`getAndDelete`) 구현 | 2차 작업 1의 Redis 토큰 패턴(기술 포트는 common 소유·infra 구현). 재사용 거부는 consume의 원자적 조회+삭제가 소유 — 별도 "사용됨" 상태 관리 불요. 네임스페이스 파라미터 하나로 재설정·인증 격리(전용 포트 2개는 같은 코드 복제) |
| 토큰 형태 | `UUID.randomUUID()` 불투명 토큰 | 리프레시 토큰 선례. Redis 존재 = 유효 |
| 엔티티 | `Member.emailVerifiedAt`(@Nullable Instant) + `verifyEmail(now)`. 마이그레이션 `V5__add_member_email_verified_at.sql` | 인증됨 상태 enum 확장 기각 — MemberStatus는 계정 활성 축이고 이메일 인증은 독립 사실. 토큰이 1회용이라 재인증 시나리오가 없어 전이 가드 불요 |
| 도메인 서비스 | `MemberModifier.resetPassword(memberId, newRawPassword)`(대조 없이 정책 검증 후 교체), `MemberModifier.verifyEmail(memberId)` | 기존 replacePassword·validatePassword 재사용. Clock은 OrderModifier 선례로 생성자 주입 |
| 재설정 API | `POST /api/v1/auth/password-reset-request`(email, 항상 204) · `POST /api/v1/auth/password-reset`(token+newPassword, 204) — AuthController에 추가 | 자격 회복은 인증 슬라이스. AuthController가 포트 직결하는 선례(login의 refreshTokenStore) 그대로 — 파사드 불요(단일 도메인 + 포트). 클래스 `@Anonymous` 유지: 로그인 상태의 재설정은 기존 비밀번호 변경 API가 있다(트레이드오프로 기록) |
| 인증 API | 가입(`MemberController.register`) 직후 토큰 발급·메일 발송, `POST /api/v1/members/email-verification`(token, 204) — 마커 없는 공개 표면 | 인증 링크는 로그인 여부와 무관하게 도달해야 하므로 `@Anonymous` 불가(인증 주체 403), 마커 없음 + SecurityConfig 공개 경로(작업 2-5에서 확립한 공개 표면 패턴) |
| 실패 응답 | 무효 토큰 401 `UNAUTHENTICATED`(UnauthenticatedException), 비밀번호 정책 위반 400(도메인 예외 전파), 이메일 형식 오류 400 | 새 ErrorCode 없음 — 전부 기존 코드 재사용 |
| TTL 설정 | `auth.password-reset-token-ttl: 30m`·`auth.email-verification-token-ttl: 24h` — application.yml, `@Value` 주입 | refresh-token-ttl 선례 |
| 조회 노출 | `MemberInfo`·`MemberResponse`에 `emailVerifiedAt` 추가 | 인증의 유일한 관측 가능 효과. 없으면 검증 완료를 확인할 표면이 없다 |
| 메일 발송 실패 | 가입 커밋 후 발송이라 발송 실패해도 가입은 유지된다. 재발송 API는 범위 밖 | 재발송·발송 재시도는 완료 기준에 없다 |

### 완료 기준(테스트 목록)

`MemberTest`(엔티티):

1. verifyEmail이 인증 시각을 기록한다.

`RedisOneTimeTokenStoreTest`(infra-redis):

2. save 후 consume은 subject를 반환하고, 재consume은 빈 결과다. — 완료 기준 "재사용 토큰 거부"의 저장소 수준
3. TTL 경과 후 consume은 빈 결과다. — "만료 토큰 거부"
4. 네임스페이스가 다르면 consume은 빈 결과다(재설정·인증 토큰 격리).
5. 미보관 토큰 consume은 빈 결과다.

`AuthControllerTest`(웹):

6. 재설정 요청 204 → 캡처한 토큰으로 재설정 204 → 새 비밀번호 로그인 200·기존 비밀번호 로그인 401. — 완료 기준 1
7. 같은 토큰 재사용 401, 위조 토큰 401. — 완료 기준 2
8. 미존재 이메일 재설정 요청도 204이고 메일을 보내지 않는다.
9. 정책 위반 새 비밀번호 400, 공백 요청 값 400.

`MemberControllerTest`(웹):

10. 가입 시 인증 메일 토큰 발급 → 인증 204 → 본인 조회에 emailVerifiedAt 노출. — 완료 기준 3
11. 위조·재사용 인증 토큰 401.

## 설계 리뷰

- 단순함: 로그인 차단·재발송 API·인증 상태 enum·전용 토큰 포트 2개·파사드·새 ErrorCode를 전부 기각했다. 새 표면은 포트 2(메일·토큰)·구현 2·엔티티 필드 1·Modifier 메서드 2·핸들러 3·DTO 3이다. "시니어가 과설계라 할까?" — consume 기반 1회용 토큰은 상태 관리가 Redis 키 하나로 끝나는 최소 형태다.
- 외과적 수정: 기존 파일 변경은 Member·MemberInfo·MemberResponse·MemberModifier·AuthController·MemberController·SecurityConfig·application.yml·settings.gradle.kts·app-api build.gradle.kts로 한정. 기존 비밀번호 변경·로그인 경로는 건드리지 않는다.
- 규칙 대조:
  - 포트 소유: 메일은 도메인 의미(자격 회복 흐름)가 있어 domain-member `port/` 소유 + external 구현, 토큰 저장은 도메인 의미 없는 기술이라 common 소유 + infra 구현 — architecture.md의 infra vs external 판정 기준 그대로.
  - DIP: 인터페이스(domain-member)와 구현(external-mail)이 다른 모듈. 포트명 벤더 중립(`MailGateway`).
  - external 모듈 의존 제한: domain-member + common만 — convention.external-module이 컴파일 강제.
  - 아키텍처 테스트 검증 대상은 모듈 목록에서 파생 — 신규 모듈 수동 등록 불요(확인함).
  - 상태 전이 엔티티 소유·마이그레이션 신규 버전·request 검증·@Operation/@ApiResponse — 기존 규칙 그대로.
- 지적·반영 1: 초안은 재설정 요청에서 새 Reader 메서드를 추가하려 했다 — `getMemberByEmail`이 이미 있어 컨트롤러가 MemberNotFoundException을 잡아 조용히 204로 바꾸는 형태로 단순화(activeMember의 catch 선례).
- 지적·반영 2: 초안의 인증 엔드포인트 `@Anonymous`는 로그인 상태의 링크 클릭을 403으로 거부하는 결함 — 작업 2-5에서 확립한 "마커 없음 + SecurityConfig 공개 경로" 패턴으로 교체.
- 트레이드오프: 이메일 인증이 아무 기능도 게이트하지 않아 "기록뿐"이다 — 완료 기준 범위를 지키고, 차단 정책은 후속 확장으로 남긴다. 재설정 표면의 클래스 `@Anonymous`는 로그인 상태의 재설정 요청을 403으로 거부한다 — 로그인 상태에는 기존 변경 API가 있어 수용.

## 코드 리뷰

- 추적성: 신규 10파일 — OneTimeTokenStore(common-auth)·RedisOneTimeTokenStore+테스트(infra-redis)·MailGateway(domain-member port)·external-mail 모듈 3파일·요청 DTO 3(PasswordResetMailRequest·PasswordResetRequest·EmailVerificationRequest)·V5 마이그레이션. 기존 변경 — Member(필드·verifyEmail)·MemberInfo·MemberModifier(Clock·resetPassword·verifyEmail)·MemberResponse·AuthController(핸들러 2)·MemberController(가입 후 발급·인증 핸들러)·SecurityConfig(공개 경로)·application.yml(TTL 2)·settings.gradle.kts·app-api build.gradle.kts와 테스트 3파일. 전부 이 작업으로 추적된다.
- TDD 기록: 스토어 4건 red(컴파일 실패) → 포트·구현 → green. 엔티티 1건 red → 필드·메서드·마이그레이션 → green. 웹 6건 red(DTO·핸들러 부재) → 구현 → green.
- 규칙 대조 결과:
  - DIP — MailGateway(domain-member) ← LoggingMailGateway(external-mail, package-private @Component). app-api는 인터페이스만 컴파일 참조, 구현은 runtimeOnly 배선(FakePaymentGateway 선례). OneTimeTokenStore(common-auth) ← RedisOneTimeTokenStore(infra-redis, RedisRefreshTokenStore 선례).
  - external 모듈 의존 제한(domain-member+common)은 convention.external-module의 targetDomain으로 컴파일 강제. 아키텍처 테스트 검증 대상은 모듈 목록 파생이라 수동 등록 없음.
  - 무효 토큰 401 UNAUTHENTICATED 동일 거부(위조·만료·기사용 비구분) — refresh 선례. 미가입 이메일 204 — 존재 비노출.
  - 컨트롤러 직결(파사드 없음) — login의 포트 직결 선례. 트랜잭션은 MemberModifier가 소유.
  - request DTO는 @NotBlank만(형식 비검증) — LoginRequest·MemberPasswordReplacementRequest 관용구. @Operation·@ApiResponse·@Schema 게이트 대상 전부 선언.
- 트레이드오프 확인: consume이 재설정 검증 앞이라 정책 위반 400이면 토큰이 소진된다(재요청 필요) — peek-후-삭제는 재사용 창을 열므로 원자 소비를 우선. 웹 테스트가 400 경로를 고정한다.
- 고아 검사: 신규 import 전부 사용. LoggingMailGateway는 전용 단위 테스트 없이 웹 통합 컨텍스트(spy 대상 원본 빈)로 실행 커버 — 로그 한 줄 구현이라 수용. 데드 코드 없음.
- 비고: CouponAdminControllerTest의 ArgumentSelectionDefectChecker 경고는 기존 의도된 기간 역전 검증으로 이 작업과 무관(미수정).

## 게이트

- 1차 전체 빌드 실패: AuthControllerTest·MemberControllerTest의 `@MockitoSpyBean(MailGateway)`이 새 스프링 컨텍스트를 만들며 공유 Postgres의 커넥션 예산(기본 max_connections=100, 컨텍스트당 Hikari 10)을 초과 — 신규 컨텍스트가 커넥션을 못 얻어 Dialect 결정 실패로 기동 불가.
- 해결: `SharedPostgresContainer`에 `max_connections=300` 설정(1줄). 풀 크기 축소 대안은 12스레드 동시성 테스트(CheckoutConcurrencyTest)의 타이밍을 바꿀 위험이 있어 기각 — 기존 테스트 동작 무변경으로 상한만 올렸다. 이 변경은 테스트 하네스 수용량 조정으로 이 작업의 컨텍스트 추가가 원인이라 범위에 포함된다.
- `MemberTest` 1건, `RedisOneTimeTokenStoreTest` 4건, `AuthControllerTest` 18건(기존 14 + 신규 4), `MemberControllerTest` 신규 2건 포함 전건 통과.
- 재실행 `./gradlew build` 성공(아키텍처 테스트·Spotless·NullAway·Error Prone 게이트 포함).
