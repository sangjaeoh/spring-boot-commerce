# 작업 4-3: 트랜잭션 아웃박스

## 설계

### 현황

- 유일한 도메인 이벤트는 `OrderPaid`다. `OrderModifier.markPaid`(@Transactional)가 `MessagePublisher` 포트로 발행하고, infra-messaging `InProcessMessagePublisher`가 스프링 `ApplicationEventPublisher`로 재발행한다.
- 소비자는 `OrderPaidListener`(장바구니 비우기, @TransactionalEventListener AFTER_COMMIT + REQUIRES_NEW)로 app-api·app-batch에 병존한다(작업 1 기록 참조 — in-process라 발행 프로세스가 소비도 소유).
- 유실 창: AFTER_COMMIT 발행은 커밋 후 프로세스가 죽으면 이벤트가 사라진다. REQUIREMENTS도 "in-process(무손실 보장 아님)"로 인정해 왔다.
- `SchemaFlywayFactory`(common-jpa)가 스키마 목록을 정적으로 소유하고 `db/migration/{schema}` 로케이션 규약으로 마이그레이션한다. `MigrationApplication`은 기본 스캔(자기 패키지)이라 runtimeOnly 의존의 빈을 조립하지 않는다 — 마이그레이션 리소스만 실린다.

### 구조 결정

- 포트·계약(common-messaging 소유, DIP): `MessagePublisher`는 그대로 두고(도메인 코드 무변경), 릴레이가 소비할 기술 포트 `outbox/OutboxStore`(fetchUnpublished·markPublished)와 경계 record `outbox/OutboxMessage`(id·eventType·payload)를 신설한다. "기술 포트는 common이 소유하고 infra가 구현한다".
- 저장 구현(infra-messaging): `OutboxMessagePublisher`가 `MessagePublisher`를 구현해 진행 중 트랜잭션 안에서 outbox 행을 INSERT한다(직렬화: Jackson, 타입: 클래스 FQCN). `JdbcOutboxStore`가 `OutboxStore`를 구현한다. JPA 엔티티 대신 JdbcTemplate을 쓴다 — outbox는 도메인 애그리거트가 아니고, infra에 @Entity를 두면 아키텍처 테스트의 엔티티 소유 모듈 감지·면제 체계에 불필요한 표면이 생긴다. `InProcessMessagePublisher`는 대체되어 삭제한다.
- 원자성 가드: 발행 시 활성 트랜잭션이 없으면 거부한다(IllegalStateException) — 트랜잭션 밖 발행은 커밋-발행 원자성 계약 위반이므로 조용한 자동커밋 대신 기동적 실패를 택한다.
- 스키마: `messaging.outbox` 신설. 마이그레이션은 테이블 소유자인 infra-messaging이 `db/migration/messaging/V1__create_outbox_table.sql`로 소유하고, `SchemaFlywayFactory` 목록에 `messaging`을 추가하며, app-migration에 runtimeOnly(infra-messaging)를 더해 리소스를 싣는다.
- 릴레이(app-batch): `com.commerce.batch.messaging.OutboxRelay` — @Scheduled(fixed-delay, `messaging.outbox.relay.fixed-delay`=1s) + @SchedulerLock(`outbox-relay`). 미발행 행을 생성순으로 읽어 FQCN→클래스 역직렬화 후 `ApplicationEventPublisher`로 발행(전달 방식 유지)하고 발행 표시한다. 행 단위 try/catch로 한 이벤트 실패가 릴레이를 중단시키지 않는다(스윕과 동일 격리). 발행 후 표시 순서라 크래시 시 재전달(at-least-once)이고, 소비 멱등이 중복을 흡수한다.
- 소비 재배치: 발행이 아웃박스 경유가 되면 in-process 전달은 릴레이 프로세스(app-batch)에서만 일어난다.
  - app-batch `OrderPaidListener`: 릴레이는 스케줄 스레드에서 트랜잭션 없이 발행하므로 @TransactionalEventListener(AFTER_COMMIT)는 트리거되지 않는다 → 평 @EventListener + 자체 @Transactional로 전환한다. AFTER_COMMIT이 주던 "커밋된 이벤트만 소비" 보장은 이제 아웃박스(커밋된 행만 존재)가 제공한다.
  - app-api `OrderPaidListener`: 발행 지점이 사라져 죽은 코드가 된다 → 삭제. 두 앱 병존 중복(작업 1의 관찰)이 해소된다.
- 관측 가능한 행동 변화(의도된 것): 체크아웃 직후 장바구니 비우기가 동기(커밋 직후 인프로세스)에서 비동기(릴레이 주기)로 바뀐다. 장바구니 비우기는 기존에도 정합성 비필수·소비 실패 무시 경로였다.

### 가정·트레이드오프

- 이벤트 역직렬화는 FQCN `Class.forName` + DomainEvent 구현 검증으로 한다. 이벤트가 1종(OrderPaid)이라 레지스트리 맵은 과설계다. 클래스 이동 시 잔여 행이 실패하지만 행 단위 격리·로그로 드러난다.
- 릴레이 단일 실행은 ShedLock이 보장하므로 SELECT는 FOR UPDATE SKIP LOCKED 없이 표준 SQL로 둔다.
- 발행 표시 실패로 인한 재전달은 소비 멱등(removeItems는 자연 멱등)이 흡수한다. 죽은 편지 처리(재시도 상한·격리 테이블)는 요청 밖이라 넣지 않는다 — 실패 행은 매 주기 재시도되고 경고 로그로 드러난다.
- app-api·app-admin 프로세스는 outbox 행만 쓰고 릴레이를 돌리지 않는다(관할: 배치). 두 앱 테스트 픽스처의 markPaid가 남기는 행은 무해하다.

### 완료 기준(검증 가능한 테스트 목록 — TDD)

app-batch `com.commerce.batch.messaging.OutboxRelayTest`(통합):

1. 롤백 원자성 — 트랜잭션 안에서 발행 후 롤백하면 outbox에 행이 없고, 릴레이를 돌려도 소비(장바구니 변화)가 없다.
2. 커밋-릴레이-소비 — 장바구니에 담긴 회원의 `OrderPaid`를 커밋하면 릴레이가 발행·소비해 장바구니가 비고 행이 발행 표시된다.
3. 멱등 소비 — 발행 표시 전 크래시를 재현(발행 표시를 SQL로 되돌림)한 뒤 릴레이를 재실행해도 중복 부작용 없이 같은 상태로 수렴하고 행이 표시된다.
4. 트랜잭션 밖 발행 거부 — 활성 트랜잭션 없이 publish하면 IllegalStateException으로 거부되고 행이 남지 않는다.

## 설계 리뷰

AGENTS.md 작업 원칙·docs 규칙 대조:

- (확인) 도메인 코드 무변경 — `MessagePublisher` 포트 유지로 발행 경로 전환이 infra 구현 교체로 끝난다. "영속 단계만 앞에 세운다"의 직접 구현.
- (확인) 포트 소유가 규칙 정합 — OutboxStore는 도메인 의미 없는 기술 포트라 common-messaging 소유·infra-messaging 구현(DIP). 릴레이(app-batch)는 포트만 코드 참조한다.
- (확인) JPA 엔티티 대신 JdbcTemplate — outbox는 애그리거트가 아니고, infra @Entity는 아키텍처 테스트의 엔티티 소유 감지·면제 체계에 새 표면을 만든다. 더 단순한 쪽을 택했다.
- (확인) 과설계 회피 — 이벤트 1종에 타입 레지스트리·DLQ·SKIP LOCKED·재시도 상한을 넣지 않는다. 실패 행은 행 단위 격리 + 경고 로그 + 다음 주기 재시도로 드러난다.
- (확인) 리스너 전환 근거 — AFTER_COMMIT의 "커밋된 이벤트만" 보장을 아웃박스 존재 자체가 대체하므로 평 @EventListener + 자체 트랜잭션이 맞다. app-api 리스너는 발행 지점 소멸로 죽은 코드가 되어 삭제(내 변경이 만든 고아).
- (확인) 테스트 결정성 — 백그라운드 스케줄 릴레이와 명시 relay() 호출이 겹쳐도 상태 기반 단언(장바구니·발행 표시)이라 어느 쪽이 소비했든 통과한다. 데이터는 테스트별 임의 키로 격리(기존 스위트 규약).
- (반영) app-batch build.gradle 무변경을 확인해 배선 목록을 정정했다(common-messaging은 이미 implementation).
- (확인) 관측 행동 변화(동기→비동기 장바구니 비우기)는 이 작업의 본질이다. app-api 검증은 "outbox 기록"으로, 소비 검증은 배치로 이동 — testing.md "행동마다 가장 낮은 레벨 하나" 정합.
