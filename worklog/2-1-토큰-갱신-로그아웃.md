# 작업 2-1 — 토큰 갱신·로그아웃

## 설계

### 가정·해석

- 리프레시 토큰은 JWT가 아니라 불투명 랜덤 토큰(UUID)이다. Redis 키 존재 = 유효, 부재 = 위조·만료·로그아웃 — 세 경우 모두 동일한 401 거부다. 별도 서명·만료 검증 로직이 필요 없어 가장 단순하다.
- 갱신 시 회원을 DB에서 다시 읽는다(`MemberReader.getMember`). 역할 클레임을 갱신 시점 기준으로 싣고, 탈퇴 회원의 갱신을 자연히 거부한다(로그인 → 탈퇴 → 갱신 시나리오는 외부 입력으로 도달 가능).
- 로그아웃은 리프레시 토큰 소지 자체를 자격으로 본다(표준 관행). 인증 헤더를 요구하지 않고 `@Anonymous` 표면에 둔다 — 액세스 토큰이 만료된 뒤에도 로그아웃할 수 있어야 한다.
- 갱신·로그아웃 표면의 레이트리밋은 범위 밖(WORKPLAN 범위에 없음).

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 회전(rotation) | 회전 없음 — 갱신은 새 액세스 토큰만 발급, 리프레시 토큰은 TTL·로그아웃까지 유지 | 회전은 재사용 탐지·이전 토큰 유예 처리 등 상태 관리가 붙는다. TTL이 노출 시간을 상한하므로 연습 규모에서 단순한 쪽을 택한다 |
| 세션 수 | 회원당 다중 세션 — 키를 토큰 값으로 잡으면(`refresh-token:{token}` → memberId) 로그인마다 독립 토큰이 생겨 자연히 다중이 된다 | 단일 세션은 memberId → 토큰 역색인과 기존 세션 무효화 코드가 추가로 필요 |
| 액세스 토큰 블랙리스트 | 없음 — 로그아웃은 리프레시 토큰만 무효화 | 블랙리스트는 모든 요청마다 Redis 조회를 강제해 무상태 JWT의 이점을 없앤다. 액세스 TTL 1h가 잔존 시간을 상한 |
| 포트 위치 | `common-auth` `token` 패키지에 `RefreshTokenStore` 인터페이스, `infra-redis`가 `RedisRefreshTokenStore`로 구현 | 도메인 의미 없는 기술 포트는 common 소유·infra 구현(architecture.md). `JwtTokenCodec`·`LoginRateLimitStore`/`RedisLoginRateLimitStore` 선례. infra-redis에 common-auth 의존 추가(infra는 common-{name} 의존 가능) |
| 포트 시그니처 | `save(token, subject, ttl)` / `findSubject(token): Optional<String>` / `delete(token)` | TTL은 앱 정책이라 호출자가 전달(`LoginRateLimitStore.incrementAndCount(key, window)` 선례). 저장소는 정책 무지 |
| TTL 설정 | `auth.refresh-token-ttl: 336h`(14일) — application.yml 리터럴, `AuthController` 생성자 `@Value` 주입 | 액세스 TTL(`auth.jwt.access-token-ttl`) 선례. JWT가 아니므로 `auth.jwt.*` 밖에 둔다 |
| API | 로그인 응답에 `refreshToken` 추가, `POST /api/v1/auth/refresh`, `POST /api/v1/auth/logout`(204) | WORKPLAN 범위 그대로 |
| 요청 DTO | `RefreshTokenRequest(refreshToken @NotBlank)` 하나를 refresh·logout이 공용 | 두 엔드포인트의 요청 형상이 동일(리프레시 토큰 하나). 중복 record 2개보다 단순 |
| 응답 DTO | `LoginResponse`에 `refreshToken` 컴포넌트 추가(`from(accessToken)` → `of(accessToken, refreshToken)`), refresh는 새 `TokenRefreshResponse(accessToken, tokenType)` | 회전이 없으므로 refresh 응답에 refreshToken을 싣지 않는다 — LoginResponse 재사용 시 null·echo 중 하나를 강제해 계약이 흐려짐. `of`는 값 조합 네이밍 규칙(원본이 둘) |
| 갱신 거부 응답 | `UnauthenticatedException(WebErrorCode.UNAUTHENTICATED)` → 401 `UNAUTHENTICATED` | 새 ErrorCode 없이 기존 코드 재사용. 위조·만료·로그아웃·탈퇴를 구분하지 않는 동일 거부(계정 존재 노출 방지, `MemberCredentialValidator` 선례) |
| 로그아웃 멱등 | 없는 토큰 delete는 no-op, 항상 204 | 재시도·중복 클릭에 안전. 존재 여부 검증은 클라이언트에 이득 없음 |
| SecurityConfig | `PUBLIC_AUTH_PATHS`에 `/api/v1/auth/refresh`·`/api/v1/auth/logout` 추가 | 둘 다 액세스 토큰 만료 후 호출되는 표면 |

### 완료 기준(테스트 목록)

`AuthControllerTest` 추가(웹 통합):

1. 로그인 응답에 refreshToken이 실리고, 그 토큰으로 refresh하면 새 액세스 토큰이 발급된다(코덱 검증: subject=memberId, role=BUYER, tokenType=Bearer).
2. 로그아웃(204) 후 같은 리프레시 토큰의 refresh는 401 `UNAUTHENTICATED`로 거부된다.
3. 위조(미발급 UUID) 리프레시 토큰의 refresh는 401로 거부된다.
4. 탈퇴 회원의 유효 리프레시 토큰 refresh는 401로 거부된다.
5. 로그아웃은 멱등이다 — 같은 토큰 재로그아웃도 204.
6. 공백 refreshToken은 400 `VALIDATION_FAILED`.

`RedisRefreshTokenStoreTest` 추가(infra-redis, RedisIdempotencyStoreTest 하네스 선례):

7. save 후 findSubject는 subject를 반환한다.
8. TTL 경과 후 findSubject는 빈 결과다(만료 거부의 저장소 수준 검증 — 웹 계층에서 만료는 키 부재와 동일 경로).
9. delete 후 findSubject는 빈 결과, 없는 토큰 delete는 no-op.
10. 미보관 토큰의 findSubject는 빈 결과다.

### 구현 계획

```text
1. RefreshTokenStore 포트 + RedisRefreshTokenStore + 스토어 테스트(7~10) → 검증: infra-redis 테스트 green
2. AuthControllerTest에 실패 테스트 1~6 추가 → 검증: red(엔드포인트 부재로 실패)
3. LoginResponse 확장·RefreshTokenRequest·TokenRefreshResponse·AuthController 갱신·SecurityConfig·application.yml → 검증: green
4. ./gradlew build → 검증: 전체 게이트 통과
```

## 설계 리뷰

- 단순함: 새 타입 4개(포트·구현·요청·응답 record)가 전부다. 회전·단일세션·블랙리스트를 전부 기각해 상태는 Redis 키 하나. "시니어가 과설계라 할까?" — 회전 없는 불투명 토큰 + TTL은 이 규모의 최소 형태다.
- 외과적 수정: 기존 파일 변경은 AuthController·LoginResponse·SecurityConfig·application.yml·infra-redis build.gradle.kts·기존 테스트의 `LoginResponse.from` 호출처로 한정. 파급 전부 이 작업으로 추적된다.
- 규칙 대조:
  - 포트 소유(common)·구현(infra) 방향, `Store` 접미사, 벤더 중립 이름 — architecture.md 준수.
  - `of(...)` 값 조합·`from(source)` 단일 원본 규칙 — LoginResponse 팩토리를 `of`로 교체.
  - request record Bean Validation, `@Schema`·`@Operation`·`@ApiResponse` 명시 — 아키텍처 테스트 게이트 대상.
  - 컨트롤러가 도메인 서비스(MemberReader)·공용 포트를 직접 호출 — 단일 도메인 + 기술 포트 조합은 파사드 불요(AuthController 로그인 선례).
- 지적·반영 1: 초안은 Redis 값에 memberId+role을 함께 저장해 DB 조회를 피했다 — 탈퇴·역할 변경이 반영되지 않는 보안 구멍이라 기각하고 갱신 시 MemberReader 재조회로 변경.
- 지적·반영 2: 초안의 refresh 응답 `LoginResponse` 재사용은 refreshToken 컴포넌트가 계약을 흐려(회전 없음인데 토큰 echo) `TokenRefreshResponse` 분리로 변경.
- 트레이드오프: 갱신마다 DB 1회 조회가 붙는다. 갱신 빈도(액세스 TTL 1h당 1회)에서 무시 가능하고, 최신 역할·탈퇴 반영을 얻는다.

## 코드 리뷰

- 추적성: 신규 5파일(포트·구현·스토어 테스트·요청·응답 record) + 변경 6파일(AuthController·LoginResponse·SecurityConfig·application.yml·infra-redis build.gradle.kts·AuthControllerTest) 전부 이 작업 범위. 무관 리팩터 없음.
- TDD 기록: 스토어 테스트 4건 red(포트 부재 컴파일 실패) → 포트·구현 → green. 웹 테스트 6건 red(DTO 부재 컴파일 실패) → 구현 → green(기존 8건 포함 14건 통과).
- 규칙 대조 결과:
  - 팩토리 네이밍 — `LoginResponse.of(accessToken, refreshToken)`(값 조합, 원본 둘), `TokenRefreshResponse.from(accessToken)`(단일 원본).
  - 포트 소유·구현 방향 — `RefreshTokenStore`(common-auth) ← `RedisRefreshTokenStore`(infra-redis, package-private `@Component`, `RedisIdempotencyStore` 선례). app-api는 컴파일 의존 없이 runtimeOnly 배선 그대로.
  - 새 핸들러 2개에 `@Operation`·`@ApiResponse`·request `@NotBlank`·`@Schema` 명시(아키텍처 테스트 게이트 대상). 204는 `@ResponseStatus(HttpStatus.NO_CONTENT)` — MemberAddressController 선례와 동일 관용구.
  - 401 응답은 위조·만료·로그아웃·탈퇴를 구분하지 않는 동일 거부(`UNAUTHENTICATED`) — 존재 노출 방지 원칙 유지.
- 고아 검사: `LoginResponse.from` 제거에 따른 호출처(`AuthController.login`) 갱신 완료. 신규 import 전부 사용. 데드 코드 없음.
- 비고: `.claude/`(랄프 루프 상태)는 커밋에 포함하지 않는다.
