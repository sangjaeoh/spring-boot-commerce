# 작업 2-5 — 상품 문의(Q&A)

## 설계

### 가정·해석

- review와의 동거를 기각하고 별도 `domain-inquiry` 모듈로 간다(WORKPLAN 기본값 유지). 문의와 리뷰는 애그리거트·정책(자격·비밀글·답변)·수명이 전부 달라 스키마를 공유할 이유가 없고, 도메인 모듈은 도메인별 독립 스키마를 소유한다(architecture.md).
- 문의 작성에 구매 자격을 걸지 않는다 — 배경 자체가 "구매 전 질문 채널"이다. 인증 회원이면 작성 가능. 교차 도메인 검증이 없으므로 작성 파사드도 없다(리뷰와 다른 지점).
- 답변은 별도 엔티티가 아니라 Inquiry의 nullable 필드(answer)다. 1문의 1답변이고 답변의 독립 생명주기(목록·삭제)가 요구에 없다. 재답변은 덮어쓰기로 허용한다 — 관리자 답변 수정을 자연 지원하고 예외 한 종류를 줄인다.
- 답변 시각(answeredAt)은 두지 않는다 — 요구는 "답변 등록·노출"뿐이다. 답변 존재는 answer 비null로 판단한다(Clock 의존 제거).
- 비밀글 노출 정책: 상품별 목록은 공개(미인증 열람, 리뷰 목록과 일관)하되 비밀글 행은 존재만 보이고 content·answer를 숨긴다. 작성자 본인·관리자 열람 시 내용을 포함한다(범위의 "작성자·관리자만 열람").
- 마스킹 소유: 도메인이 아니라 앱 계층(응답 조립)이 소유한다. "누가 보느냐"는 역할(ADMIN) 판단을 포함하는 노출 정책인데, 역할은 앱 소유 개념이라 도메인에 들이지 않는다. Info는 전체 내용을 실어 경계를 넘고 유일한 소비자인 컨트롤러 응답 조립이 마스킹한다(트레이드오프로 기록).
- 공개 목록의 선택적 인증: 목록 GET은 `@Anonymous`지만 토큰이 있으면 시큐리티 컨텍스트가 채워진다(JwtAuthenticationFilter fail-open). 컨트롤러가 시큐리티 컨텍스트에서 AuthUser를 선택적으로 읽는다 — IdempotencyFilter의 principal 판독 선례.

### 완료 기준(테스트 목록)

`InquiryTest`(엔티티):

1. 정상 생성(secret 포함), content 공백·1001자 거부.
2. answer 등록·덮어쓰기, 답변 content 검증.

`InquiryPersistenceTest`:

3. validate 정합 — 작성 후 상품별 페이지에 보인다(secret·answer 포함).
4. 답변이 페이지에 반영되고, 없는 문의 답변은 INQUIRY_NOT_FOUND.
5. 최신 문의 우선 정렬.

`InquiryControllerTest`(웹 통합):

6. 작성 201 → 공개 목록 노출. — 완료 기준 "작성·상품별 목록"
7. 비밀글: 작성자 조회는 내용 포함, 타인·미인증 조회는 secret=true에 content·answer 미노출. — 완료 기준 "타인 비밀글"
8. 관리자 토큰 조회는 타인 비밀글 내용도 포함.
9. 미인증 작성 401, 미인증 목록 200.

`InquiryAdminControllerTest`(웹 통합):

10. 관리자 답변 등록 204 → 공개 목록에 답변 노출. — 완료 기준 "답변 등록·노출"
11. 일반 회원 답변 호출 403, 없는 문의 답변 404.

## 설계 리뷰

- 단순함: 답변 엔티티 분리·answeredAt·작성 자격·제목·관리자 목록 API·작성자 수정/삭제를 전부 기각했다. 파사드도 없다(교차 도메인 없음). "시니어가 과설계라 할까?" — 남긴 표면이 완료 기준과 1:1이다.
- 외과적 수정: 기존 파일 변경은 배선 5곳 + SecurityConfig 공개 경로 1줄이다.
- 규칙 대조:
  - admin 표면 4중 요건(admin 패키지·*AdminController·클래스 @Admin·/api/v1/admin/ 프리픽스) — 아키텍처 테스트 게이트 대상.
  - 도메인은 역할을 모른다 — 마스킹(역할 판단)은 앱 소유. 도메인 예외·에러코드는 도메인 소유.
  - 서비스 접미사·create 팩토리·Info from·페이징 규칙 — 기존 모듈과 동일.
- 트레이드오프: 비밀글 전체 내용이 Info로 경계를 넘고 마스킹이 응답 조립에 걸려 있다 — 소비자가 컨트롤러 하나라 수용하고, 웹 테스트 7·8이 마스킹 계약을 고정한다. 관리자 전용 목록 부재는 공개 목록+관리자 토큰으로 대체된다.
- 지적·반영: 초안은 목록 GET에서 `AuthUser`를 필수 주입받는 인증 전용 조회였다 — 공개 목록 요구와 충돌해 선택적 판독(시큐리티 컨텍스트)으로 교체. 초안의 answeredAt은 요구 밖 기능이라 제거.

## 코드 리뷰

- 추적성: 신규는 domain-inquiry 모듈 전체와 app-api 문의 슬라이스(공개 컨트롤러·admin 컨트롤러·DTO 5·웹 테스트 2). 기존 파일 변경은 배선 5곳 + SecurityConfig 공개 경로 1줄 + ReviewController 마커 수정(아래) — 전부 추적된다.
- TDD 기록: 엔티티 4건·영속 3건 red → 도메인 구현 → green. 웹 테스트 red → 구현 → 마스킹 테스트 2건이 403으로 실패.
- 구현 중 결함 발견·수정: 설계의 `@Anonymous`(목록 GET) 사용이 잘못이었다 — 이 마커는 `@PreAuthorize("isAnonymous()")` 익명 전용이라 인증 열람자를 403으로 거부한다(로그인의 "재로그인 403" 계약용). 공개 GET의 올바른 패턴은 마커 없음(ProductController 선례) + 쓰기 핸들러만 메서드 레벨 `@Authenticated`(허용 규칙). InquiryController를 이 구조로 수정.
  - 같은 결함이 작업 4의 ReviewController 목록 GET에도 잠복해 있었다(인증 회원의 공개 리뷰 목록 조회 403 — 당시 테스트가 미인증 조회만 검증해 통과). 내 이전 작업이 만든 결함이므로 함께 수정하고, ReviewControllerTest에 인증 열람자 200 회귀 검증을 추가했다. 이 2파일 변경은 작업 5 범위 밖이지만 결함 수정으로 추적을 남긴다.
- 규칙 대조 결과:
  - admin 표면 4중 요건(admin 패키지·InquiryAdminController·클래스 @Admin·/api/v1/admin/ 프리픽스) — 아키텍처 테스트 통과로 확인.
  - 마스킹은 앱 소유(InquiryPageResponse.of의 isReadable) — 역할(ADMIN) 판단이 도메인에 들어가지 않는다. 웹 테스트가 미인증·타인·작성자·관리자 4경로를 고정.
  - 시큐리티 컨텍스트 선택 판독은 IdempotencyFilter의 principal 판독 관용구와 동일.
  - 요청 DTO는 존재만 검증, 본문 범위는 엔티티 단독 소유 — 리뷰와 동일 원칙.
- 고아 검사: InquiryController·ReviewController에서 제거된 `@Anonymous` import 정리 완료. 신규 import 전부 사용.
