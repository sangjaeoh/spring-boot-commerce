# 작업 2-2 — 체크아웃 멱등키

## 설계

### 가정·해석

- 멱등 배선은 이미 존재한다 — 전 앱 unsafe 표면에 걸리는 `IdempotencyFilter`(common-web) + `RedisIdempotencyStore`(infra-redis, SETNX 원자 선점·fail-closed). 체크아웃·취소·어드민 상품 등록의 중복 키 409 웹 테스트도 있다.
- 이 작업의 실질 격차는 하나다: 키 축이 헤더 원문뿐이라 서로 다른 회원이 같은 키를 쓰면 충돌한다(뒤 요청이 남의 키에 409). WORKPLAN이 요구하는 키 축은 회원+키다.
- "주문 생성 계열 API에 지원"은 전 앱 필터로 이미 충족된다. 주문 경로 전용 재배선은 기존 계약(취소·어드민 표면의 멱등 키 — 기존 테스트 존재)을 깨므로 하지 않는다.
- 필터는 시큐리티 체인(order -100) 뒤(LOWEST_PRECEDENCE)에서 돌므로 시큐리티 컨텍스트의 인증 주체(`AuthUser`)를 읽을 수 있다 — RateLimitConfig 주석("인증·멱등 등 나머지 필터보다 먼저")이 이 순서를 확인해 준다.

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 키 축 | 저장소 키를 `{memberId}:{key}`로 스코프(필터에서 `AuthUser.memberId()` 접두). 미인증 요청은 `anon:{key}` | 회원 간 키 충돌 제거. `anon:` 네임스페이스는 미인증 요청이 `{memberId}:{key}` 꼴 키를 위조 선점하는 것을 구조적으로 차단(회원 축 키는 항상 UUID 접두, anon 키는 항상 `anon:` 접두 — 서로소) |
| 같은 키 재요청 정책 | 409 유지(기존 계약) | 최초 응답 재반환은 응답 본문 저장·완료 전 도착 처리·재생이 필요한 과설계. 409 클라이언트는 주문 목록 조회로 결과 확인 |
| 헤더 필수/선택 | 선택 유지 — 키 없으면 멱등 보장 없이 통과(허용 계약) | 전 앱 공용 필터라 필수화는 모든 unsafe 표면을 깬다. 주문 경로만 필수화하는 경로 분기는 요청되지 않은 복잡도. 완료 기준의 "허용 또는 400" 중 허용 |
| TTL | 기존 유지 — in-flight 5m·완료 후 창 10s | 1차 작업에서 근거와 함께 결정된 값. 재결정할 사유 없음 |
| 동시 직렬화 | 기존 Redis SETNX 원자 선점 그대로 | `RedisIdempotencyStoreTest` 동시성 테스트(32스레드 1승자)가 저장소 수준을 이미 보장. 웹 종단 동시성 테스트만 추가 |

### 완료 기준(테스트 목록)

`IdempotencyFilterTest`(common-web):

1. 인증 주체가 다르면 같은 키도 각각 통과한다(시큐리티 컨텍스트 수동 세팅). — 회원 축 도입의 단위 red 대상
2. 기존 6건(중복 409·구분 키 통과·키 없음·공백 키·장애 503·safe 통과)은 anon 네임스페이스로 그대로 통과.

`OrderControllerTest`(app-api):

3. 서로 다른 회원이 같은 키로 각자 체크아웃하면 둘 다 201, 각자 주문 1건. — 종단 red 대상(현재는 둘째가 409)
4. 같은 회원·같은 키로 동시 체크아웃 4건이면 정확히 1건 201, 나머지 409, 주문 1건. — 완료 기준 "동시 중복 직렬화"
5. 키 없는 체크아웃 2회는 각각 201로 주문 2건 — 허용 계약의 명시 테스트.
6. 같은 키의 바로구매 재요청은 409로 이중 생성하지 않는다 — 주문 생성 계열의 나머지 한 표면.
7. 기존: 같은 키 체크아웃·취소 재요청 409 테스트는 회원 스코프 도입 후에도 통과 유지(완료 기준 "같은 키 2회 → 1건").

### 구현 계획

```text
1. 테스트 1(필터 단위)·3(종단) 작성 → 검증: red(현재 원문 키 공유로 409)
2. IdempotencyFilter 키 스코핑 구현 → 검증: 1·3 green + 기존 멱등 테스트 회귀 없음
3. 테스트 4·5·6 추가 → 검증: green
4. ./gradlew build → 검증: 전체 게이트 통과
```

## 설계 리뷰

- 단순함: 프로덕션 변경은 IdempotencyFilter의 키 조합 로직 한 곳(+Javadoc)이다. 새 타입·새 설정·재배선 없음. "시니어가 과설계라 할까?" — 응답 재반환·경로 분기·전용 필터 신설을 모두 기각했으므로 아니다.
- 외과적 수정: 저장소·컨트롤러·설정은 건드리지 않는다. 기존 멱등 테스트는 계약 그대로 통과해야 한다(회귀 게이트).
- 규칙 대조:
  - 필터·포트 소유(common-web)·구현(infra-redis) 방향 유지 — architecture.md.
  - AuthUser는 common-web 소유 타입이라 동일 모듈 참조 — 의존 추가 없음.
  - 완료 기준 3종(같은 키 1건·동시성·키 없음 계약)을 테스트 4·5·7이 직접 커버.
- 트레이드오프: 회원 스코프 도입으로 "다른 회원의 같은 키"가 잠기지 않게 되는 것은 의도된 계약 변화다(멱등 키는 클라이언트 세션 소유). 키 축 변경으로 배포 시점에 걸쳐 있는 기존 키(원문 축)는 새 축과 불일치하지만, 창이 최대 10s라 영향 무시 가능.
- 지적·반영: 초안은 완료 기준 4(동시성)를 저장소 테스트 인용으로 갈음하려 했다 — 완료 기준이 "주문이 1건만 생성"이라는 종단 사실을 요구하므로 웹 동시성 테스트로 격상.
