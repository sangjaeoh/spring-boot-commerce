# 작업 2 — 비밀번호 변경

## 설계

### 가정·해석

- "본인 비밀번호 변경" = 로그인된 주체가 현재 비밀번호를 증명하고 새 비밀번호로 교체하는 것. 비밀번호 재설정(이메일 링크)·관리자 강제 변경은 범위 밖(WORKPLAN 범위 명시).
- 정지(SUSPENDED) 회원도 변경 허용. 로그인이 정지 회원을 통과시키는 기존 정책(`MemberCredentialValidator` — "정지 회원도 통과한다")과 정합.
- 변경 후 기존 발급 토큰 무효화는 하지 않는다. 무상태 JWT 구조에 세션 무효화 개념이 없고(토큰 블랙리스트 없음), 도입은 인증 설계 변경이라 범위 밖. 트레이드오프로 기록.
- 변경 표면의 시도 제한(레이트리밋)은 WORKPLAN이 명시적으로 제외. 탈취 토큰의 현재 비밀번호 추측 방어는 후속 과제로 기록.

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 엔드포인트 | `PATCH /api/v1/members/me/password`, 204 | `/me` 필드 교체라 rename(`PATCH /me`)과 같은 PATCH 계열. rename에 password 필드를 합치는 안은 관심사 혼합이라 기각 |
| 동사 네이밍 | 엔티티 `replacePassword(newHash)`, 서비스 `MemberModifier.replacePassword(memberId, currentRaw, newRaw)` | `change`·`update`는 금지 범용 동사(coding-conventions 네이밍). `rename` 선례와 같은 의도 동사 |
| 검증 배치 | 현재 비밀번호 대조·새 비밀번호 정책 검증·인코딩 모두 `MemberModifier` 트랜잭션 안 | 조회→대조→교체가 한 원자 단위. Validator 접미사는 상태 변경 부적합 |
| 현재 비밀번호 불일치 | 새 에러코드 `PASSWORD_MISMATCH`(400) + 새 예외 `PasswordMismatchException` | `INVALID_CREDENTIALS`(401) 재사용은 로그인 만료로 오독됨. 요청 필드 값이 틀린 것이므로 400 |
| 새 비밀번호 정책 | `MemberAppender.validatePassword` 재사용 — private static → package-private static으로 완화 | 정책(8자 이상 72바이트 이하)의 소유는 한 곳 유지. 복제는 정책 이원화 위험. 별도 정책 클래스 추출은 검증 1개에 과설계 |
| PasswordEncoder | `MemberModifier`가 `new BCryptPasswordEncoder()` 필드 생성 | `MemberAppender`와 동일 패턴. 인코더 빈 통합 리팩터는 이 작업 범위 밖 |
| Request DTO | `MemberPasswordReplacementRequest(currentPassword, newPassword)` — 둘 다 `@NotBlank` | 가입 요청처럼 형식 검증은 도메인에 위임(요청은 공백만 차단). 명사형 이름으로 금지 동사 회피 |
| 응답 계약 | 204 / 400(요청 값 무효·현재 비밀번호 불일치·새 비밀번호 정책 위반) / 401(미인증) / 404(회원 없음) | rename 계약과 동일 골격 + 400 사유 확장 |

### 완료 기준(테스트 목록)

`MemberControllerTest`에 추가:

1. 성공 — 변경 204 후 새 비밀번호 로그인 200, 기존 비밀번호 로그인 401.
2. 현재 비밀번호 불일치 — 400 `MEMBER_PASSWORD_MISMATCH`.
3. 새 비밀번호 정책 위반(8자 미만) — 400 `MEMBER_INVALID_PASSWORD_FORMAT`.
4. 미인증 — 401 `UNAUTHENTICATED`.

### 구현 계획

```text
1. MemberControllerTest에 실패 테스트 4건 추가 → 검증: 컴파일 실패(red)
2. MemberErrorCode.PASSWORD_MISMATCH + PasswordMismatchException + Member.replacePassword
   + MemberAppender.validatePassword 완화 + MemberModifier.replacePassword → 검증: 컴파일
3. MemberPasswordReplacementRequest + MemberController 핸들러 → 검증: 신규 테스트 통과(green)
4. ./gradlew build → 검증: 전체 게이트 통과
```

## 설계 리뷰

- 단순함: 새 서비스·파사드 없음. 도메인 예외 1개·에러코드 1개·엔티티 메서드 1개·서비스 메서드 1개·DTO 1개·핸들러 1개 — 최소 표면.
- 외과적 수정: 기존 코드 변경은 `validatePassword` 접근제한자 한 단계 완화(같은 패키지 내 재사용) 하나뿐. 테스트 편의가 아닌 프로덕션 재사용 목적이라 "테스트를 위한 완화 금지" 규칙과 무관.
- 규칙 대조:
  - `BaseException` 상속은 경계 도달 예외만 — `PasswordMismatchException`은 컨트롤러 경계까지 전파되므로 적합.
  - 에러코드는 `{Name}ErrorCode` enum에 추가, 메시지·status 형식은 기존 항목과 동일.
  - 쓰기 메서드는 `@Transactional` 소유(Modifier) — 트랜잭션 경계 규칙 충족.
  - 평문 비보관: 로그·엔티티에 raw 비밀번호를 남기지 않는다. 엔티티는 해시만 받는다(가입 선례).
- 보안 검토: 응답 400의 문제 상세가 현재 비밀번호 불일치를 드러내는 것은 본인 인증된 컨텍스트라 계정 존재 노출 문제 없음. bcrypt 72바이트 초과 신규 입력은 정책 검증이 선차단.
- 지적·반영: 초안은 `MemberCredentialValidator.authenticate(email, raw)` 재사용을 검토했으나, 이메일 재조회·401 의미 오염이 생겨 기각하고 Modifier 내 대조로 확정.

## 코드 리뷰

- 추적성: 변경 6파일 + 신규 2파일 전부 이 작업 범위. `MemberController`·`MemberModifier` 클래스 요약 갱신은 책임 확장에 따른 필수 수정.
- 규칙 대조 결과:
  - 엔티티 메서드 `replacePassword`는 의도 동사, 해시만 수신(평문 비보관 선례 유지).
  - `PasswordMismatchException`은 경계 도달 예외라 `BaseException` 상속 적합, 에러코드 400 매핑.
  - 72바이트 초과 현재 비밀번호는 `MemberCredentialValidator`와 동일하게 encoder 예외 선차단 가드 적용.
  - bcrypt 인코딩이 `@Transactional` 안에서 수행되는 것은 가입 경로와 동일 패턴 — 통일 유지.
  - `validatePassword` 완화는 package-private 최소 범위, 정책 소유처 단일 유지(주석으로 명시).
- 고아 검사: 신규 import 전부 사용(javadoc `@throws` 참조 포함). 데드 코드 없음.
- 지적 없음. 수정 사항 없이 통과.
