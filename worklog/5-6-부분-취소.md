# 작업 5-6: 부분 취소

## 설계

### 현황

- 취소는 주문 전체 단위다: `OrderCancellationFacade.cancel` — ① 취소 개시 마커(`requestCancellation`, 출고 차단) ② `PaymentProcessor.cancel`(전액, 트랜잭션 밖, PG 멱등 키) ③ `orderModifier.cancel` 전이 ④ 쿠폰 `restoreUse` ⑤ 라인별 재고 복원. 환불이 복원의 선행조건.
- `OrderLine`은 상태가 없다(스냅샷 + 수량). `Order`는 `@Version` 없음, 환불 누계 없음.
- `Payment`: REQUESTED→APPROVED→CANCELLED 단일 축, 부분 금액 개념 없음. `PaymentGateway.cancel(pgTxId, idempotencyKey)` 전액 전용. FakePaymentGateway는 멱등 키로 취소 트랜잭션 캐시.
- 마이그레이션: ordering·payment 모두 V1 단일. 표면 선례: `POST /api/v1/orders/{orderId}/cancel`.

### 결정

- 라인 상태 축: `OrderLineStatus { ORDERED, CANCELLING, CANCELLED }` (`@Enumerated STRING`). `CANCELLING`은 환불 진행 중 마커 — 전체 취소의 `requestCancellation` 마커를 라인 수준으로 미러링해 PG 실패 후 재시도(재개)를 가능하게 한다. 작업 7의 반품 상태는 이 enum 확장으로 수용한다(라인 상태 배타).
- 안분 산식(끝전 규칙): 라인 할인 = `discount * lineAmount / totalAmount` 내림. 끝전은 별도 배분하지 않는다 — 마지막 활성 라인의 환불액을 "결제 잔액 전액"(`payAmount - refundedAmount`)으로 정의해 끝전·배송비가 마지막 환불에 자동 흡수되고 전량 취소 시 환불 총합이 결제액과 정확히 일치한다(수렴이 산식을 완결).
  - 라인 환불액 = 마지막 활성 라인이면 결제 잔액 전액, 아니면 `lineAmount - 라인 할인`.
- Order 확장: `refundedAmount`(Money, 누계) + `@Version` 신설. 라인 취소마다 루트 필드(refundedAmount)가 바뀌므로 버전이 증가한다 — 같은 주문의 동시 부분 취소를 낙관락으로 직렬화(완료 기준 4). 잔여 결제 금액 = `payAmount - refundedAmount`.
- 3단계 흐름(파사드 — 전체 취소의 환불 선행 원칙 유지):
  1. `orderModifier.beginLineCancellation(orderId, memberId, lineId)` — 트랜잭션 1: 가드(PAID + PREPARING/ON_HOLD, 라인 ORDERED), 환불액 계산·라인에 기록(`refundAmount`), 라인 → CANCELLING, `refundedAmount` 누적(버전 증가·직렬화). 환불액 반환.
  2. `paymentProcessor.cancelPartially(paymentId, amount, refundKey=lineId)` — 트랜잭션 밖. PG 멱등 키 `partial-refund:{lineId}`로 재시도 안전.
  3. `orderModifier.completeLineCancellation(orderId, lineId)` — 트랜잭션 2: 라인 → CANCELLED, 마지막 라인이었으면 주문 CANCELLED 수렴(cancelledAt·사유 CUSTOMER_REQUEST). 이후 파사드가 재고 복원(그 라인 수량만), 수렴 시에만 쿠폰 `restoreUse`(WORKPLAN 규칙 — 부분 취소 중 쿠폰 사용 유지: 최소 주문 금액 등 발급 조건을 잔여 주문이 계속 충족하는지 재판정하는 복잡성을 배제하고, 혜택 회수는 전량 취소 수렴 시점으로 미룬다).
  - 크래시·PG 실패 시 라인은 CANCELLING으로 남고, 재시도는 파사드가 CANCELLING 라인을 감지해 2단계부터 재개한다(기록된 `refundAmount` 사용, PG 멱등). 이미 CANCELLED 라인 재요청은 전체 취소 선례대로 복원 없이 통과(409 대신 멱등 204).
- OrderLine 확장: `status`, `refundAmount`(Money nullable — CANCELLING 진입 시 확정 기록).
- Payment 부분 취소: `refundedAmount`(Money, 누계) 신설. `cancelPartially(Money amount)` — APPROVED 가드, 초과 환불 거부, 누계가 결제액에 도달하면 CANCELLED 전이. `PaymentProcessor.cancelPartially(paymentId, amount, UUID refundKey)` — 전액 취소와 같은 골격(무 PG 승인 생략, PG 후 기록 영속). `PaymentGateway.cancelPartially(pgTxId, amount, idempotencyKey)` 추가, Fake는 키 캐시로 멱등.
- 마이그레이션: ordering `V2__add_partial_cancellation_columns.sql` — orders: `version BIGINT NOT NULL DEFAULT 0`, `refunded_amount BIGINT NOT NULL DEFAULT 0`; order_line: `status VARCHAR(20) NOT NULL DEFAULT 'ORDERED'`, `refund_amount BIGINT`. payment `V2__add_payment_refunded_amount.sql` — `refunded_amount BIGINT NOT NULL DEFAULT 0`.
- 표면: `POST /api/v1/orders/{orderId}/lines/{lineId}/cancel` — 전체 취소 `/cancel` 선례 미러. 본문 없음(전체 취소 선례), 204.
- 경계 노출: `OrderInfo.refundedAmount`, `OrderLineInfo.status`(+`lineId`는 기존 노출 확인) 추가. `PaymentInfo.refundedAmount` 추가.

### 전체 취소와의 상호 배제

- 부분 취소 시작 가드에 `cancelRequestedAt == null`(전체 취소 미진행)을 요구한다 — 전체 취소 진행 중 부분 환불이 끼어들면 이중 환불 창이 생긴다.
- 전체 취소는 부분 취소 이력이 있으면(`refundedAmount > 0`) 거부한다 — 전액 PG 취소가 이미 부분 환불된 금액을 다시 환불하는 것을 차단한다. 남은 라인은 라인 단위로 마저 취소한다(마지막 라인 수렴이 전체 취소와 등가).

### 가정·트레이드오프

- 이행 축은 주문 단위 유지 — 부분 취소된 라인을 제외한 출고 지시는 이행 계층(관리자) 운영 문제로 범위 밖. 부분 취소는 출고를 차단하지 않는다(전체 취소 마커 미사용).
- 배송비는 부분 취소에서 환불하지 않는다(잔여 라인 배송 지속). 마지막 라인 수렴 시 잔액 전액에 포함되어 환불된다.
- 이미 CANCELLING인 라인의 새 요청은 재개(2단계부터)로 처리한다 — 이중 PG 호출은 멱등 키가 방어.
- 낙관락 409는 클라이언트 재시도 계약(기존 규칙).

## 설계 리뷰

AGENTS.md 작업 원칙·docs 규칙 대조:

- (반영) 이중 환불 창 — 초안에 없던 전체 취소↔부분 취소 상호 배제 가드를 추가했다(전체 취소 진행 중 부분 취소 거부, 부분 이력 있는 주문의 전체 취소 거부). 돈이 걸린 경로의 가장 위험한 상호작용이다.
- (확인) 끝전 규칙의 단순성 — 라인별 끝전 배분(최대 잉여법 등) 대신 "마지막 라인 = 잔액 전액"으로 수렴이 산식을 완결한다. 배송비 환불 시점 문제도 같은 규칙이 해소. 시니어 관점에서 과설계 없는 최소 산식.
- (확인) 3단계(CANCELLING 마커)의 필요성 — 2단계 축약(라인 확정 후 PG)은 PG 실패 시 환불 없는 취소 라인이 영구 잔존한다. 전체 취소의 환불 선행·마커 선례를 라인 수준으로 미러링한 것으로, 재시도 재개 경로가 검증 가능하다. 상태 1개(CANCELLING)와 컬럼 1개(refund_amount)가 그 비용.
- (확인) 경합 직렬화 — 루트 `refundedAmount` 누적이 모든 라인 취소를 루트 버전 증가로 만들어 낙관락이 주문 단위 직렬화를 소유한다. 자식만 바뀌는 우회가 없다.
- (확인) 트랜잭션 경계 — begin·complete 각각 단일 애그리거트(Order) 트랜잭션, PG는 트랜잭션 밖(기존 PaymentProcessor 골격), 파사드는 트랜잭션을 열지 않는다.
- (기록) `beginLineCancellation`이 환불액(Money)을 반환한다 — "명령은 최소 결과만 반환" 규칙의 경계 사례로, 환불액은 후속 단계(PG)의 필수 재료라 최소 결과로 판정.
- (확인) 쿠폰 유지 근거 — 부분 취소 중 발급 조건 재판정(최소 주문 금액 등) 복잡성 배제, 전량 수렴 시 복원. WORKPLAN 명시 규칙과 일치.

## 코드 리뷰

리뷰어 서브에이전트 파견(적대적 — 이중 환불·유실 창 전수 추적, 전 스위트 실행). 판정: No(Critical 2건) → 전건 반영 후 재검증.

- (Critical 1, 반영) Payment 환불 기록 이중 누적 — 재개 경로에서 PG는 멱등이지만 기록이 무조건 누적이라 원장 붕괴·잔여 환불 유실 창. `cancelPartially`를 누적에서 **주문 측 확정 누계의 단조 동기화**(`Payment.syncPartialRefund`)로 전환: 같은·낮은 누계 재동기화는 무반영 통과, 종결 후 재동기화 멱등 통과. 파사드가 begin 후 주문 누계를 읽어 전달한다.
- (Critical 2, 반영) 부분 취소 이력 주문의 전체 반품 환불 이중 집행 — `Order.refund()`에 `refundedAmount > 0` 거부(PARTIALLY_CANCELLED), `Payment.cancel()`에도 이중 방어 가드. 라인 단위 반품은 작업 7이 해소(기록).
- (Important 3, 반영) CANCELLING 라인 존재 시 `ship()` 거부(CANCEL_IN_PROGRESS) — 환불 진행 중 주문이 출고된 채 취소 수렴·전액 환불되는 창 봉쇄. CANCELLED 라인만 있는 주문의 출고는 계속 허용(설계 가정 유지).
- (Important 4, 반영) 결제액 0 주문 웨지 — 0원 환불은 PG 생략, 동기화가 `refunded >= amount`로 즉시 종결 전이해 후속 라인이 CANCELLED 멱등 통과.
- (Important 5, 반영) 재개 경로 통합 테스트 2건 추가 — PG 실패 후 재개 완결(환불 1회), 완결 실패 재시도의 기록 무이중.
- (Minor 6, 반영) 불균등 금액·복수 수량 안분 단위 테스트 추가.
- (Minor 7, 기록) complete 커밋 후 크래시 시 그 라인 재고(수렴 시 쿠폰) 복원 유실 창 — 전체 취소 선례("복원 정확히 한 번")와 동일한 수용. (Minor 8, 인지) 안분 long 곱셈 오버플로는 KRW 실범위에서 도달 불가.
- 리뷰어 확인 사항: 끝전 산식의 구조적 합계 보존, refundedAmount 루트 버전 직렬화, 트랜잭션 경계 규칙 정합, 마이그레이션 default 정합.
