# 작업 5-5: 재입고 알림

## 설계

### 현황

- 이벤트는 `OrderPaid` 1종. 발행은 `DefaultOrderModifier.markPaid`가 `MessagePublisher`(아웃박스)로, 소비는 domain-cart `OrderPaidListener`(평 `@EventListener`, 릴레이는 app-batch)가 한다. `EventTypeRegistry`는 app-batch가 배선한다.
- `Stock`은 variant 단위 애그리거트. 상태 전이는 전부 수동 의도 메서드다: `markSoldOut`(판매가능→품절), `markSellable`(품절→판매가능), `increase`(수량 증량 — 상태 무변화). "품절→판매 가능 전이"의 유일한 지점이 `markSellable`이다.
- `WishlistItem`은 (memberId, productId) — 변형이 아니라 상품 단위 찜.
- 메일 포트는 domain-member `application/required/MailGateway`(도메인 어휘 메서드: sendPasswordResetMail 등) + `adapter/integration/LoggingMailGateway`(로그 발송).
- `DomainEvent`는 `eventType()` 키만 요구. `OrderPaid`는 식별자·시각을 싣지 않는다(이벤트 모듈 규칙 "식별자·시각은 발행 측이 생성해 값으로 전달"보다 먼저 작성됨).
- 도메인 간 등록 간선(`dependsOnDomains`)은 현재 0건. 컨벤션 플러그인이 등록·비순환 검증을 지원한다.

### 결정

- 이벤트: `module-events/event-stock`의 `StockRestocked(UUID eventId, UUID variantId, Instant occurredAt)` — 이벤트 모듈 규칙대로 발행 측이 식별자·시각을 생성해 싣는다(eventId는 소비 멱등 가드의 dedup 키를 겸한다). 타입 키 `stock.StockRestocked`.
- 발행 조건(범위 판정): `DefaultStockModifier.markSellable` 성공 시 항상 발행. 엔티티 가드가 품절→판매가능 전이만 허용하므로 조건 검사가 중복되지 않는다. `increase`(단순 증량)·`restore`는 상태 전이가 아니므로 발행하지 않는다. 발행 주체는 도메인 쓰기 서비스(규칙), `Clock` 주입은 `DefaultOrderModifier` 선례.
- 소비 주체(범위 판정): domain-wishlist `application`의 package-private `StockRestockedListener`. 반응(찜 회원 알림)의 소유자가 wishlist다. 앱 소유는 앱 증식 시 복사 문제로 규칙이 금지한다.
- variantId→productId 매핑(선택지: 이벤트에 productId 동봉 vs 소비 측 동기 조회): 소비 측이 `ProductVariantReader.getVariant`로 해석한다. 동봉하려면 stock이 product를 알아야 해 발행 측에 간선이 생긴다 — 조회 필요는 소비 측 관심사이므로 소비 측 간선이 맞다.
- 등록 간선: `domain-wishlist → domain-product`(변형→상품 해석·상품명), `domain-wishlist → domain-member`(수신자 이메일). 첫 등록 간선 2건, 비순환.
- 수신자 도출: `WishlistItemRepository.findAllByProductId`(신규 파생 쿼리) → memberIds → `MemberReader.getMembers(Collection<UUID>)`(신규 bulk provided — 활성 회원만, 반복 단건 조회 금지 규칙) → email.
- 메일 계약(선택지: common 승격 vs wishlist 자체 포트): wishlist 자체 `required/MailGateway.sendRestockMail(to, productName)` + `adapter/integration/LoggingMailGateway`. WORKPLAN은 "두 번째 소비자 등장으로 승격 조건 충족"으로 적었지만, 승격 규칙의 조건은 "같은 계약을 요구하는 두 번째 소비자"다 — wishlist가 요구하는 건 다른 도메인 어휘 계약(재입고 메일)이고, 도메인 어휘 있는 계약은 common으로 올리지 않는다(구현 배치 판정). 공유 가능한 것은 전송 기술(SMTP)뿐인데 현재 전송이 로그 스텁이라 승격하면 common+infra 모듈이 로그 한 줄을 감싼다 — 과설계. 실벤더 전송이 등장하는 시점이 기술 계약 승격 시점이다. member 선례 패턴을 그대로 미러링한다.
- 멱등 가드(선택지: 자연 멱등 없음 — eventId 처리 기록 vs 회원·상품별 알림 시각): wishlist 도메인 소유 `RestockNotification` 엔티티(id + eventId 유니크)로 eventId 처리 기록. 리스너는 ① 기록 존재 시 스킵 ② 수신자 해석·메일 발송(부작용 — 트랜잭션 밖) ③ 처리 기록 저장. 발송 후 기록이라 크래시 창은 at-least-once(재발송 허용, 알림 유실보다 낫다). 마이그레이션 `V2__create_restock_notification_table.sql`(wishlist 스키마).
- 소비 실패 처리: 수신자 해석 실패(미지 변형 등)는 시끄럽게 실패시켜 재시도로 격리한다(판정 재료 규칙). cart 선례의 "삼키기"는 정합성 비필수 근거였고, 알림은 유실 방지가 목적이라 전파한다.
- 배선: app-batch `EventTypeRegistryConfig`에 `StockRestocked` 등록. 소비 커버리지 불변식 — domain-stock을 싣는 실행 앱(app-api·app-admin·app-batch)에 domain-wishlist 임베드 확인·추가(runtimeOnly).

### 가정·트레이드오프

- 알림은 회원·상품 단위 dedup이 아니라 이벤트 단위 dedup이다 — 같은 상품이 다시 품절→재입고되면 새 이벤트로 다시 알린다(의도된 동작).
- 알림 발송 시점에 찜을 해제한 회원은 수신하지 않는다(소비 시점 조회).
- 메일 내용은 상품명만 싣는다(구현 소유). 변형 옵션 상세는 요청 밖.
- `OrderPaid`에 식별자·시각 소급 추가는 범위 밖 — 언급만 한다.

## 설계 리뷰

AGENTS.md 작업 원칙·docs 규칙 대조:

- (확인) 단순함 — 메일 계약 승격 기각(로그 스텁 감싸는 common+infra 모듈 회피), 이벤트 단위 dedup(회원·상품 단위 알림 이력 관리 회피), productId 동봉 기각(발행 측 간선 회피). 신규 표면은 이벤트 모듈 1·리스너 1·포트 1·엔티티 1·bulk 메서드 1이다.
- (확인) 규칙 정합 — 발행은 도메인 쓰기 서비스 트랜잭션 안(아웃박스 원자 기록), 소비는 소비 도메인 application 소유, 식별자·시각은 발행 측 생성(신규 이벤트부터 규칙 준수 — OrderPaid 소급은 범위 밖 기록), 동기 간선은 조회 전용 + 등록 간선 + 비순환.
- (기록) 목 정책 이탈 — 소비자 슬라이스 테스트에서 타 도메인 provided(product·member)를 목으로 대체한다. 목 허용 경계 표에 없는 사례지만, 타 도메인 provided는 "도메인 내부 협력"(목 금지의 근거인 구현 세부 결합 대상)이 아니라 모듈 경계 밖 계약이다. 실 조립 검증은 크로스 스키마 셋업 비용 대비 이득이 없어 기각. 첫 사례라 근거를 남긴다.
- (확인) 멱등 가드의 트랜잭션 경계 — 발송(취소 불가 부작용)은 트랜잭션 밖, 처리 기록만 영속. 발송 후 기록이라 크래시 창은 재발송(at-least-once)이고 순차 중복 전달은 가드가 흡수한다 — 테스트는 순차 중복(시나리오 4)이 소유(스레드 경합 재현 금지 규칙).
- (정정) 수신자 해석 실패의 격리 — 초안은 "dead letter 부재·무기한 재시도"로 적었으나 코드 리뷰가 사실 오류를 지적했다: `OutboxRelay`는 유계 재시도(MAX_ATTEMPTS=10) 후 dead letter로 격리한다(4-3 이후 추가된 인프라). 전파된 소비 실패는 이 계약으로 격리된다.
- (확인) 리스너 메일 루프 부분 실패 시 재전달로 앞 수신자 중복 발송 가능 — 알림 성격상 수용, 기록.
