# 작업 3 — 체크아웃 미리보기

## 설계

### 가정·해석

- "미리보기"는 체크아웃 1~3단계(회원 자격 → 주문 가능 게이트 → 금액 산출)를 부작용 없이 수행하고 금액 4종(상품 합계·할인·배송비·결제 예정액)을 돌려주는 것.
- 금액 정합(완료 기준 1)은 별도 계산 로직 복제가 아니라 체크아웃과 같은 코드 경로 재사용으로 보장한다. `CheckoutFacade`의 사적 메서드(`requireEligibleMember`·`requireNonEmptyCart`·`buildOrderableSnapshots`·`totalOf`·`resolveCouponDiscount`)가 이미 단계별로 분리돼 있어 preview는 이들을 호출만 하면 된다 — 기존 checkout 본문 수정 없음.
- 실패 계약도 체크아웃 앞 단계와 동일하게 미러링한다(빈 장바구니·주문 불가·재고 부족·자격 없음·쿠폰 부적격). 주문서 화면이 결제 전에 같은 이유로 미리 실패를 보여줄 수 있다.

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 엔드포인트 | `POST /api/v1/orders/preview`, 200 | 계산 요청이라 본문이 필요(배송비·쿠폰). GET+쿼리 파라미터는 숫자 `@RequestParam` 직접 선언 금지 규칙 취지와 Bean Validation 일관성에서 불리해 기각 |
| 요청 DTO | `CheckoutPreviewRequest(shippingFee, issuedCouponId)` | `CheckoutRequest`에서 주소·결제수단을 뺀 대칭형. 주소·수단은 금액에 영향 없음 |
| 파사드 | `CheckoutFacade.preview(memberId, shippingFee, issuedCouponId)` 추가 | 금액 정합의 원천. 새 파사드는 게이트 로직 중복·추출이 필요해 기각 |
| 뷰 | `facade/view/CheckoutPreviewView(totalAmount, discountAmount, shippingFee, payAmount)` — Money 필드 record | `*View`는 facade/view 소유 규칙 |
| 응답 DTO | `CheckoutPreviewResponse` — 원 단위 long 4필드 | 기존 응답들의 `Money.amount()` 노출 패턴 동일 |
| 에러 계약 | 400(요청 무효) / 401(미인증) / 409(빈 장바구니·주문 불가·재고 부족·자격 없음·쿠폰 부적격) | 체크아웃 계약에서 결제 관련(402·수단 누락 400)만 제외 |
| 부작용 | 없음 — 주문 생성·재고 차감·쿠폰 사용·결제 요청 전에 반환 | 범위 명시 사항 |

### 완료 기준(테스트 목록)

`OrderControllerTest`에 추가:

1. 정합 — 쿠폰(10%)·배송비 3000원 미리보기가 `totalAmount=20000, discountAmount=2000, shippingFee=3000, payAmount=21000`을 반환하고, 이어서 같은 조건 체크아웃의 `payAmount`와 일치한다. 미리보기 후 체크아웃이 성공하는 것 자체가 부작용 없음(장바구니·쿠폰·재고 보존)의 증거다.
2. 빈 장바구니 — 409 `API_EMPTY_CART`.
3. 쿠폰 부적격(최소 주문 금액 미달) — 409 `API_COUPON_NOT_APPLICABLE`.
4. 미인증 — 401 `UNAUTHENTICATED`.

### 구현 계획

```text
1. OrderControllerTest에 실패 테스트 4건 추가 → 검증: 컴파일 실패(red)
2. CheckoutPreviewView + CheckoutFacade.preview → 검증: 컴파일
3. CheckoutPreviewRequest + CheckoutPreviewResponse + OrderController 핸들러 → 검증: 신규 테스트 통과(green)
4. ./gradlew build → 검증: 전체 게이트 통과
```

## 설계 리뷰

- 단순함: 신규 클래스 3개(뷰·요청·응답) + 파사드 메서드 1개 + 핸들러 1개. 계산 로직 신규 작성 없음(재사용만).
- 외과적 수정: 기존 checkout 메서드 본문·사적 메서드 시그니처 변경 없음. `CheckoutFacade` 클래스 요약은 미리보기 책임 추가를 반영해 갱신.
- 규칙 대조:
  - facade는 트랜잭션을 열지 않고 도메인 서비스 조립만 수행 — preview도 Reader 호출 조합이라 동일.
  - `*View`는 facade/view 소유, 응답 DTO는 web 슬라이스 소유 — 충족.
  - 핸들러 `@Operation`·`@ApiResponses` 필수, 요청 검증은 Bean Validation — 충족 계획.
- 트레이드오프: 미리보기 시점과 체크아웃 시점 사이 가격·재고·쿠폰 상태 변경은 정합을 깨뜨릴 수 있으나, 이는 조회 API의 본질적 한계로 수용(체크아웃이 최종 검증을 다시 수행).
- 지적·반영: 초안의 GET 방식은 위 표 근거로 POST로 확정. 응답에 주문 라인 상세 포함안은 장바구니 조회 API와 중복이라 금액 4종만으로 축소.

## 코드 리뷰

- 추적성: 변경 3파일 + 신규 3파일 전부 이 작업 범위. `CheckoutFacade`·`OrderController` 클래스 요약 갱신은 책임 확장 반영.
- 검토 쟁점: `preview`와 `checkout` 사이 1~3단계 조율 글루 7줄이 중복된다. 공유 `Pricing` 레코드 추출안을 검토했으나 기각 — 실제 게이트·산식은 공유 사적 메서드가 소유하고, 중복은 호출 순서 글루뿐이며, 정합은 통합 테스트가 고정한다. 단일 사용처를 위한 중간 타입 추출은 과설계로 판단.
- 규칙 대조 결과:
  - facade는 트랜잭션 없이 Reader 조합만 수행 — preview 충족.
  - `*View`는 facade/view 소유, 파생 컴포넌트 `@param` 문서화 — 충족.
  - 핸들러 `@Operation`·`@ApiResponses`·요청 Bean Validation — 충족.
  - 에러 계약이 체크아웃에서 결제 관련만 제외한 부분집합인지 확인 — 충족.
- 고아 검사: 신규 import 전부 사용. 데드 코드 없음.
