# 작업 5-2: 관리자 리뷰 모더레이션

## 설계

### 현황

- `Review`는 소프트삭제 없는 엔티티(rating·content·member/product 논리 참조)다. `(member_id, product_id)` 전행 유니크 인덱스로 상품당 한 건.
- 삭제는 본인 하드삭제(`ReviewRemover.remove(reviewId, memberId)` → `repository.delete`)뿐이다. app-admin에 리뷰 표면이 없다.
- 선례: `IssuedCoupon.revoke(reason, now)` — 사유를 `revoke_reason VARCHAR(255)` nullable 컬럼으로 행에 보존. 요청은 `@NotBlank String reason`.
- 선례: member·product의 소프트삭제 — nullable `deletedAt` + 파생 쿼리 `...AndDeletedAtIsNull` 네이밍. `IncludingDeleted` 실물 선례는 없다.

### 결정

- 제거 방식(선택지: 소프트삭제 vs 블라인드 상태 enum 신설 vs 하드삭제+감사 테이블): 소프트삭제를 택한다. entity-persistence의 기본 규칙(논리삭제·nullable `deletedAt`)과 정합하고, 상태 enum·감사 테이블 신설은 이 요구에 과설계다. `Review`에 `deletedAt`·`removedReason`(nullable) 필드를 추가하고 엔티티 의도 메서드 `delete(String reason)`이 두 값을 설정한다.
- 본인 삭제와의 구분(선택지: 본인 삭제도 소프트로 통합 vs 기존 하드삭제 유지): 기존 하드삭제 유지. 외과적 수정 원칙 — 본인 삭제 행동 변경은 요청 밖이다. 사유 구분 문제는 소멸한다: 사유는 관리자 제거에만 존재한다. 본인 하드삭제 후엔 유니크가 풀려 재작성 가능, 관리자 제거 후엔 행이 남아 재작성 불가.
  - 기존 `remove`가 하드삭제인데 명시 동사(`purge*`) 규칙과 긴장이 있다 — 기존 코드라 손대지 않고 기록만 남긴다.
- 재작성 정책(선택지: 재작성 불가(전행 유니크 유지) vs 재작성 허용(부분 유니크 인덱스 전환)): 재작성 불가. 부적절 리뷰 제거 후 같은 작성자의 재게재를 막는 것이 모더레이션 목적에 부합하고, 기존 유니크 인덱스·중복 검사와 정합해 마이그레이션이 컬럼 추가로 끝난다.
- 리포지토리(소프트삭제 네이밍 전환):
  - `findByProductIdOrderByIdDesc` → `findByProductIdAndDeletedAtIsNullOrderByIdDesc`
  - `findByIdAndMemberId` → `findByIdAndMemberIdAndDeletedAtIsNull`
  - `existsByMemberIdAndProductId` → `existsByMemberIdAndProductIdIncludingDeleted`(`@Query` JPQL — 파생 파서가 `IncludingDeleted` 접미를 처리하지 못한다. 삭제 포함이 맞는 의미: 재작성 불가 정책·전행 유니크와 정합)
  - 관리자 제거 대상 조회 `findByIdAndDeletedAtIsNull(UUID id)` 추가(소프트삭제 엔티티의 base finder 직접 호출 금지 규칙).
- provided: `ReviewRemover.removeByAdmin(UUID reviewId, String reason)` 추가. 이미 제거됐거나 없는 리뷰는 `ReviewNotFoundException`(활성 조회 실패). 별도 Modifier 계약 신설은 과설계 — 삭제 행동이므로 Remover 소유.
- 관리자 목록 조회(범위 판정): 신설하지 않는다. 완료 기준에 없고, 최소 범위는 상품별 공개 조회 재사용으로 충족된다.
- 표면: app-admin `web/v1/admin/review/ReviewAdminController` 신설 — `DELETE /api/v1/admin/reviews/{reviewId}` + `ReviewRemovalRequest(reason @NotBlank)`(쿠폰 무효화 요청 선례), 204. 사유 검증은 선례와 동일하게 `@NotBlank`만 둔다(DB `VARCHAR(255)` 초과는 선례에도 미검증 — 관리자 전용 표면이라 위험 낮음, 기록만).
- 마이그레이션: `V2__add_review_removal_columns.sql` — `deleted_at TIMESTAMPTZ NULL`, `removed_reason VARCHAR(255) NULL` 컬럼 추가. review 스키마는 `SchemaFlywayFactory`에 등록돼 있어 파일 추가로 끝난다.

### 가정·트레이드오프

- 관리자 제거의 멱등성: 이미 제거된 리뷰 재제거는 404로 거부한다(활성 조회 실패). 멱등 204보다 단순하고 쿠폰 무효화(409/404 거부) 선례와 정합.
- `ReviewInfo`·공개 응답은 무변경 — 제거 사유는 공개 표면에 노출하지 않는다.
- 엔티티 `delete(reason)`에 이미-삭제 가드를 두지 않는다 — 서비스가 활성 조회로만 도달하므로 일어날 수 없는 상황이다.

### 완료 기준(검증 가능한 테스트 목록 — TDD)

영속 슬라이스 `ReviewPersistenceTest`(도메인 모듈):

1. 관리자 제거 후 상품별 공개 목록에서 제외된다.
2. 관리자 제거는 행에 사유·삭제 시각을 보존한다.
3. 없는 리뷰·이미 제거된 리뷰의 관리자 제거는 `REVIEW_NOT_FOUND`로 거부된다.

웹 통합 `ReviewAdminControllerTest`(app-admin, 신설):

4. 관리자 제거는 204이고 상품별 공개 목록에서 빠진다.
5. 일반 회원의 제거는 403으로 거부된다.
6. 공백 사유는 400으로 거부된다(Bean Validation 배선 대표).

## 설계 리뷰

AGENTS.md 작업 원칙·docs 규칙 대조:

- (확인) 단순함 — 상태 enum·감사 테이블·관리자 목록·부분 유니크 인덱스 전환을 모두 배제. 컬럼 2개 추가 + 계약 메서드 1개 + 컨트롤러 1개가 전부다.
- (확인) 소프트삭제 전환의 파급 — 리포지토리 네이밍 전환 3건은 규칙(활성-only는 `DeletedAtIsNull`, 삭제 포함은 `IncludingDeleted`)이 강제하는 최소 변경이다. 호출부는 이름 교체만으로 동작 불변(기존 행은 전부 활성).
- (반영) 사유 보존 단언 방법 — 영속 테스트에서 제거된 행 조회에 base `findById`를 쓰면 소프트삭제 엔티티 base finder 금지 규칙과 긴장이 생긴다. `@DataJpaTest`의 `TestEntityManager.find`로 단언해 리포지토리 표면을 우회한다(테스트 전용 프로덕션 표면 확장도 회피).
- (확인) Bean Validation 축 — `ReviewRemovalRequest`는 제약이 `@NotBlank` 하나뿐이라 웹 슬라이스 배선 대표(시나리오 6)가 제약 전수 검증을 겸한다. 쿠폰 무효화 요청 선례도 standalone Validator 단위 테스트를 두지 않았다.
- (확인) 트랜잭션 경계 — `removeByAdmin`은 `Remover` 쓰기 메서드로 `@Transactional` 소유, 단일 애그리거트 변경.
