# 작업 6 — 배송지 관리

## 설계

### 가정·해석

- 배송지첩은 회원 도메인 소속(domain-member, member 스키마). 별도 도메인 신설은 과설계.
- `MemberAddress`는 독립 애그리거트 루트로 두고 `member_id` 논리 참조만 보관한다. Member 자식 컬렉션으로 두면 배송지 조작마다 회원 로딩이 강제되고, 생명주기 결합 이득이 없다.
- 주문의 `Address` 값 객체(스냅샷)와 필드가 유사하나 공유 VO로 추출하지 않는다 — 배송지첩은 기본 여부·수명주기를 가진 엔티티고, 주문 스냅샷은 불변 복사본이라 개념이 다르다. 컬럼 폭만 주문 DDL과 일치시킨다.
- 삭제는 하드 삭제. 주문이 배송지 스냅샷을 보관하므로 이력 참조가 없고, 소프트삭제는 이 데이터에 이득이 없다. 하드삭제 노출 명명 규칙(`purge*`)을 따른다.
- 기본 배송지 삭제 시 다른 배송지를 자동 승격하지 않는다(조용한 마법 회피). 기본 없음 상태를 허용한다.
- 체크아웃 요청 형태는 변경하지 않는다 — 클라이언트가 저장된 배송지로 `shippingAddress`를 채워 보낸다(WORKPLAN 범위).

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 엔티티 | `MemberAddress` — id(UUIDv7)·memberId·recipientName·zipCode·roadAddress·detailAddress(@Nullable)·phone·defaultAddress(`is_default`) | `BaseTimeEntity` 상속, 시각 필드는 Auditing. `default`는 자바 키워드라 필드명 `defaultAddress` |
| 개수 제한 | 회원당 최대 10개 — `Appender`에서 `countByMemberId` 선검사, 초과 시 409 | 국내 커머스 통상 한도. 상수는 Appender 소유 |
| 첫 배송지 | 자동 기본 지정(count == 0이면 default로 생성) | 배송지 1개인데 기본이 없는 상태는 UX 결함 |
| 기본 단일성 | 지정 시 같은 트랜잭션에서 벌크 해제(`releaseDefaultByMemberId`) 후 대상만 지정 + DB 부분 유니크 인덱스(`member_id WHERE is_default`) 백스톱 | 애플리케이션 불변식 + DB 강제 이중 방어. 대상이 이미 기본이면 조기 반환 — 벌크 해제가 영속성 컨텍스트를 우회해 dirty checking이 재지정을 놓치는 함정 회피 |
| 기본 지정 API | `POST /{addressId}/default` 액션 분리. 생성·수정 요청에는 기본 플래그 없음 | 단일성 로직을 한 진입점에 고정. 요청 플래그 방식은 생성·수정 두 곳에서 단일성을 다뤄야 해 기각 |
| 엔드포인트 | `/api/v1/members/me/addresses` — POST(201)·GET(200 목록)·PATCH `/{id}`(204)·DELETE `/{id}`(204)·POST `/{id}/default`(204) | 본인 리소스라 `/me` 하위. 새 컨트롤러 `MemberAddressController`(회원 슬라이스) |
| 목록 | 페이징 없음, 기본 우선·최신순(`is_default desc, id desc`) | 최대 10개라 페이지 불필요 |
| 조회 범위 | 모든 단건 접근은 `findByIdAndMemberId` — 타인 배송지는 404 | 주문 조회의 본인 스코프 선례 |
| 에러 | `MemberErrorCode`에 `ADDRESS_NOT_FOUND`(404)·`ADDRESS_LIMIT_EXCEEDED`(409) 추가 + 대응 예외 2종 | 기존 코드 형식 동일 |
| 서비스 | `MemberAddressReader`·`Appender`·`Modifier`(revise·designateDefault)·`Remover`(purge) | 접미사 표 준수. 컨트롤러 직접 호출(단일 도메인, 파사드 불필요) |
| 마이그레이션 | `member/V4__create_member_address_table.sql` — 컬럼 폭은 주문 주소 DDL과 일치, 논리 FK 인덱스 + 부분 유니크 인덱스, 물리 FK 없음 | 규칙(물리 FK 금지·인덱스는 Flyway) 준수. `SchemaFlywayFactory.migrateAll`이 테스트에 자동 반영 |
| 동시성 수용 | 동시 첫 등록·동시 기본 지정 경합은 부분 유니크 인덱스가 차단(위반 시 500 계열) | 낙관락·재시도 설계는 이 표면의 트래픽 특성상 과설계. 한계로 기록 |

### 완료 기준(테스트 목록)

`MemberAddressControllerTest`(신규):

1. 등록·목록 — 첫 등록은 자동 기본, 둘째는 기본 아님, 목록이 기본 우선 정렬로 2건.
2. 기본 지정 — 둘째를 기본 지정하면 기본이 하나뿐이고 대상이 기본이 된다(단일성).
3. 수정 — PATCH 후 필드가 반영된다.
4. 삭제 — DELETE 후 목록에서 사라진다.
5. 타인 접근 — 다른 회원 토큰의 PATCH·DELETE·기본 지정이 404 `MEMBER_ADDRESS_NOT_FOUND`.
6. 한도 — 10개 등록 후 11번째가 409 `MEMBER_ADDRESS_LIMIT_EXCEEDED`.
7. 미인증 — 401.
8. 엔티티·연관관계 규칙은 기존 아키텍처 테스트(전체 빌드)가 검증.

### 구현 계획

```text
1. MemberAddressControllerTest 작성 → 검증: 컴파일 실패(red)
2. 마이그레이션 V4 + MemberAddress 엔티티 + 리포지토리 + Info + 예외·에러코드 → 검증: 컴파일
3. Reader·Appender·Modifier·Remover + 요청·응답 DTO + 컨트롤러 → 검증: 신규 테스트 통과(green)
4. ./gradlew build → 검증: 아키텍처 테스트 포함 전체 게이트 통과
```

## 설계 리뷰

- 단순함: 파사드·페이징·자동 승격·낙관락 없이 최소 표면. 요청 DTO 1개를 생성·수정에 공용.
- 외과적 수정: 기존 파일 변경은 `MemberErrorCode` 항목 추가뿐. Member 엔티티·체크아웃 무변경.
- 규칙 대조:
  - 엔티티 골격(BaseTimeEntity·Auditing·수동 @Id·@GeneratedValue 금지·전 영속 필드 매핑 애노테이션) — 반영.
  - 물리 FK 금지·논리 참조 UUID·인덱스는 Flyway — 반영.
  - 하드삭제 명시 동사 `purge*` — `MemberAddressRemover.purge`.
  - 벌크 UPDATE는 리포지토리 `@Modifying` 의도 명명(`releaseDefaultByMemberId`) — `claimIssuanceSlot` 선례.
  - 한 트랜잭션 한 애그리거트 규칙과의 긴장: 기본 해제·지정은 같은 테이블의 단일 비즈니스 불변식이라 벌크 UPDATE 허용 규칙으로 정당화. 크로스 애그리거트 아님.
- 지적·반영: 초안의 "요청에 default 플래그" 방식을 기본 지정 액션 분리로 교체(단일성 로직 집중). 이미 기본인 대상 조기 반환을 명시(벌크·dirty checking 상호작용 함정).

## 코드 리뷰

- 추적성: 기존 파일 변경은 `MemberErrorCode` 항목 2건 추가뿐이고 나머지는 전부 신규 파일 — 계획대로 외과적.
- 규칙 대조 결과:
  - 엔티티 골격: `BaseTimeEntity` 상속, 수동 `@Id`(UUIDv7 팩토리), 전 영속 필드 매핑 애노테이션, nullable만 `@Nullable`, `@GeneratedValue` 없음, 시각 필드 미선언.
  - 물리 FK 없음. `member_id` 논리 참조 + Flyway 인덱스. 부분 유니크 인덱스로 기본 단일성 DB 백스톱.
  - 수정은 리포지토리로 조회한 관리 엔티티의 의도 동사(`revise`·`markDefault`)와 dirty checking — `save()` 재호출 없음.
  - 하드삭제는 `Remover.purge`로만 노출(명명 규칙).
  - 벌크 UPDATE(`releaseDefaultByMemberId`)와 dirty checking 상호작용은 이미-기본 조기 반환으로 방어(주석 명시).
  - 컨트롤러는 Info 경유로만 응답 — 엔티티 모듈 밖 비노출(아키텍처 테스트가 강제).
- 고아 검사: 신규 import 전부 사용. 데드 코드 없음.
- 지적 없음. 수정 사항 없이 통과.
