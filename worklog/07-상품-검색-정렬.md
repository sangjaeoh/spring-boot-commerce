# 작업 7 — 상품 검색·정렬

## 설계

### 가정·해석

- 검색은 DB LIKE 수준의 상품명 부분 일치(대소문자 무시)다. 검색엔진·형태소 분석·패싯은 범위 밖(WORKPLAN). LIKE 특수문자(`%`·`_`)는 이스케이프하지 않는다 — LIKE 수준 검색의 수용 한계로 기록.
- "가격순"의 가격은 카탈로그 대표가와 같은 정의 — ACTIVE 변형의 최저가. 정렬은 페이지 정합을 위해 파사드 후처리가 아니라 쿼리에서 수행한다(기존 노출 술어와 같은 원칙).
- QueryDSL은 이 저장소에 아직 배선돼 있지 않다. 한 기능을 위해 배선을 추가하는 것은 과설계라, 조건 조합(키워드 유무 × 정렬 3종)은 `@Query` JPQL의 nullable 파라미터 술어 + 정렬별 메서드로 처리한다.

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| API | `GET /api/v1/products?keyword=&sort=` — keyword 선택, sort는 `LATEST`(기본)·`PRICE_ASC`·`PRICE_DESC` | 기존 엔드포인트 확장(WORKPLAN). 새 검색 엔드포인트 분리는 계약 중복 |
| sort enum | `com.commerce.api.facade.ProductSort` — 파사드 소유 | 파사드가 web 타입을 아는 방향 역전 회피(`DirectOrderLine` 선례). 도메인 enum으로 두면 위치(entity·info) 어느 쪽도 부자연 |
| 파사드 | `getCatalogPage(keyword, sort, pageable)`로 시그니처 변경, sort 분기로 Reader 의도 메서드 호출 | 호출처가 컨트롤러·파사드 테스트뿐이라 오버로드 불필요 |
| Reader | `getExposedPage(keyword, pageable)`(최신순, 기존 확장)·`getExposedPageOrderByPriceAsc/Desc(keyword, pageable)` | 의도 메서드 3개가 enum 파라미터보다 도메인 표면이 명확 |
| 리포지토리 | `findExposedPage`에 `:keyword is null or lower(p.name) like ...` 술어 추가. 가격순은 ACTIVE 변형 조인 + `group by p` + `order by min(v.price)`(명시 countQuery) | 노출 술어·검색·정렬을 쿼리에 둬야 페이지 크기·총건수가 어긋나지 않음 |
| 위험 | `min(v.price)`가 `@Convert`(Money) 속성 집계라 Hibernate가 거부할 수 있음 | 실패 시 대안: 네이티브 쿼리(컨버터 집계는 HQL 처리 불가 사유) — 구현 단계에서 판정 |
| 응답 | 변경 없음(`ProductPageResponse`) | 정렬·검색은 목록 내용만 바꿈 |

### 완료 기준(테스트 목록)

`ProductControllerTest`에 추가(공유 DB 오염 회피를 위해 고유 토큰 키워드로 스코프):

1. 키워드 매칭 — 토큰 포함 상품만 반환, 미포함 상품 제외.
2. 키워드 비매칭 — 존재하지 않는 토큰은 빈 목록.
3. 숨김 제외 유지 — 토큰 상품을 숨기면 키워드 검색에서도 제외.
4. 정렬 — 같은 토큰의 상품 3건(가격 30000·10000·20000)에 대해 `PRICE_ASC`는 저가 우선, `PRICE_DESC`는 고가 우선, 기본(LATEST)은 최신 등록 우선.

### 구현 계획

```text

2. ProductSort + 리포지토리 쿼리 3종 + Reader 3메서드 + 파사드 분기 + 컨트롤러 파라미터 → 검증: green
```

## 설계 리뷰

- 단순함: 새 타입은 enum 1개. QueryDSL 배선·검색 인프라 없이 기존 쿼리 확장.
- 외과적 수정: `findExposedPage`·`getExposedPage`·`getCatalogPage` 시그니처 확장의 파급은 각 1~2 호출처로 한정.
- 규칙 대조:
  - `@Query` 사용 기준(파생 쿼리로는 조건·조인 표현 불가) 충족. 정렬별 메서드는 이름 비대 대신 의도 명확성 선택.
  - 페이징 규칙(PaginationRequest·1-based 보정) 기존 경로 유지.
  - 컨트롤러 파라미터는 문자열·enum이라 int `@RequestParam` 금지 규칙과 무관, `@Parameter(description)` 명시.
- 트레이드오프: 가격순 쿼리는 조인·집계로 최신순보다 무겁다. 카탈로그 규모(연습 프로젝트)에서 수용하고, 인덱스 최적화는 하지 않는다.
- 지적·반영: 초안의 도메인 sort enum을 파사드 소유로 이동(위치 부자연 해소). 파사드 후정렬안은 페이지 정합 문제로 기각.

## 코드 리뷰

- 추적성: 신규 파일 1(enum) + 변경 5파일(리포지토리·Reader·파사드·컨트롤러·테스트) 전부 이 작업 범위. 파사드 테스트 호출처 5곳은 기계적 시그니처 갱신.
- 구현 중 판정: `min(v.price)` — `@Convert`(Money) 속성 집계는 Hibernate 6에서 정상 동작 확인(설계의 위험 항목 해소, 네이티브 쿼리 불필요). null 키워드 파라미터는 PostgreSQL bytea 오추론으로 실패해 `cast(:keyword as string)`으로 타입 고정 — 이유를 라인 주석으로 명시.
- 리뷰 반영 2건: cast 이유 주석 추가, `EXPOSED_COUNT_QUERY` 상수를 사용처 앞으로 이동(선언-후-사용 가독성).
- 규칙 대조 결과:
  - `@Query` 사용 기준 충족(조인·집계·nullable 술어는 파생 쿼리 불가). group by 쿼리에 명시 countQuery로 페이지 총건수 정합.
  - 컨트롤러 파라미터는 String·enum + `@Parameter(description)` — int `@RequestParam` 금지 규칙 무관.
  - enum은 파사드 소유(`DirectOrderLine` 선례), web→facade 방향 유지.
  - 가격 정렬의 tie-breaker(`p.id desc`)로 결정적 순서 보장.
- 고아 검사: 신규 import(`Nullable`·`RequestParam`·`ProductSort`) 전부 사용. 데드 코드 없음.
