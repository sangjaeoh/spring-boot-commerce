# 작업 3-1 — 교환·반품 요청

## 설계

### 가정·해석

- 현재 관리자 refund는 "요청 없는 즉시 환불"이다. 이 작업은 그 앞에 사용자 요청·관리자 심사 단계를 붙이는 것이지 환불 실행 경로를 새로 만드는 게 아니다 — 승인 = 기존 환불 경로 실행.
- 반품 상태는 결제 축(OrderStatus)을 확장하지 않고 별도 nullable 축으로 둔다. 반품 요청 중에도 주문은 PAID·DELIVERED 그대로다 — 거절 시 "원복"이 상태 되돌리기가 아니라 마커 해제가 되어 전이 규칙이 단순해진다. `cancelRequestedAt` 마커와 같은 계보의 설계다.
- 요청 사유는 `RefundReason`을 재사용한다. 반품 요청은 승인되면 그 사유 그대로 환불되므로 별도 enum은 같은 값의 복제다. CS_MANUAL을 사용자가 고를 수 있게 되는 어색함은 수용한다(트레이드오프).
- 요청 시각 `returnRequestedAt`만 두고 거절·완료 시각은 두지 않는다 — 완료 시각은 `refundedAt`이 이미 있고, 거절 시각 요구가 없다.
- 반품 요청 이력(거절된 요청 보관)은 만들지 않는다. 단일 활성 요청이면 충분하고, 거절 후 재요청은 덮어쓰기로 허용한다.

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 부분 반품(라인 단위) | 제외. 전체 주문 단위만 | 범위 지시가 "기존 환불 경로 재사용"인데, 부분 반품은 부분 환불 산식(할인·배송비 안분)·라인별 상태·부분 재고 복원이 필요해 재사용이 성립하지 않는다. 완료 기준 4건 모두 주문 단위로 검증 가능 |
| 교환 | 별도 모델 없음. 반품 후 재주문으로 단순화 | 교환 전용 상태·재고 예약·차액 정산이 전부 새로 생기는데 완료 기준에 교환 검증이 없다 |
| 상태 모델 | `ReturnStatus` nullable 축: `REQUESTED`·`REJECTED`·`COMPLETED` | OrderStatus 확장(RETURN_REQUESTED 삽입)은 결제 축에 비종료 상태를 끼워 넣어 기존 전이 가드 전부를 재검토하게 만든다. "승인됨(APPROVED)" 영속 상태는 두지 않는다 — 승인은 동기적으로 환불을 완결하므로 저장할 중간 상태가 없다 |
| 엔티티 필드 | `returnStatus`, `returnRequestedAt`, `returnReason` (전부 @Nullable) | 요청 사유(returnReason)와 환불 확정 사유(refundReason)는 별도 유지 — 거절되면 환불 없이 요청 사유만 남아야 한다 |
| 전이 소유 | `Order.requestReturn(reason, now)`: PAID·DELIVERED 필수, REQUESTED 중복 거부. `Order.rejectReturn()`: REQUESTED 필수. `Order.refund()`: REQUESTED였다면 COMPLETED 마킹 추가 | 상태 전이는 엔티티 의도 동사 소유(entity-persistence.md). 거절 후 재요청은 허용(REJECTED → REQUESTED) |
| 에러코드 | `OrderErrorCode`: RETURN_NOT_ALLOWED(409)·RETURN_ALREADY_REQUESTED(409)·RETURN_NOT_REQUESTED(409). `ApiErrorCode`: ORDER_RETURN_NOT_REQUESTED(409) | 도메인 전이 위반은 도메인 코드, 파사드 조율 게이트는 앱 코드 — refund의 requireRefundable 선례 |
| 사용자 API | `POST /api/v1/orders/{orderId}/return-request` (body: reason, 204). `OrderModifier.requestReturn(orderId, memberId, reason)` — findByIdAndMemberId로 소유 검증(타인 주문 404) | 단일 애그리거트 변경이라 파사드 불요. cancel처럼 파사드를 두는 건 교차 도메인이 없는 이 흐름엔 과설계 |
| 관리자 승인 | `POST /api/v1/admin/orders/{orderId}/return-approval` (204) → `OrderRefundFacade.approveReturn(orderId)`: REQUESTED 확인 후 기존 refund(orderId, returnReason) 실행. COMPLETED면 멱등 통과 | 새 파사드 대신 기존 반품 환불 파사드에 승인 진입점 추가 — 클래스 책임("전체 반품·전액 환불 흐름 조율")과 일치하고 재사용 지시를 문자 그대로 충족 |
| 관리자 거절 | `POST /api/v1/admin/orders/{orderId}/return-rejection` (204) → `OrderModifier.rejectReturn(orderId)` | 단일 애그리거트 변경, ship·confirmDelivery처럼 컨트롤러 → Modifier 직결 |
| 조회 노출 | `OrderInfo`·`OrderResponse`에 returnStatus·returnRequestedAt·returnReason 추가 | 파사드 게이트가 Info의 returnStatus를 읽어야 하고, 요청한 사용자가 요청 상태를 볼 수 없으면 흐름이 완결되지 않는다. 관리자용 반품 요청 목록 필터는 완료 기준에 없어 제외 |
| 마이그레이션 | `V10__add_orders_return_columns.sql`: return_status VARCHAR(30)·return_requested_at TIMESTAMPTZ·return_reason VARCHAR(30) | V2(refund 컬럼) 형식 답습 |

### 완료 기준(테스트 목록)

`OrderTest`(엔티티):

1. 배송 완료 결제 주문 반품 요청 → REQUESTED·사유·시각 기록.
2. 배송 완료 아님(PREPARING·SHIPPED)·미결제(PENDING)·환불 완료(REFUNDED) 주문 요청 거부. — 완료 기준 "배송 완료 아닌 주문 거부"
3. REQUESTED 중복 요청 거부. — 완료 기준 "중복 요청 거부"
4. 거절: REQUESTED → REJECTED, 주문은 PAID·DELIVERED 유지. — 완료 기준 "거절 시 원복"
5. 요청 없는(또는 이미 거절된 것 재차) 거절 거부. — 완료 기준 "허용되지 않은 전이 거부"
6. 거절 후 재요청 허용.
7. REQUESTED 주문 refund() → returnStatus COMPLETED.

`OrderRefundFacadeTest`(통합):

8. 요청 → 승인 → REFUNDED·COMPLETED·결제 취소·재고·쿠폰 복원. — 완료 기준 "승인 시 환불·재고 복원"
9. 요청 없는 주문 승인 409(ORDER_RETURN_NOT_REQUESTED).
10. 거절 → 주문 PAID·DELIVERED 유지·재고 그대로, 이후 재요청·승인 가능.

`OrderControllerTest`(웹):

11. 반품 요청 204 → 상세 조회에 returnStatus REQUESTED 노출. 타인 주문 404.

`OrderAdminControllerTest`(웹):

12. 승인 204(환불 완결), 거절 204, 요청 없는 승인 409.

## 설계 리뷰

- 단순함: 부분 반품·교환 모델·APPROVED 영속 상태·요청 이력 엔티티·전용 사유 enum·관리자 반품 목록 필터·신규 파사드를 전부 기각했다. 남는 표면은 엔티티 전이 3곳(신규 2 + refund 1줄)·Modifier 2 메서드·파사드 1 메서드·엔드포인트 3개로, 완료 기준 4건과 1:1로 추적된다. 시니어 관점 과설계 요소 없음.
- 외과적 수정: 기존 파일 변경은 Order(필드 3·메서드 2·refund 1줄)·OrderErrorCode·ApiErrorCode·OrderInfo·OrderResponse·OrderModifier·OrderRefundFacade·컨트롤러 2곳뿐이다. 기존 refund·cancel 경로의 동작은 바꾸지 않는다(refund의 COMPLETED 마킹은 추가 동작이며 기존 전이 가드는 그대로).
- 규칙 대조:
  - 상태 전이는 엔티티 의도 동사가 소유, 전이 불가 시 도메인 예외(entity-persistence.md 상태 전이) — requestReturn·rejectReturn이 준수.
  - 새 enum은 `@Enumerated(EnumType.STRING)`, nullable 값만 `@Nullable` — 준수.
  - 단일 애그리거트 전이는 Modifier가 엔트리, 여러 도메인 조율은 파사드(architecture.md 트랜잭션 경계) — 요청·거절은 Modifier, 승인(환불 조율)은 파사드로 분리 일치.
  - 소유 검증을 Modifier의 member 스코프 find로 처리하는 선례 확인: `ReviewModifier.findByIdAndMemberId`. 파사드 없는 requestReturn 설계 유지.
  - 적용된 마이그레이션 수정 금지 — V10 신규 추가로 준수.
- 지적·반영: 초안은 승인 게이트에서 `order.returnReason()`이 null일 수 없다고 가정했다 — NullAway가 Info의 @Nullable 반환을 거부하므로 게이트에서 null 검사와 REQUESTED 검사를 함께 수행하는 형태로 수정. 초안의 사용자 요청 401/404 구분은 기존 컨트롤러 계약(@Authenticated + 소유 스코프 조회 404)을 그대로 따르는 것으로 확정.

## 코드 리뷰

- 추적성: 신규는 ReturnStatus·OrderReturnRequest·V10 마이그레이션 3파일. 기존 변경은 Order(필드 3·메서드 2·refund 마킹)·OrderErrorCode(+3)·OrderInfo(+3 필드)·OrderModifier(+2 메서드)·OrderRefundFacade(approveReturn)·ApiErrorCode(+1)·OrderController(핸들러+OrderModifier 주입)·OrderAdminController(핸들러 2)·OrderResponse(+3 필드)와 테스트 3파일 — 전부 이 작업으로 추적된다. 무관 리팩터 없음.
- TDD 기록: 엔티티 8건 red(컴파일 실패) → 도메인 구현 → green. 파사드 4건 red → approveReturn·requestReturn·rejectReturn 구현 → green. 웹 7건 red → 컨트롤러·DTO·응답 필드 → green.
- 규칙 대조 결과:
  - 전이는 엔티티 의도 동사 소유, 실패는 OrderStatusException — requestReturn/rejectReturn/refund 마킹 모두 엔티티 안. Modifier는 트랜잭션 경계만 소유.
  - 파사드는 트랜잭션을 열지 않고 조율만 한다 — approveReturn은 게이트 후 기존 refund 경로 위임(재사용 지시 충족). REQUESTED→COMPLETED 마킹은 주문 환불 전이와 같은 트랜잭션이라 원자적이다.
  - 승인 재시도 의미론은 기존 refund와 동일 계보: PG 취소 후 전이 실패 시 REQUESTED가 남아 재시도 가능, 완결 후 재호출은 멱등 통과. 전이 커밋과 복원 사이 크래시의 복원 누락 갭은 기존 refund의 알려진 트레이드오프로 이 작업에서 바꾸지 않는다(외과적 수정).
  - 소유 검증은 findByIdAndMemberId 스코프 조회(ReviewModifier 선례), 타인 주문 404 미존재 취급 — 기존 계약과 일치.
  - 어드민 표면: 기존 OrderAdminController(@Admin·admin 패키지·프리픽스)에 핸들러만 추가. @Operation·@ApiResponse·@Schema 게이트 대상 전부 선언.
  - 마이그레이션: 신규 V10, 물리 FK 없음, 상태 컬럼 VARCHAR(30) — V2 형식 일치.
- 고아 검사: 신규 import 전부 사용(OrderController의 OrderModifier·OrderReturnRequest, 파사드의 ReturnStatus, 테스트의 RefundReason·OrderNotFoundException·ReturnStatus). 데드 코드 없음.
