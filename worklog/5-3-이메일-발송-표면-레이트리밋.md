# 작업 5-3: 이메일 발송 표면 레이트리밋

## 설계

### 현황

- `RateLimitConfig`(app-api)는 로그인(`login:`)·가입(`signup:`) 두 표면만 `LoginRateLimitFilter`로 보호한다. 한도 10회/5분, IP 고정창.
- `POST /api/v1/auth/password-reset-request`는 무인증으로 메일 발송(재설정 토큰)을 유발하는데 무제한 호출이 가능하다.
- `LoginRateLimitFilter`(common-web)는 scope 파라미터화된 재사용 가능한 필터다 — 429 problem+json, Retry-After, 저장소 장애 시 503 fail-closed. 계약은 `LoginRateLimitFilterTest`(common-web 단위)가 소유한다.
- 선례: `AuthControllerTest.loginRejectsAfterTooManyAttempts` — 전용 클라이언트 IP로 한도를 채워 429·코드·시큐리티 헤더 단언.

### 결정

- 보호 표면(선택지: password-reset-request만 vs email-verification 등 포함): `password-reset-request`만 보호한다.
  - 목표 문구가 "이메일 발송을 유발하는 표면"이다. `POST /api/v1/members/email-verification`과 `POST /api/v1/auth/password-reset`은 토큰을 소비할 뿐 메일을 보내지 않는다 — 이메일 폭탄 표면이 아니다. 토큰 추측 위협은 UUID 불투명 토큰의 공간으로 방어된다(worklog 3-2).
  - 가입(메일 발송 유발)은 이미 `signup:` scope가 보호한다.
- scope: `reset:` 접두, 코드 `TOO_MANY_PASSWORD_RESET_REQUESTS`. `RateLimitConfig`에 `RESET_SCOPE` 상수와 필터 등록 빈 하나 추가 — 기존 `register` 헬퍼 재사용.
- 한도·창(재검토): 기존값 10회/5분 유지. 같은 위협 모델(무인증 POST 남용)이고 표면별 차등 근거가 없다. 상수 공유로 단순.
- 회원 존재 열거: 재설정 요청은 이미 항상 204(존재 비노출, worklog 3-2)라 열거는 응답으로 불가능하다. 레이트리밋은 이메일 폭탄(실존 회원 주소로의 반복 발송) 차단이 주 효과다.

### 가정·트레이드오프

- fail-closed 계약(저장소 장애 503)은 `LoginRateLimitFilter` 클래스 계약으로 `LoginRateLimitFilterTest`의 단위 시나리오가 이미 소유한다. 신규 scope는 같은 필터 클래스의 인스턴스라 새 행동이 없다 — 상위 레벨 반복 금지 규칙에 따라 통합 테스트를 추가하지 않고, 완료 기준 3은 기존 테스트 통과로 충족한다.
- IP 고정창의 한계(프록시 뒤 다수 사용자 공유 IP, 분산 공격)는 기존 로그인·가입과 동일한 수용 사항이다.

### 완료 기준(검증 가능한 테스트 목록 — TDD)

웹 통합 `AuthControllerTest`(app-api):

1. 한 창 안에서 재설정 요청이 한도를 넘으면 429 `TOO_MANY_PASSWORD_RESET_REQUESTS`로 거부된다.
2. 로그인 시도로 한도를 소진해도 같은 IP의 재설정 요청은 통과한다(표면별 카운터 분리).

기존 스위트:

3. `LoginRateLimitFilterTest`의 503 fail-closed 시나리오 통과 유지(완료 기준 3 귀속).

## 설계 리뷰

AGENTS.md 작업 원칙·docs 규칙 대조:

- (확인) 단순함 — scope 상수 1개 + 필터 등록 빈 1개. 필터·스토어·헬퍼 전부 재사용, 신규 타입 없음.
- (확인) 카운터 분리 방향 — 완료 기준 문구 "로그인 시도가 재설정 한도를 소모하지 않음"은 시나리오 2(로그인 한도 소진 후 같은 IP 재설정 통과)와 정확히 대응한다. 역방향 반복은 독립 카운터 증명에 불필요.
- (확인) 테스트 오염 방지 — 두 시나리오 모두 전용 클라이언트 IP를 쓴다(로그인 429 선례). 시나리오 1은 미존재 이메일로 한도를 채워 실제 메일 발송 없이 카운터만 소모한다(재설정 요청은 항상 204).
- (확인) fail-closed 귀속 — 같은 필터 클래스 재사용이라 새 행동 없음. 테스트 레벨 규칙(같은 행동 상위 레벨 반복 금지)에 따라 기존 단위 시나리오 통과로 충족. 리뷰에서 재론 시 근거 제시.
