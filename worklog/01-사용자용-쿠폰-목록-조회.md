# 작업 1 — 사용자용 쿠폰 목록 조회

## 설계

### 가정·해석

- "발급 가능한 쿠폰"의 의미는 `Coupon.checkIssuable(now)` 및 발급 한도 선점(`claimIssuanceSlot`)과 정합해야 한다. 즉:
  - `status = ACTIVE`
  - 발급 가능 기간 내 (`validFrom <= now <= validUntil`, 양 끝 포함 — `ValidityPeriod.isValidAt`과 동일)
  - 한도 미소진 (`maxIssuance is null or issuedCount < maxIssuance`)
- 한도 소진 필터를 포함하는 이유: 목록에 노출해 놓고 발급 클릭 시 409를 주는 것보다 목록에서 제외하는 쪽이 사용자 계약에 맞다. 트레이드오프: "소진됨" 배지 노출 같은 마케팅 요구는 지원하지 않는다(요구 없음).

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 엔드포인트 | `GET /api/v1/coupons` — 기존 `CouponController`에 핸들러 추가 | 리소스 URL 일치. 새 컨트롤러 불필요 |
| 인증 정책 | 인증 필요(클래스 레벨 `@Authenticated` 유지) | 발급 자체가 로그인 전제. 공개하려면 `SecurityConfig` 공개 경로 추가·`@Anonymous` 배선이 필요해 표면이 늘어남. 공개 요구가 생기면 그때 전환 |
| 조회 경로 | 컨트롤러 → `CouponReader` 직접 호출, 파사드 없음 | 단일 도메인 조회. `OrderController`·`CouponAdminController` 선례와 동일 |
| 리포지토리 | `CouponRepository.findIssuablePage(Instant now, Pageable)` — `@Query` JPQL | 3중 조건 + `or`라 파생 쿼리는 이름 비대(docs/architecture.md의 @Query 사용 기준). `ACTIVE`는 쿼리 안에 fully-qualified enum 리터럴로 고정 — "발급 가능" 의미를 메서드가 소유 |
| 정렬 | `order by c.id desc` | UUIDv7이 시간순이라 최신 등록순. `findPage`와 동일 패턴 |
| 서비스 | `CouponReader.getIssuableCoupons(Pageable)` 추가, `Clock` 주입 | `IssuedCouponReader` 등 도메인 서비스의 기존 Clock 주입 패턴. 시각 기준을 호출자가 아닌 Reader가 소유 |
| 응답 DTO | 사용자 슬라이스 `web/v1/coupon/response/`에 `IssuableCouponResponse`·`IssuableCouponPageResponse` 신규 | admin `CouponResponse`는 운영 필드(`issuedCount`·`status`·`maxIssuance`·타임스탬프)를 노출하므로 공유 부적합. simple name 충돌 회피를 위해 `Issuable` 접두 |
| 응답 필드 | id, name, discountType/amount/percent/maxCap, minOrderAmount, validFrom, validUntil, usageValidDays | 사용자 의사결정에 필요한 것만. 운영 카운트는 제외 |
| 인덱스 | 추가하지 않음 | 쿠폰 정책 테이블은 소규모(정책 수 단위). 미리 최적화하지 않는다 |

### 완료 기준(테스트 목록)

`CouponControllerTest`에 추가:

1. 발급 가능 쿠폰만 노출 — ACTIVE·기간 내 쿠폰만 반환되고, DISABLED·기간 밖·한도 소진 쿠폰은 제외된다.
2. 페이징 — 2건 생성 후 size=1 요청 시 1건 + 페이지 메타(totalElements=2), 최신 등록순.
3. 미인증 401 — 토큰 없이 요청 시 401 UNAUTHENTICATED.
4. 응답 필드 계약 — name·discountType·minOrderAmount 등 jsonPath 검증(1번 테스트에 포함).

### 구현 계획

```text
1. CouponControllerTest에 실패 테스트 4건 추가 → 검증: 신규 테스트 실패(404/컴파일 에러) 확인
2. CouponRepository.findIssuablePage + CouponReader.getIssuableCoupons(Clock 주입) → 검증: 컴파일
3. IssuableCouponResponse·IssuableCouponPageResponse + CouponController 핸들러(@Operation·@ApiResponses) → 검증: 신규 테스트 통과
4. ./gradlew build → 검증: Spotless·NullAway·Error Prone·전체 테스트·아키텍처 테스트 통과
```

## 설계 리뷰

- 단순함: 새 모듈·파사드·인덱스 없이 기존 슬라이스에 핸들러 1개, 리포지토리 메서드 1개, Reader 메서드 1개, DTO 2개. 과설계 요소 없음.
- 외과적 수정: 기존 `findPage`·admin 응답은 건드리지 않는다. `CouponReader` 생성자에 Clock 파라미터가 추가되나 스프링 주입이라 프로덕션 호출처 영향 없음 — 직접 생성하는 테스트가 있으면 구현 단계에서 함께 수정(내 변경이 만든 영향).
- 규칙 대조:
  - 리포지토리 `@Query` 사용 기준 충족(파생 쿼리 이름 비대). `custom/` 하위 패키지 없음.
  - 경계: `Page<Entity>` → `Page<Info>` 변환은 Reader의 `@Transactional(readOnly=true)` 안에서 완료.
  - 컨트롤러 핸들러 `@Operation`·`@ApiResponse`·`@Schema` 필수(아키텍처 테스트 강제) 반영.
  - 페이징은 `PaginationRequest`/`PaginationResponse` 사용, 1-based↔0-based 보정 규칙 준수.
  - DTO는 record, 컬렉션 방어적 복사(`List.copyOf`) — admin `CouponPageResponse` 패턴 동일.
- 지적·반영: 초안에서 상태 필터를 `:status` 파라미터로 받으려 했으나, "발급 가능" 의미가 호출자에게 새므로 쿼리 내 enum 리터럴 고정으로 변경. 소진 필터 포함 여부는 사용자 계약 관점에서 포함으로 확정.

## 코드 리뷰

- 추적성: 변경 4파일 + 신규 2파일 전부가 이 작업 범위다. `CouponController` 클래스 주석·`@Tag` 갱신은 엔드포인트 표면 변화에 따른 필수 수정.
- 규칙 대조 결과:
  - `@Query` JPQL의 enum은 fully-qualified 리터럴로 고정 — "발급 가능" 의미를 리포지토리 메서드가 소유.
  - 핸들러에 `@Operation`·`@ApiResponses` 명시, 페이징은 `PaginationRequest`/`PaginationResponse` 사용(아키텍처 테스트 강제 규칙 충족).
  - DTO 전 컴포넌트 `@Schema` 선언, 컬렉션 `List.copyOf` 방어적 복사 — admin `CouponPageResponse` 패턴과 동일.
  - `Page<Entity>` → `Page<Info>` 변환이 Reader의 `@Transactional(readOnly=true)` 안에서 완료 — OSIV off 경계 규칙 충족.
  - nullable 할인 필드는 `@Nullable` 명시(NullAway).
- 고아 검사: 신규 import 전부 사용됨. 기존 import 중 미사용 전환 없음.
- 지적 없음. 수정 사항 없이 통과.
