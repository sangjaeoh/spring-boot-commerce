# 작업 5-7: 부분 반품 환불

## 설계

### 현황

- 반품은 주문 전체 단위: `Order.requestReturn`(PAID·DELIVERED, returnStatus REQUESTED) → 관리자 `OrderRefundFacade.approveReturn` → `refund`(전액 PG 취소·REFUNDED 전이·전 라인 재고·쿠폰 복원) / `rejectReturn`.
- 작업 6이 만든 재사용 자산: `OrderLineStatus` 축, 라인 `refundAmount` 확정 기록, 루트 `refundedAmount` 누계(낙관락 직렬화), 안분 산식(내림 + 마지막 활성 라인 잔액 전액), `PaymentProcessor.cancelPartially`(PG 멱등 키 + 누계 단조 동기화), 3단계 begin→PG→complete 흐름.
- 작업 6 가드: `Order.refund`·`Payment.cancel`은 부분 환불 이력(refundedAmount>0)을 거부한다 — 전체 반품 환불과 부분 이력의 이중 집행은 이미 봉쇄.

### 결정

- 라인 상태 확장: `OrderLineStatus`에 `RETURN_REQUESTED`(요청 대기), `RETURNING`(승인·환불 진행 중 — CANCELLING 미러), `RETURNED`(완결) 추가. 배타는 상태 기계가 자연 강제한다 — 반품 요청·취소 개시 모두 ORDERED 라인에서만 허용되므로 같은 라인에 두 경로가 겹칠 수 없다.
- 라인 반품 사유: `OrderLine.returnReason`(@Nullable RefundReason) 신설 — 요청 시 기록, 거절 시 클리어. 승인 완결의 주문 수렴 사유로도 쓴다.
- 회원 요청: `Order.requestLineReturn(lineId, reason, now)` — 가드: PAID·DELIVERED(전체 선례), 라인 ORDERED(재요청·취소 라인·반품 진행 라인 거부), 주문 전체 반품 미진행(returnStatus != REQUESTED). 라인 → RETURN_REQUESTED.
- 관리자 거절: `Order.rejectLineReturn(lineId)` — RETURN_REQUESTED → ORDERED, 사유 클리어(거절 후 재요청 허용 — 전체 선례 계승).
- 관리자 승인(3단계 — 작업 6 미러): `beginLineReturn(lineId)` — RETURN_REQUESTED 가드, 환불액 확정(공유 산식 `computeLineRefund`: 환불 미확정 라인이 마지막이면 결제 잔액 전액, 아니면 라인금액−안분할인), 라인 → RETURNING, `refundedAmount` 누적(직렬화) → PG `cancelPartially`(멱등 키 lineId — 라인 배타라 취소와 키 충돌 없음) → `completeLineReturn(lineId, now)` — RETURNED, 전 라인 종결 시 수렴.
- 수렴 규칙(범위 확정): 전 라인이 종결(CANCELLED 또는 RETURNED)이고 RETURNED가 하나라도 있으면 주문 REFUNDED(refundedAt·refundReason=마지막 완결 라인의 returnReason). 전부 CANCELLED인 수렴은 작업 6의 CANCELLED 유지 — 취소 수렴은 출고 전, 반품 수렴은 배송 후라 혼합 종결의 마지막은 항상 반품 경로다(completeLineCancellation 무변경 근거).
- 재반품 허용 규칙(범위 확정): 부분 반품 후 잔여 ORDERED 라인의 새 반품 요청은 허용한다. RETURNED 라인 재요청은 거부한다(완료 기준).
- 전체 반품↔라인 반품 상호 배제: `requestReturn`(전체)에 두 가드 추가 — 부분 환불 이력(refundedAmount>0) 거부(PARTIALLY_CANCELLED), 라인 반품 진행(RETURN_REQUESTED·RETURNING 존재) 거부(RETURN_ALREADY_REQUESTED). 역방향은 requestLineReturn의 returnStatus 가드.
- 표면: 회원 `POST /api/v1/orders/{orderId}/lines/{lineId}/return-request`(기존 `OrderReturnRequest` 사유 본문 재사용), 관리자 `POST /api/v1/admin/orders/{orderId}/lines/{lineId}/return-approval`·`return-rejection`(전체 선례 미러). 승인 파사드는 `OrderRefundFacade.approveLineReturn` — begin→PG→complete→그 라인 재고 복원→수렴 시 쿠폰 복원(작업 6 규칙). 이미 RETURNED 라인 승인 재호출은 멱등 통과, RETURNING은 재개.
- 마이그레이션: ordering `V3__add_order_line_return_reason.sql` — `return_reason VARCHAR(30)`.
- 에러코드: 전부 재사용(RETURN_NOT_ALLOWED·RETURN_ALREADY_REQUESTED·RETURN_NOT_REQUESTED·INVALID_ORDER_STATE_TRANSITION·ORDER_LINE_NOT_FOUND·PARTIALLY_CANCELLED). 신규 없음.

### 가정·트레이드오프

- Payment 수렴: 전 라인 환불 도달 시 `syncPartialRefund`가 CANCELLED로 전이한다 — 주문 REFUNDED와 결제 CANCELLED 조합은 전체 반품(refund→Payment.cancel→CANCELLED)과 동일 결말.
- complete 커밋 후 크래시 시 재고·쿠폰 복원 유실 창은 작업 6과 동일한 수용.
- 배송비: 부분 반품도 마지막 라인 잔액 전액 규칙이 흡수(반품 배송비 차감 정책은 요청 밖).

### 완료 기준(검증 가능한 테스트 목록 — TDD)

단위 `OrderTest`:

1. 라인 반품 요청·승인(begin/complete)이 환불액을 공유 산식으로 확정하고 라인을 RETURNED로 완결한다.
2. 이미 반품된(RETURNED) 라인·취소된(CANCELLED) 라인·배송 전 주문의 반품 요청은 거부된다(재요청·배타 전수).
3. 전 라인 반품 완결 시 주문이 REFUNDED로 수렴한다. 혼합(취소 1 + 잔여 반품) 종결도 REFUNDED다.
4. 거절은 라인을 ORDERED로 되돌리고 재요청이 가능하다.
5. 전체 반품 요청은 부분 환불 이력·라인 반품 진행 중이면 거부된다.

7. 전 라인 반품 승인 시 주문 REFUNDED 수렴·결제 CANCELLED·쿠폰 복원.

웹 통합:

## 설계 리뷰

AGENTS.md 작업 원칙·docs 규칙 대조:

- (확인) 재사용 최대화 — 산식·누계 직렬화·PG 부분 환불·3단계 흐름·복원 규칙 전부 작업 6 자산 재사용. 신규 에러코드 0, 신규 컬럼 1(returnReason), 마이그레이션 1.
- (반영) 마지막 라인 판정 통일 — 작업 6의 `isLastOrderedLine`(ORDERED 1개)을 "환불 미확정(refundAmount 없음) 라인 1개" 판정으로 일반화한다. 취소 시점엔 두 판정이 등가(반품 상태는 배송 후에만 존재)라 행동 불변이고, 반품 시점엔 RETURN_REQUESTED 라인이 미확정으로 정확히 집계된다. 작업 6 단위 테스트가 회귀를 가드한다.
- (확인) 배타의 강제 방식 — 별도 배타 검사 없이 "두 경로 모두 ORDERED에서만 개시"라는 상태 기계가 강제한다. 완료 기준의 배타 테스트는 이 성질을 단언한다.
- (확인) 혼합 종결의 수렴 — 취소 수렴(전부 CANCELLED)은 출고 전, 반품 수렴은 배송 후라 혼합 종결의 마지막 완결은 항상 반품 경로다. `completeLineCancellation` 무변경 근거가 성립한다.
- (확인) 전체↔라인 반품 상호 배제 — 양방향 가드(전체 요청의 부분 이력·라인 진행 거부, 라인 요청의 전체 진행 거부). 작업 6의 refund·Payment.cancel 가드가 환불 단계 이중 방어를 이미 소유.
- (확인) 시니어 과설계 자문 — 반품 배송비 차감·부분 수량 반품(라인 내 일부 수량)은 요청 밖으로 배제.

## 코드 리뷰

리뷰어 서브에이전트 파견(적대적 — 상호 배제 매트릭스 전수, 전 스위트 실행). 판정: No(Critical 2건) → 반영 후 재검증.

- (Critical 1, 반영) 집행 단계 상호 배제 부재 — 상호 배제가 요청 단계에만 있어 관리자 직접 전액 환불(`/refund`)과 라인 반품이 순차 조합만으로 이중 재고 복원, 동시 조합으로 PG 이중 환불(전액·부분 멱등 키가 서로 다름)이 성립했다. `Order.refund`에 라인 이력 가드(ORDERED 아닌 라인 존재 시 거부 — 환불 미확정 요청 라인·0원 환불 라인의 refundedAmount 프록시 침묵까지 커버), `beginLineReturn`에 주문 수준 가드(PAID·DELIVERED·전체 반품 미진행) 추가.
- (Critical 2, 강등 수용) `requestReturn`↔`requestLineReturn` 쓰기 스큐 — 라인 쓰기가 루트 버전을 올리지 않아 요청 단계 가드가 직렬화되지 않는다. Critical 1 가드 반영으로 피해가 "양쪽 승인 모두 409, 관리자가 한쪽 거절로 해소"로 강등되어 수용한다(루트 강제 버전 증가는 요청 단계 과잉 직렬화 — 기록). 스큐 잔여 상태는 리플렉션 재현 단위 테스트가 집행 가드를 고정한다.
- (Important 3, 기록) 전체 환불 파사드의 검사-집행 창(PG 전액 환불 후 `Order.refund` 거부 → PAID 주문 × CANCELLED 결제 잔류) — 기존에 수용된 부분 실패 클래스와 동일(재시도·대사 대상), 이중 지급은 라인 승인의 Payment CANCELLED 멱등 통과가 차단. 수용 기록.
- (Minor, 반영) `requestLineReturn`의 미사용 `now` 파라미터 제거. 전체 반품 거절 후 라인 반품 허용 테스트 추가(가드 과강화 회귀 방지).
- (Minor, 기록) 라인 반품 수렴 시 `returnStatus=REJECTED` 이력이 REFUNDED와 공존 가능 — 조회 표면 오해 소지, 범위 밖 기록.
- 리뷰어 확인 사항: 작업 6 Critical 수정의 재개 경로 계승 정확, 멱등 키 충돌 불가(라인 배타), computeLineRefund 일반화 등가 보존.

반영 후 OrderTest 50건·파사드·HTTP 전 스위트 통과, `./gradlew build` 성공.
