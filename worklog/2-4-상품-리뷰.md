# 작업 2-4 — 상품 리뷰

## 설계

### 가정·해석

- 코드베이스에 "구매확정" 상태가 없다. `FulfillmentStatus`는 NOT_STARTED→PREPARING→(ON_HOLD)→SHIPPED→DELIVERED로 끝나고, WORKPLAN이 참고로 지목한 구매확정 상태는 존재하지 않는다. 해석 후보:
  - (a) 배송 완료(DELIVERED)를 구매확정으로 해석 — 추가 상태·전이 없이 기존 종료 상태를 자격 기준으로 쓴다.
  - (b) CONFIRMED 상태와 구매확정 전이(수동 확정 API·자동 확정 정책)를 신설 — 요청되지 않은 기능 신설이라 기각.
  - (a)를 채택한다. 리뷰 자격 = 결제 축 PAID·이행 축 DELIVERED 주문에 해당 상품 라인 보유. 반품 완료(REFUNDED)는 결제 축 조건(PAID)이 자격에서 걸러낸다.
- 작성 단위는 1상품 1리뷰(회원당)다. 주문라인 단위는 재구매마다 같은 회원의 중복 리뷰를 허용해 목록을 오염시키고, 유니크 축도 복잡해진다. 재구매 후 갱신은 수정으로 해결한다. 유니크 (member_id, product_id).
- 상품별 목록은 공개다(미인증 열람). 상품 카탈로그·상세가 공개이므로 리뷰도 공개 콘텐츠다.
- 평균 별점은 노출하지 않는다(단순한 쪽). 필요 시 후속 작업.
- 응답에 작성자 정보를 싣지 않는다 — member 조인 합성 회피 + 개인 정보 최소화. 작성자 표시가 필요해지면 후속 작업에서 합성한다(트레이드오프로 기록).
- 삭제는 하드 삭제다. 리뷰에 soft delete 요구·감사 요구가 없다.

### 완료 기준(테스트 목록)

`ReviewTest`(엔티티 단위):

1. rating 0·6 거부(REVIEW_INVALID_RATING), content 공백·1001자 거부(REVIEW_INVALID_CONTENT), 정상 생성.
2. revise가 값을 바꾸고 같은 검증을 적용한다.

`ReviewPersistenceTest`(@DataJpaTest·testcontainers·flyway validate):

3. 작성 후 상품별 페이지에 보인다(validate 정합 포함).
4. (member, product) 중복 직접 저장은 유니크 위반.
5. 수정·삭제 roundtrip, 최신 리뷰 우선 정렬.

`ReviewControllerTest`(웹 통합):

6. 배송 완료 회원의 작성 201 → 공개 목록에 rating·content 노출. — 완료 기준 "작성·목록"
7. 구매확정(배송 완료) 없는 회원의 작성은 409 API_REVIEW_NOT_ELIGIBLE. — 완료 기준 "자격 거부"(결제만 완료·미배송 포함)
8. 같은 상품 재작성은 409 REVIEW_ALREADY_WRITTEN. — 완료 기준 "중복 정책"
9. 본인 수정 204 → 목록 반영. 타인 리뷰 수정은 404.
10. 본인 삭제 204 → 목록에서 제거.
11. 미인증 목록 조회 200(공개), 미인증 작성 401.
12. rating 6 작성 400.

## 설계 리뷰

- 단순함: 구매확정 기능 신설(b안)·평균 별점·작성자 합성·soft delete를 전부 기각했다. 새 상태·새 전이 없이 기존 DELIVERED를 자격으로 쓴다. "시니어가 과설계라 할까?" — 반대로 (b)안이야말로 과설계다.
- 외과적 수정: order 도메인 변경은 리포지토리 쿼리 1개 + Reader 메서드 1개뿐. 기존 전이·엔티티는 건드리지 않는다.
- 규칙 대조:
  - 교차 도메인 검증 파사드 소유(facade 규칙·WORKPLAN 명시), 파사드는 트랜잭션을 열지 않고 조회(자격)→명령(작성) 순 조율만.
  - 자격 실패 409는 ApiErrorCode(앱 소유 크로스 도메인 정책), 도메인 내부 정책(중복·소유·검증)은 ReviewErrorCode — 소유 경계 일치.
  - `@Anonymous` 메서드 레벨 허용 규칙, 페이징 규칙(PaginationRequest·1-based 보정), `@Query` 사용 기준 충족.
- 트레이드오프: DELIVERED=구매확정 해석은 반품 진행 중(DELIVERED·PAID) 회원의 작성을 허용한다 — 환불 완료 전까지는 구매자였으므로 수용. 작성 후 환불해도 리뷰는 남는다(실서비스 관행과 일치).
- 지적·반영: 초안은 목록을 wishlist처럼 비페이징으로 했다 — 공개·무한 성장 표면이라 페이징으로 격상(불특정 다수 작성). 초안의 REVIEW_INVALID(단일 400)는 사유 분리(RATING·CONTENT)로 교체(CartErrorCode 선례).

## 코드 리뷰

- 추적성: 신규는 domain-review 모듈 전체(엔티티·예외 4·리포지토리·Info·서비스 4·마이그레이션·테스트 2)와 app-api 리뷰 슬라이스(파사드·컨트롤러·DTO 4·웹 테스트). 기존 파일 변경은 배선 5곳 + order 도메인 2파일(리포지토리 쿼리 1·Reader 메서드 1) + ApiErrorCode 상수 1 + SecurityConfig 공개 경로 1 — 전부 이 작업으로 추적된다.
- TDD 기록: 엔티티 테스트 5건·영속 테스트 6건 red(클래스 부재) → 도메인 구현 → green. 웹 테스트 7건 red(웹 계층 부재) → 파사드·컨트롤러·DTO·SecurityConfig → green. 구현 중 수정 1건: 페이지 메타 jsonPath를 실제 컴포넌트명(`$.page.number`)으로 정정.
- 규칙 대조 결과:
  - 요청 DTO는 존재만 검증(@NotNull), 범위 규칙(1~5·1~1000자)은 엔티티가 단독 소유 — 규칙 이중화 회피. 범위 위반은 도메인 400(REVIEW_INVALID_RATING)으로 계약.
  - 교차 도메인 자격은 파사드(트랜잭션 없음, 조회→명령 조율만), 자격 실패는 ApiErrorCode 409 — 소유 경계 일치.
  - order 도메인 추가 쿼리는 `@Query` 사용 기준(파생 쿼리 이름 비대) 충족, 근거 라인 주석. 웹 테스트의 자격 양/음성 경로가 종단 커버.
  - 수정·삭제는 (id, memberId) 스코프 404 — 소유 비노출(`findByIdAndMemberId` 선례). 컨트롤러 `@Anonymous` 메서드 레벨은 목록 GET에만.
  - 아키텍처 테스트 매핑 모듈 가드에 `com.commerce.review` 등록 후 15건 통과, order 도메인 테스트 회귀 없음.
- 고아 검사: 신규 import 전부 사용. 데드 코드 없음.
