# 작업 2-3 — 위시리스트

## 설계

### 가정·해석

- 찜 단위는 상품(productId)이다(WORKPLAN 기본값 재검토 결과 유지). 장바구니는 구매 준비라 변형 단위지만, 찜은 변형 선택 이전의 관심 표현이다. 변형 단위 찜은 옵션 선택을 강제해 UX·모델 모두 복잡해진다.
- 애그리거트는 `WishlistItem` 단건이다(회원당 `Wishlist` 루트 없음). 장바구니의 루트-자식 구조는 수량 합산·전체 비우기 같은 집합 불변식 때문이고, 찜은 (회원, 상품) 유니크 외 불변식이 없다. 루트를 만들면 get-or-create 경합 처리(CartAppender의 재시도 패턴)까지 복제해야 한다 — 평탄한 행이 더 단순하다.
- 찜 추가에 상품 존재·판매중 검증을 하지 않는다. 장바구니의 파사드 게이트(자격·판매중)는 주문 가능성 보장이 목적이고, 찜은 구매 자격과 무관하다(숨김 상품 찜 유지가 자연스러운 UX). 존재하지 않는 UUID 찜은 무해한 행이며, 논리 FK 무결성은 이 저장소 전반에서 DB가 강제하지 않는 기존 태도와 일치한다.
- 목록 응답은 도메인 정보(productId·찜 시각)만 싣는다. 상품명·가격 합성은 크로스 도메인 파사드·숨김/삭제 상품 정책을 끌고 들어온다 — 요청받지 않은 기능이라 범위 밖으로 남기고, 클라이언트는 상품 상세 API로 조인한다.
- 페이지네이션·회원당 찜 수 상한은 두지 않는다(WORKPLAN 미요구). 무한 성장은 수용 한계로 기록한다.

### 완료 기준(테스트 목록)

`WishlistPersistenceTest`(domain, CartPersistenceTest 하네스 선례 — @DataJpaTest·testcontainers·flyway validate):

8. 삭제 204 → 목록에서 제거. 같은 삭제 재요청도 204(멱등).

## 설계 리뷰

- 단순함: 루트 애그리거트·파사드·예외·ErrorCode·페이지네이션·검증 게이트를 전부 기각했다. 남은 것은 엔티티 1·리포지토리 1·서비스 3·Info 1·컨트롤러 1·DTO 3·마이그레이션 1이다. "시니어가 과설계라 할까?" — 오히려 최소 표면이다.
- 외과적 수정: 기존 파일 변경은 배선 4곳(settings·SchemaFlywayFactory·app-api build·app-migration build)뿐이다.
- 규칙 대조:
  - 도메인 모듈 의존은 common-core·common-jpa만(convention.domain-module 허용 범위) — cart와 동일.
  - 서비스 접미사 Reader/Appender/Remover, 엔티티 `create(...)` 정적 팩토리, Info `from(entity)`, 경계 밖 엔티티 비노출 — coding-conventions·architecture 준수.
  - 소프트삭제 없음 → base finder 직접 호출 금지 규칙 비대상. `deletedAt` 없는 엔티티는 그대로 사용(architecture.md 예외).
  - 컨트롤러 int `@RequestParam` 없음, `@Operation`·`@ApiResponse`·`@Schema` 명시 예정(아키텍처 테스트 게이트).
- 트레이드오프: 존재 검증 없음(유령 찜 수용)·목록 비합성(ID만)·무상한 성장 — 전부 단순함을 위해 수용하고 위에 근거를 남겼다.
- 지적·반영: 초안의 `deleteBy...` 반환값(int) 노출은 사용처가 없어 void로 축소. 초안의 `findAllByMemberIdOrderByCreatedAtDesc`는 동일 밀리초 동률이 비결정적이라 UUIDv7 `OrderByIdDesc`로 교체(정렬 의도는 주석으로).
