# 작업 3-4 — 카테고리

## 설계

### 가정·해석

- 카테고리는 상품 탐색 축이다 — 분류(상품→카테고리 ID 지정)와 필터(카테고리별 목록)가 전부이고, 카테고리 자체는 이름 하나를 소유하는 작은 애그리거트다.
- 카테고리 삭제는 "카테고리 목록에서의 제거"다 — 상품의 분류 해제·재배치가 아니다. 삭제된 카테고리 ID를 보유한 상품은 재지정 전까지 그 ID를 유지하지만, 그 ID는 어떤 목록에도 노출되지 않아 탐색 표면에서 자연 소멸한다(대안인 상품 일괄 해제는 한 트랜잭션 한 애그리거트 규칙과 충돌하고, 참조 상품 존재 시 삭제 거부는 재지정 도구 없이는 관리자를 막다른 길에 세운다).
- 상품 수정의 카테고리는 명시 전달이다 — 기존 편집(PATCH)이 name·description을 항상 반영하는 계약이므로 categoryId도 같은 계약(null이면 미분류로 해제)을 따른다.

### 결정 사항

| 항목 | 결정 | 근거·기각 대안 |
|---|---|---|
| 계층 | 계층 없음(플랫 단일 레벨) | WORKPLAN이 "최대 2단계로 단순화할지"를 설계 결정으로 위임 — 완료 기준에 계층 검증이 없고, 계층은 하위 포함 필터·부모 이동·자식 처리 복잡성을 유발한다. 필요 시 parentId 컬럼 추가로 후속 확장 가능 |
| 모델 | domain-product `Category` 애그리거트: id(UUIDv7)·name·deletedAt(소프트삭제). `Product.categoryId`(@Nullable UUID) ID 참조 | 애그리거트 간 ID 참조 규칙. 이름 유니크는 두지 않는다(완료 기준 밖, 부분 유니크 인덱스·경합 처리 비용) |
| 서비스 | CategoryAppender.create(name)→id · CategoryModifier.rename · CategoryRemover.delete(소프트) · CategoryReader.getCategories(활성, 이름순) | CRUD 접미사 규칙 그대로. Remover는 Clock 주입(ProductRemover 선례) |
| 분류 지정 | `Product.assignCategory(@Nullable UUID)` 의도 동사 + `ProductAppender.register(name, desc, categoryId)` 시그니처 확장 + `ProductModifier.assignCategory(productId, categoryId)` — 두 경로 모두 활성 카테고리 존재 검증(null은 해제라 비검증) | 상태 변경은 엔티티 의도 동사. 검증은 같은 모듈 내 CategoryRepository 읽기(읽기는 교차 애그리거트 허용, 쓰기는 상품 하나) |
| 파사드 시그니처 | `registerProduct(name, desc, categoryId, price, options, qty)` 신설, 기존 5-인자는 categoryId=null 위임 오버로드로 유지 | 기존 5-인자 호출처가 테스트 34곳 — 전부 기계적 null 추가로 바꾸는 건 무관 파일 대량 수정(외과적 수정 위반). 위임 오버로드 1개가 최소 변경 |
| 관리자 API | 신규 `CategoryAdminController`(admin 4중 요건): POST `/api/v1/admin/categories`(201 categoryId) · GET(활성 목록) · PATCH `/{categoryId}`(이름, 204) · DELETE `/{categoryId}`(204). 목록은 비페이징(MemberAddressListResponse 선례 — 소수 고정 집합) | CRUD 범위 그대로. 공개 카테고리 목록 API는 범위에 명시가 없어 제외(관리자 R로 충족) |
| 상품 등록·수정 | `ProductRegistrationRequest`·`ProductEditRequest`에 `categoryId`(@Nullable) 추가. edit 핸들러가 assignCategory 호출 | WORKPLAN "등록·수정 시 지정" 그대로 |
| 공개 필터 | `GET /api/v1/products?categoryId=`(선택) — 컨트롤러 → `ProductCatalogFacade.getCatalogPage(categoryId, keyword, sort, pageable)` → Reader 3종 → 기존 JPQL 3종+count에 `(:categoryId is null or p.categoryId = :categoryId)` 술어 추가 | 키워드 null 처리 선례와 동일 형태라 검색·정렬과 자연 조합. QueryDSL 전환은 기존 @Query 스타일을 갈아엎는 무관 리팩터라 기각 |
| 조회 노출 | `ProductInfo`·`ProductAdminResponse`에 categoryId 추가 | 지정의 관측 표면(관리자가 현재 분류를 봐야 재지정 가능). 공개 응답 노출은 범위 밖 |
| 에러코드 | `ProductErrorCode.CATEGORY_NOT_FOUND`(404) + `CategoryNotFoundException` | 존재하지 않는 카테고리 지정·수정·삭제 거부 |
| 마이그레이션 | `V3__create_category_and_product_category.sql`: category 테이블 + product.category_id 컬럼 + 인덱스, 물리 FK 없음 | 규칙 그대로 |

## 설계 리뷰

- 단순함: 계층·이름 유니크·공개 카테고리 목록 API·삭제 시 상품 일괄 해제·페이징 목록을 전부 기각했다. 카테고리는 엔티티 1·서비스 4·컨트롤러 1·DTO 4로 끝난다. "시니어가 과설계라 할까?" — 필터 술어 한 줄 추가가 조회 확장의 전부다.
- 외과적 수정: 기존 파일 변경은 Product(필드·메서드 1)·ProductAppender(시그니처)·ProductModifier(+메서드)·ProductInfo·ProductRepository(술어 4곳)·ProductReader(파라미터)·ProductCatalogFacade(파라미터)·ProductController(파라미터)·ProductAdminController(edit 1줄)·요청 DTO 2·ProductAdminResponse·ProductErrorCode·테스트 3파일 + ProductCatalogFacadeTest 호출처 5곳(null 추가)로 한정. 34곳 registerProduct 호출처는 위임 오버로드로 보존.
- 규칙 대조:
  - 소프트삭제 기본·Remover가 delete() 호출·활성 조회 DeletedAtIsNull 네이밍 — entity-persistence.md.
  - Category는 base finder 규제 대상(소프트삭제 엔티티) — 커스텀 finder만 노출.
  - 애그리거트 간 ID 참조·한 트랜잭션 한 애그리거트(지정 시 카테고리는 읽기만) — 준수.
  - 파생 쿼리로 못 담는 null 가드 조건은 @Query(JPQL) — 기존 키워드 선례.
  - admin 4중 요건·@Operation·@ApiResponse·@Schema — 게이트 대상.
- 지적·반영 1: 초안은 등록 파사드 시그니처를 직접 바꿔 34개 호출처를 수정하려 했다 — 무관 테스트 대량 수정이라 위임 오버로드로 교체.
- 지적·반영 2: 초안의 카테고리 목록 페이징은 소수 고정 집합에 과설계라 비페이징 목록(MemberAddressListResponse 선례)으로 축소.
- 트레이드오프: 삭제된 카테고리를 참조하는 상품의 dangling ID 수용(위 가정 참조 — 목록 비노출로 탐색 표면에서 자연 소멸). 5-인자 오버로드는 main에 남는 테스트 편의 표면이지만, 34곳 수정 대비 1메서드 유지가 싸다.
