# WORKPLAN — 커머스 API 기능 확장 작업 목록

랄프 루프가 이 파일을 진행 상태 저장소로 사용한다. 각 단계를 완료할 때마다 해당 체크박스를 `[x]`로 갱신하고 파일을 저장한다.

## 진행 방법

- 작업은 위에서부터 순서대로, 한 번에 하나만 진행한다.
- 한 이터레이션은 한 작업 안에서만 진행한다. 한 작업의 연속된 단계 여러 개를 끝내도 되지만, 작업 경계를 넘어 다음 작업을 시작하지 않는다.
- 각 작업은 아래 5단계를 순서대로 거친다.

| 단계 | 내용 | 산출물·검증 |
|---|---|---|
| 1. 설계 | `docs/architecture.md`, `docs/coding-conventions.md`, `docs/entity-persistence.md` 중 해당 문서를 로딩한다. 가정·해석 후보·트레이드오프를 명시하고, 완료 기준을 검증 가능한 테스트 목록으로 구체화한다. | `worklog/<번호>-<작업명>.md`에 설계 기록 |
| 2. 설계 리뷰 | 설계를 `AGENTS.md` 작업 원칙(단순함 우선, 외과적 수정, 목표 주도)과 `docs/` 규칙에 대조해 리뷰한다. "시니어 엔지니어가 이걸 과설계라 할까?"를 자문하고 지적 사항을 설계에 반영한다. | worklog 파일에 리뷰 결과·반영 내역 추가 |
| 3. 구현 | TDD로 진행한다. 완료 기준의 테스트를 먼저 작성해 실패를 확인하고, 통과하는 최소 코드를 구현한다. 기존 코드 스타일·패키지 구조(controller → facade → domain service)를 따른다. | 테스트 전부 통과 |
| 4. 코드 리뷰 | 변경 diff를 검토한다. 바뀐 모든 줄이 이 작업으로 추적되는지, `docs/` 규칙 위반이 없는지, 고아 import·데드 코드가 없는지 확인하고 수정한다. | worklog 파일에 리뷰 결과 추가 |
| 5. 게이트·커밋 | `./gradlew build`(Spotless·NullAway·Error Prone·전체 테스트) 통과를 확인하고 작업 단위로 커밋한다. 커밋 메시지는 기존 컨벤션(`feat: ...`)을 따른다. | build 성공, 커밋 완료 |

- 막히면 해당 작업의 worklog 파일에 시도한 것과 막힌 지점을 기록하고 이터레이션을 종료한다. 같은 단계에서 3회 연속 실패하면 작업 상태에 `중단`을 표기하고 사유를 기록한 뒤 다음 작업으로 넘어간다.
- 보류 섹션의 작업은 이 루프에서 구현하지 않는다.
- 설계 단계에서 결정이 필요한 항목은 worklog에 선택지와 결정·근거를 남긴다. 사용자에게 되묻지 않고 더 단순한 쪽을 택한다.

## 작업 목록

### 1. 사용자용 쿠폰 목록 조회

- 진행: [x] 설계 / [x] 설계 리뷰 / [x] 구현 / [x] 코드 리뷰 / [x] 게이트·커밋
- 목표: 사용자가 발급 가능한 쿠폰 목록을 조회하는 `GET /api/v1/coupons`를 제공한다.
- 배경: 발급 API(`POST /api/v1/coupons/{couponId}/issues`)는 있으나 사용자가 couponId를 알 방법이 없다. 관리자 목록과 응답 스키마를 분리한다.
- 범위: 발급 가능한(활성) 쿠폰만 페이징 조회. 개인화·추천·발급 여부 표시는 제외.
- 완료 기준:
  - 활성 쿠폰만 노출되고 비활성 쿠폰은 제외되는 테스트 통과.
  - 페이징 동작 테스트 통과.
  - 인증 정책(공개 여부)을 설계에서 결정하고 테스트로 고정.
- 참고: `web/v1/coupon/CouponController.java`, `web/v1/admin/coupon/CouponAdminController.java`, domain-coupon 서비스.

### 2. 비밀번호 변경

- 진행: [x] 설계 / [x] 설계 리뷰 / [x] 구현 / [x] 코드 리뷰 / [x] 게이트·커밋
- 목표: 본인 비밀번호를 변경하는 엔드포인트를 제공한다.
- 배경: `PATCH /api/v1/members/me`는 이름 변경만 지원한다.
- 범위: 현재 비밀번호 검증 후 새 비밀번호로 교체. 비밀번호 재설정(이메일)·로그인 시도 제한은 제외.
- 완료 기준:
  - 현재 비밀번호 불일치 시 실패하는 테스트 통과.
  - 변경 후 새 비밀번호로 로그인 성공, 기존 비밀번호로 로그인 실패 테스트 통과.
- 참고: `web/v1/member/MemberController.java`, domain-member `MemberCredentialValidator`, `MemberModifier`.

### 3. 체크아웃 미리보기

- 진행: [x] 설계 / [x] 설계 리뷰 / [x] 구현 / [x] 코드 리뷰 / [x] 게이트·커밋
- 목표: 주문서 화면용으로 상품 금액·쿠폰 할인·배송비·최종 결제 금액을 서버가 계산해 반환하는 미리보기 API를 제공한다.
- 배경: 쿠폰 단건 `discount-preview`만 있어 클라이언트가 최종 금액을 스스로 계산해야 한다.
- 범위: 장바구니 기준 금액 계산. 부작용 없음(재고 차감·쿠폰 사용·주문 생성 없음).
- 완료 기준:
  - 미리보기 금액과 동일 조건 실제 체크아웃의 결제 금액이 일치하는 테스트 통과.
  - 빈 장바구니·쿠폰 부적격 케이스의 에러 계약 테스트 통과.
- 참고: `facade/CheckoutFacade.java`, `web/v1/coupon/IssuedCouponController.java`의 discount-preview.

### 4. 바로구매

- 진행: [x] 설계 / [x] 설계 리뷰 / [x] 구현 / [x] 코드 리뷰 / [x] 게이트·커밋
- 목표: 장바구니를 거치지 않고 요청 바디의 `[{variantId, quantity}]`로 주문·결제를 생성하는 엔드포인트를 제공한다.
- 배경: 현행 체크아웃은 장바구니 전체 전환이라 상품 상세에서 바로 구매할 수 없고, 우회하면 기존 장바구니 내용이 오염된다.
- 범위: 재고 차감·쿠폰 적용·결제·보상(compensation) 의미는 기존 체크아웃과 동일하게 유지한다. 장바구니 상태는 변경하지 않는다. 엔드포인트는 기존 `POST /api/v1/orders`와 분리하며 URL은 설계에서 확정한다.
- 완료 기준:
  - 바로구매 후 장바구니가 변하지 않는 테스트 통과.
  - 재고 부족·결제 거절·존재하지 않는 variant 등 실패 계약 테스트 통과.
  - 성공 시 주문·결제 상태 검증 테스트 통과.
- 참고: `facade/CheckoutFacade.java`, `web/v1/order/request/CheckoutRequest.java`, `CheckoutConcurrencyTest`, `CheckoutFaultCompensationTest`.

### 5. 선택구매(장바구니 부분 체크아웃)

- 진행: [x] 설계 / [x] 설계 리뷰 / [x] 구현 / [x] 코드 리뷰 / [x] 게이트·커밋
- 목표: 장바구니 항목 일부만 선택해 체크아웃할 수 있게 한다.
- 배경: 현행은 장바구니 전체 전환만 가능하다.
- 범위: `CheckoutRequest`에 선택 항목(예: variantIds)을 추가하고 누락 시 기존 전체 체크아웃과 동일하게 동작(후방 호환). 주문된 항목만 장바구니에서 제거한다.
- 완료 기준:
  - 부분 체크아웃 후 미선택 항목이 장바구니에 남는 테스트 통과.
  - 장바구니에 없는 variantId 선택 시 에러 계약 테스트 통과.
  - 기존 전체 체크아웃 회귀 테스트 전부 통과.
- 참고: `facade/CheckoutFacade.java`, `facade/CartCommandFacade.java`.

### 6. 배송지 관리

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 회원 배송지 CRUD와 기본 배송지 지정(`/api/v1/members/me/addresses`)을 제공한다.
- 배경: 매 주문마다 주소 전체를 다시 입력해야 한다.
- 범위: 회원당 최대 개수 제한과 기본 배송지 단일 보장(구체 수치·방식은 설계에서 결정). 체크아웃 요청 형태는 변경하지 않는다 — 클라이언트가 저장된 배송지로 요청을 채워 보낸다.
- 완료 기준:
  - CRUD 및 기본 배송지 단일성 테스트 통과.
  - 타인 배송지 접근 시 404 테스트 통과.
  - 엔티티·연관관계가 `docs/entity-persistence.md` 규칙을 따르는지 아키텍처 테스트 통과.
- 참고: domain-member, `docs/entity-persistence.md`.
