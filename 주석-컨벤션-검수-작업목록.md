# 비-도메인 모듈 주석 컨벤션 검수 작업목록 — module-apps: app-api

판정 기준: `docs/coding-conventions.md`의 "코드 주석·Javadoc" 절. 검수 범위: app-api의 main + test 전체 141개 `.java`.

## 모듈별 위반 수 요약

| 영역 | 파일 수 | 확정 위반 |
|---|---|---|
| main — web(컨트롤러·DTO·auth) | 65 | 0 |
| main — facade·config·event·exception·view | 36 | 5 |
| test — facade·컨테이너 | 22 | 0 |
| test — web | 21 | 0 |
| 합계 | 144 | 5 |

확정 위반 5건은 전부 같은 클래스다: 파사드 공개 메서드에서 정상 호출자가 마주치는 전파 예외의 `@throws` 누락. 코드베이스 규범 — 협력자 예외를 게이트 없이 통과시키는 파사드 메서드는 그 예외를 `@throws`로 명시한다(`OrderPaymentFacade.getPayment`, `ProductDetailFacade.getProductDetail`, `PaymentConfirmationFacade.confirm` 등 전 파사드 일관) — 에서 아래 5곳만 이탈했다.

`@throws`의 예외 참조는 단순 클래스명 + import로 쓴다(패키지 경로 표기 금지 — 선례: `OrderPaymentFacade`의 Javadoc 전용 import).

## 작업 항목

### T1. CartCommandFacade.removeItem — 전파 예외 `@throws` 누락

- 파일:라인 — `module-apps/app-api/src/main/java/com/commerce/api/facade/CartCommandFacade.java:74`
- 대상 — 메서드 `removeItem(UUID, UUID)`
- 현재 — `/** 라인을 제거한다. 자격 게이트 없이 허용한다. */`
- 위반 규칙 — "정상 호출자가 마주칠 전파 예외는 `@throws`로 적는다." / "관측 가능한 계약(전파 예외·외부 효과)은 협력자를 거쳐 실현돼도 적는다."
  - `CartModifier.removeItem`은 `@throws CartItemNotFoundException 장바구니나 라인이 없으면`을 선언하고, 파사드는 게이트 없이 그대로 위임한다. 없는 라인 제거 시도는 정상 호출로 도달한다. 같은 파일의 `clear()`는 협력자가 미존재를 무시하므로 `@throws` 없음이 정확한 반면, 이 메서드만 누락이다.
- 제안 수정안 — Javadoc 블록으로 확장하고 `@throws CartItemNotFoundException 장바구니나 라인이 없으면` 추가(import 포함).
- 우선순위 — P1

### T2. CartCommandFacade.changeItemQuantity — 전파 예외 `@throws` 누락

- 파일:라인 — `module-apps/app-api/src/main/java/com/commerce/api/facade/CartCommandFacade.java:60-64`
- 대상 — 메서드 `changeItemQuantity(UUID, UUID, int)`
- 현재 — `@throws ApiException 증량 시 회원 자격 비활성; 주문 불가(변형 비활성·상품 HIDDEN·삭제)` 만 존재.
- 위반 규칙 — T1과 동일.
  - `CartReader.getCart`는 장바구니 미존재 시 빈 장바구니를 반환하므로 게이트를 통과하고, `CartModifier.changeItemQuantity`가 `CartItemNotFoundException`(장바구니나 라인이 없으면)을 던져 그대로 전파된다. 파사드 자신의 라인 주석(96행 "도메인의 changeItemQuantity가 라인 미존재를 보고한다")이 이 전파를 설계된 동작으로 명시하면서 태그만 비어 있다. `InvalidCartItemException`(수량 1 미만)은 요청 DTO `@Positive`가 게이트하는 버그 백스톱이라 계약이 아니다.
- 제안 수정안 — 기존 `@throws ApiException` 아래에 `@throws CartItemNotFoundException 장바구니나 라인이 없으면` 추가(import 포함).
- 우선순위 — P1

### T3. CartCommandFacade.addItem — 전파 예외 `@throws` 누락

- 파일:라인 — `module-apps/app-api/src/main/java/com/commerce/api/facade/CartCommandFacade.java:48-52`
- 대상 — 메서드 `addItem(UUID, UUID, int)`
- 현재 — `@throws ApiException 회원 자격 비활성; 주문 불가(변형 비활성·상품 HIDDEN·삭제)` 만 존재.
- 위반 규칙 — T1과 동일.
  - `CartAppender.addItem`은 `@throws InvalidCartItemException 수량이 1 미만이거나 합산 수량이 한도를 넘으면`을 선언한다. 합산 한도 초과는 장바구니 현재 상태에 의존해 요청 DTO 검증이 막을 수 없는 정상 입력 경로다(수량 1 미만만 `@Positive` 백스톱).
- 제안 수정안 — 기존 `@throws ApiException` 아래에 `@throws InvalidCartItemException 합산 수량이 한도를 넘으면` 추가(import 포함).
- 우선순위 — P1

### T4. ProductRegistrationFacade.addVariant — 전파 예외 `@throws` 3건 누락

- 파일:라인 — `module-apps/app-api/src/main/java/com/commerce/api/facade/ProductRegistrationFacade.java:59-65`
- 대상 — 메서드 `addVariant(UUID, Money, List<ProductOption>, int)`
- 현재 — 본문 프로즈가 "완결(ACTIVE) 변형은 중복으로 거부된다"까지 서술하면서 `@throws` 태그가 없다.
- 위반 규칙 — T1과 동일.
  - 비-재개 경로가 `ProductVariantAppender.create`로 위임되고, 그 계약은 `ProductNotFoundException`(활성 상품 없음)·`DuplicateVariantOptionException`(옵션 조합 겹침)·`InvalidVariantException`(옵션 부정·최소가 미만)을 선언한다. 셋 다 관리자 API 정상 입력으로 도달한다 — 옵션명 중복·구분자 포함은 DTO `@NotBlank`/`@Size`가 못 막고, DISABLED 중복은 재개로 흡수되므로 파사드 수준 거부 표면은 완결(ACTIVE) 중복이다. 재개 경로의 `enable`·재고 생성 예외는 직전 상태 확인이 보장하는 버그 백스톱이라 계약이 아니다.
- 제안 수정안 — 본문 뒤에 추가(import 포함):
  - `@throws ProductNotFoundException 활성 상품이 없으면`
  - `@throws DuplicateVariantOptionException 완결(ACTIVE) 변형과 옵션 조합이 겹치면`
  - `@throws InvalidVariantException 옵션이 올바르지 않거나 판매가가 최소가 미만이면`
- 우선순위 — P1

### T5. ProductRegistrationFacade.registerProduct — 전파 예외 `@throws` 누락

- 파일:라인 — `module-apps/app-api/src/main/java/com/commerce/api/facade/ProductRegistrationFacade.java:47`
- 대상 — 메서드 `registerProduct(String, String, Money, List<ProductOption>, int)`
- 현재 — `/** 상품·첫 변형·재고를 시딩하고 판매를 시작한다. */`
- 위반 규칙 — T1과 동일.
  - 내부 `addVariant` 경유로 `InvalidVariantException`에 정상 입력으로 도달한다(옵션명 중복·구분자 포함은 DTO가 못 막고, `Money.of(0)`이 허용되어 최소가 미만도 파사드 호출자 수준에서 도달 가능). 반면 `ProductNotFoundException`·`DuplicateVariantOptionException`·`ProductModifier.show`의 상태 예외는 방금 만든 HIDDEN 상품·빈 변형 목록이라는 자기 불변식상 도달 불가한 버그 백스톱이라 태깅하지 않는다.
- 제안 수정안 — Javadoc 블록으로 확장하고 `@throws InvalidVariantException 옵션이 올바르지 않거나 판매가가 최소가 미만이면` 추가(import 포함).
- 우선순위 — P1

## 위반으로 보지 않은 경계 항목(수정 안 함)

- 타입 Javadoc이 `@Schema` 설명을 한 문장 복창한 뒤 추가 계약을 덧붙이는 패턴(`CouponCreationRequest`·`LoginRequest`·`AddCartItemRequest` 등 7곳) — 동일 구조가 7곳에서 반복되는 사실상의 하우스 스타일(추가 계약이 있을 때 여는 방식). 복창 없이 순수 추가만 하는 2곳(`DiscountRequest`·`ProductAdminResponse`)과 갈리나 다수가 전자라 이탈로 볼 수 없다.
- 파사드 `@throws ApiException`의 명사구 나열 문체(`CartCommandFacade.addItem`·`changeItemQuantity`, `CheckoutFacade.checkout`) — 나머지 `@throws`는 전부 "~면" 조건절로 끝나 문체가 갈리지만, 컨벤션이 `@throws` 문장 어미를 규정하지 않아 규칙 위반이 아니다.
- `PaymentConfirmationFacade`·`PendingOrderSweepFacade`의 `restoreLineWithRetry` Javadoc이 협력자 `restore`의 성질(가산성)로 재시도 안전성을 설명 — "협력자 내부 동작 금지"에 닿아 있으나 두 파사드에 동일 문구로 일관된 의도된 스타일. 재시도 안전성 자체는 이 메서드의 계약이기도 하다.
- 상수 필드에 Javadoc 블록 대신 라인 주석으로 "무엇+왜"를 함께 적는 패턴, `// N. 명사구 — 부연` 형태의 단계 주석 — 코드베이스 전반(`JwtTokenCodec`, `MemberAppender` 등)과 파사드 6곳에서 동일하게 반복되는 확립된 관행.
- `ProductAdminController.edit()`·`PaymentWebhookController.receive()`의 2문장 본문에 단계 번호 없음 — 컨벤션은 "단계가 둘 미만이면 달지 않는다"만 규정. 매우 짧고 자명한 본문이라 판단 여지 안.
- `IssuedCouponController.getDiscountPreview()`의 메서드 Javadoc(미리보기 비보증 계약) — 이 패키지 유일의 엔드포인트 메서드 Javadoc이지만 "애노테이션에 없는 계약만 Javadoc에 남긴다" 규칙에 정확히 부합하는 정당한 사용.
- 테스트 타입 Javadoc 2문단의 기법 설명(`CheckoutFaultCompensationTest` 등 11곳, "왜 spy로 던지게 했는지" 류) — 대상 파일 20개 중 11곳에서 동일 형태로 반복되는 확립된 하우스 스타일. module-domains의 `*PersistenceTest`도 같은 구조.
- `AuthControllerTest:35`·`MemberControllerTest:50` 상수 주석의 뒷문장("초과 시도가 429로 거부되는지 검증한다") — `@DisplayName` 소유 사실의 되풀이로 읽힐 수 있으나 두 파일에서 동일 문구로 일관된 패턴이고, 테스트 상수는 영속 필드 요약 규칙의 대상이 아니다.
- `WebIntegrationTest:31` `ADMIN_EMAIL`의 doc("기동 시딩되는 관리자 계정 자격증명")이 아래 `ADMIN_PASSWORD`까지 가리키는지 모호 — 관련 상수 쌍 중 하나에만 doc을 두는 사례가 코드베이스에 이미 있어 이례적 패턴이 아니다.
- `ProductRegistrationFacade.registerProduct`의 `ProductNotFoundException`·`DuplicateVariantOptionException` 미태깅 — 자기 불변식상 도달 불가한 버그 백스톱이므로 계약이 아닌 게 맞다(T5는 `InvalidVariantException`만 태깅).
