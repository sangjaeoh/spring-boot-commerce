# WORKPLAN 3차 — 도메인 심화

랄프 루프가 이 파일을 진행 상태 저장소로 사용한다. 각 단계를 완료할 때마다 해당 체크박스를 `[x]`로 갱신하고 파일을 저장한다.

## 진행 방법

- 작업은 위에서부터 순서대로, 한 번에 하나만 진행한다.
- 한 이터레이션은 한 작업 안에서만 진행한다. 한 작업의 연속된 단계 여러 개를 끝내도 되지만, 작업 경계를 넘어 다음 작업을 시작하지 않는다.
- 각 작업은 아래 5단계를 순서대로 거친다.

| 단계 | 내용 | 산출물·검증 |
|---|---|---|
| 1. 설계 | `docs/architecture.md`, `docs/coding-conventions.md`, `docs/entity-persistence.md` 중 해당 문서를 로딩한다. 가정·해석 후보·트레이드오프를 명시하고, 완료 기준을 검증 가능한 테스트 목록으로 구체화한다. | `worklog/3-<번호>-<작업명>.md`에 설계 기록 |
| 2. 설계 리뷰 | 설계를 `AGENTS.md` 작업 원칙(단순함 우선, 외과적 수정, 목표 주도)과 `docs/` 규칙에 대조해 리뷰하고 지적 사항을 설계에 반영한다. "시니어 엔지니어가 이걸 과설계라 할까?"를 자문한다. | worklog 파일에 리뷰 결과·반영 내역 추가 |
| 3. 구현 | TDD로 진행한다. 완료 기준의 테스트를 먼저 작성해 실패를 확인하고, 통과하는 최소 코드를 구현한다. 기존 코드 스타일·패키지 구조를 따른다. | 테스트 전부 통과 |
| 4. 코드 리뷰 | 변경 diff를 검토한다. 바뀐 모든 줄이 이 작업으로 추적되는지, `docs/` 규칙 위반이 없는지, 고아 import·데드 코드가 없는지 확인하고 수정한다. | worklog 파일에 리뷰 결과 추가 |
| 5. 게이트·커밋 | `./gradlew build` 통과를 확인하고 작업 단위로 커밋한다(`feat: ...`). | build 성공, 커밋 완료 |

- 막히면 worklog 파일에 시도한 것과 막힌 지점을 기록하고 이터레이션을 종료한다. 같은 단계에서 3회 연속 실패하면 작업 상태에 `중단`을 표기하고 사유를 기록한 뒤 다음 작업으로 넘어간다.
- 설계 단계에서 결정이 필요한 항목은 worklog에 선택지와 결정·근거를 남긴다. 사용자에게 되묻지 않고 더 단순한 쪽을 택한다.

## 작업 목록

### 1. 교환·반품 요청

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 배송 완료 주문에 대한 반품 요청과 관리자 승인·거절 흐름을 제공한다.
- 배경: 현재 사용자는 결제 완료 상태의 전체 취소만 가능하고, 배송 후 반품 경로가 없다.
- 범위: 주문 상태 모델에 반품 흐름(요청됨·승인·거절·완료)을 확장한다. 사용자 요청 API(사유 포함)와 관리자 승인·거절 API를 제공한다. 승인 시 환불·재고 복원은 기존 환불 경로를 재사용한다. 부분 반품(라인 단위) 포함 여부는 설계에서 결정하고 근거를 남긴다. 교환은 반품 후 재주문 모델로 단순화할지 설계에서 결정한다.
- 완료 기준:
  - 배송 완료 상태가 아닌 주문의 반품 요청이 거부되는 테스트 통과.
  - 승인 시 환불·재고 복원이 검증되는 테스트 통과.
  - 거절 시 주문 상태가 원복되는 테스트 통과.
  - 허용되지 않은 상태 전이(중복 요청 등) 거부 테스트 통과.
- 참고: `OrderAdminController`(refund·ship·delivery-confirmation), `OrderRefundFacade`, domain-order `FulfillmentStatus`·`RefundReason`.

### 2. 비밀번호 재설정·이메일 인증

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 이메일 기반 비밀번호 재설정과 가입 이메일 인증을 제공한다.
- 배경: 비밀번호를 잊으면 복구 경로가 없고, 이메일 소유 검증이 없다.
- 범위: 메일 발송 포트는 domain-member `port/`가 소유하고(벤더 중립 명명) `external-mail` 모듈이 구현한다 — 실제 SMTP 연동은 범위 제외, 개발용 로그 발송 구현으로 대체한다. 재설정·인증 토큰은 Redis에 TTL 저장한다. 미인증 이메일의 로그인 차단 여부는 설계에서 결정한다(기존 가입 회원 소급 문제 포함).
- 완료 기준:
  - 재설정 요청 → 토큰 검증 → 새 비밀번호 설정 → 기존 비밀번호 로그인 실패 테스트 통과.
  - 만료·재사용 토큰 거부 테스트 통과.
  - 가입 시 인증 토큰 발급·검증 테스트 통과.
- 참고: `docs/architecture.md` external 모듈·포트 규칙(DIP), `MemberModifier`, 2차 작업 1의 Redis 토큰 패턴.

### 3. 상품 이미지

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 관리자 상품 이미지 업로드·삭제와 상품 조회 응답의 이미지 노출을 제공한다.
- 배경: 상품에 이미지 개념이 없다.
- 범위: 스토리지 포트는 domain-product `port/`가 소유하고 `external-storage` 모듈이 구현한다 — 실제 S3 연동은 범위 제외, 개발용 로컬 파일 구현으로 대체한다. 상품-이미지 연관 모델(정렬 순서·대표 이미지)과 관리자 업로드·삭제 API를 만들고, 상품 목록·상세 응답에 이미지 URL을 싣는다. 형식·크기 검증을 포함한다.
- 완료 기준:
  - 업로드 후 상품 상세·목록에 이미지가 노출되는 테스트 통과.
  - 허용되지 않은 형식·크기 거부 테스트 통과.
  - 삭제 시 노출 제거 테스트 통과.
- 참고: `docs/architecture.md` external 모듈 규칙, `ProductAdminController`, `ProductDetailFacade`.

### 4. 카테고리

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 카테고리 모델과 상품 분류, 카테고리별 목록 조회를 제공한다.
- 배경: 상품 탐색 축이 검색·정렬뿐이다.
- 범위: domain-product를 확장한다. 계층은 최대 2단계로 단순화할지 설계에서 결정한다. 관리자 카테고리 CRUD, 상품 등록·수정 시 카테고리 지정, 공개 목록 API에 카테고리 필터를 추가하고 기존 키워드 검색·정렬과 조합 가능하게 한다.
- 완료 기준:
  - 카테고리 필터 조회(검색·정렬 조합 포함) 테스트 통과.
  - 숨김 상품 제외가 유지되는 테스트 통과.
  - 관리자 CRUD·상품 분류 지정 테스트 통과.
  - 마이그레이션 적용 상태에서 전체 빌드 통과.
- 참고: `ProductRepository`(검색·정렬 프래그먼트), `ProductAdminController`, `docs/entity-persistence.md` 연관관계 규칙.
