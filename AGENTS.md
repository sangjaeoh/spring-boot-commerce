## 작업 원칙

LLM의 흔한 코딩 실수를 줄이는 핵심 행동 지침이다. 위 도메인 규칙과 함께 우선해서 적용한다.

### 1. 코딩 전에 생각한다

가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러낸다.

- 가정을 명시한다. 불확실하면 묻는다.
- 해석이 여럿이면 나열한다. 조용히 하나를 고르지 않는다.
- 더 단순한 방법이 있으면 말한다. 필요하면 반대한다.
- 트레이드오프를 드러낸다. 무엇을 선택했고 무엇을 포기하는지 설명한다.
- 불명확하면 멈춘다. 무엇이 혼란스러운지 짚고 묻는다.
- 완료 기준이 없는 작업 요청은 구현 전에 검증 기준부터 정의해 합의한다.

### 2. 단순함이 먼저다

문제를 푸는 최소한의 코드만 작성한다. 투기적인 설계는 넣지 않는다.

- 요청받지 않은 기능은 넣지 않는다.
- 일회성 코드에 추상화를 만들지 않는다.
- 요청하지 않은 "유연성"·"설정 가능성"을 넣지 않는다.
- 일어날 수 없는 상황에 예외 처리를 넣지 않는다.
- 200줄을 썼는데 50줄로 될 일이면 다시 쓴다.
- 자문한다. "시니어 엔지니어가 이걸 과설계라 할까?" 그렇다면 단순화한다.

### 3. 외과적으로 고친다

꼭 필요한 것만 건드리고, 내가 만든 것만 치운다.

기존 코드를 고칠 때:
- 주변 코드·주석·포맷을 "개선"하지 않는다.
- 고장나지 않은 것을 리팩터하지 않는다.
- 다르게 짜고 싶어도 기존 스타일을 따른다.
- 무관한 데드 코드를 발견하면 언급만 하고 지우지 않는다.

내 변경이 고아를 만들면:
- 내 변경이 쓰지 않게 만든 import·변수·함수를 제거한다.
- 기존 데드 코드는 요청 없이 지우지 않는다.

기준:
- 바뀐 모든 줄이 사용자의 요청으로 곧장 추적돼야 한다.

### 4. 목표 주도로 실행한다

성공 기준을 먼저 정의하고, 검증될 때까지 반복한다.

작업을 검증 가능한 목표로 바꾼다.
- "검증 추가" → "잘못된 입력에 대한 테스트를 쓰고 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고 통과시킨다"
- "X 리팩터" → "전후로 테스트가 통과하는지 보장한다"

다단계 작업은 간단한 계획을 밝힌다.

```text
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
```

- 강한 성공 기준은 에이전트가 독립적으로 반복 작업하게 만든다.
- 약한 기준("되게 해줘")은 계속 되묻게 만든다.

## 편집 규약

- 아래는 `docs/`의 규칙 문서에 적용한다.
- 규칙은 마크다운 불릿과 표로 쓴다. HTML·이모지·강조 기호·편집자적 라벨("중요"·"베스트 프랙티스")을 넣지 않는다.
- 문서 섹션은 제목·언제·규칙 셋으로 고정한다.
- 현재 상태로 쓴다. 편집 이력 서술을 넣지 않는다.
- 한 규칙은 한 문서가 소유하고 나머지는 참조만 한다. 검사가 강제하는 규칙 본문은 프로즈로 재서술하지 않는다.
