# todo 자동 진행 드라이버 프롬프트

한 이터레이션에 `todo.md`의 항목 **하나만** 수행한다. 슬라이스는 순서 의존(1 → 2 → 3)이므로 반드시 위에서부터 진행한다. 아래 규칙을 순서대로 따른다.

1. **항목 선정**: `todo.md`에서 위에서부터 `상태: 대기`인 첫 항목 N을 찾는다.
   - 항목 제목에 `[결정]`이 있으면 "선행 결정" 섹션에 해당 결정이 확정 기록돼 있는지 확인한다.
     기록이 없으면 그 항목을 `상태: 보류(결정 대기)`로 바꿔 커밋하고, 그다음 `대기` 항목으로 넘어간다.
   - `대기` 항목이 하나도 남지 않았으면(완료·보류만 남음) 정확히 `<promise>RALPH_ALL_DONE</promise>`을 출력하고 종료한다. 랄프 Stop 훅은 `<promise>` 태그 안의 문자열만 완료 신호로 인식하므로 태그 없는 맨 문자열은 종료로 잡히지 않는다(설명 중 프로즈에 `RALPH_ALL_DONE`을 적어도 태그가 없으면 오종료되지 않는다).
2. **수행**: `todo-prompt.md`에서 같은 번호 N의 코드블록 프롬프트를 읽고 그 지시를 그대로 수행한다.
   AGENTS.md·docs/ 규칙 로딩, 완료 기준 충족, 슬라이스 머지 전 독립 리뷰(구현 컨텍스트와 분리된
   서브에이전트) 모두 포함한다.
3. **검증**: 해당 항목의 완료 기준 충족 + `./gradlew build` 통과 + `docs/architecture.md`의 "빌드가
   강제하는 불변식" 자기검증 + 독립 리뷰 지적 해소.
4. **마무리**: 작업 브랜치를 main에 **로컬 머지**하고, `todo.md`의 항목 N을 `상태: 완료`로 바꿔
   커밋에 포함한다. **절대 push 하지 않는다** — 원격 반영 금지.
5. 항목 하나를 마치면 이터레이션을 종료한다. 다음 항목은 다음 이터레이션이 잡는다.
6. **미완 시 안전 규약**: 이번 이터레이션에서 슬라이스를 끝내지 못하면, 작업 트리를 깨끗이
   정리(의미 있는 중간 상태는 작업 브랜치에 커밋, 나머지는 되돌리기)하고 `todo.md` 해당 항목에
   `- 진행 메모: <현재 상태와 다음 단계>`를 남겨 커밋한 뒤 종료한다. 다음 이터레이션이 메모를
   보고 같은 항목을 이어서 수행한다.

## 슬라이스 경계 주의

- 슬라이스는 순서 의존이다. #1(빌드 배선·오토컨피그 배제 게이트)은 #2(코어 전환)의 선결 리스크를 격리
  검증하므로 반드시 먼저 완료한다. #2는 인증 강제를 실제로 교체하고, #3(문서 개정)은 #2 완료 후 코드에
  맞춰 문서를 정합한다. 위에서부터 진행하면 자연히 충족된다.
- 각 슬라이스는 머지 시점에 빌드 green을 유지한다 — 절반만 전환된 인증 상태(예: common-web 컴포넌트만
  교체하고 app-api SecurityConfig 미배선)로 머지하지 않는다. #2는 common-web 컴포넌트와 app-api
  SecurityConfig가 상호 의존이라 한 슬라이스 안에서 원자적으로 끝낸다.
- 전환은 컨트롤러·토큰 포맷(common-auth)·ArchUnit(test-architecture)을 무변경으로 유지한다. 독립
  리뷰에서 이 셋이 실제로 손대지 않았는지, @AdminOnly가 마커로 살아 있는지(삭제 시 어드민 4마커 규칙
  파손), 로그인 실패가 도메인 401($.code=MEMBER_INVALID_CREDENTIALS)로 남는지, 공개 경로(/error·webhook·
  actuator·swagger 포함)가 완전 열거됐는지, csrf.disable()·진입점/핸들러 problem+json 파리티가 지켜지는지
  반드시 확인한다.

## 모델 라우팅

항목의 `범위` 필드를 기준으로 구현·리뷰를 위임한다. 세션 모델은 fable을 전제하고, 위임 모델 하한은
**opus**다 — 어떤 작업도 opus 미만 모델에 위임하지 않는다.

- `범위: 소`(#1 빌드 배선, #3 문서 개정) — 구현을 **opus** 서브에이전트에 위임하고, 독립 리뷰는 세션
  모델이 직접 수행한다.
- `범위: 중~대` 또는 보안 정합·설계 판단이 핵심인 항목(#2 인증 코어 전환) — 구현과 독립 리뷰를 각각 별도
  **fable** 서브에이전트에 위임한다. #2는 필터 순서·무상태 설정·진입점 파리티·공개 경로 완전성 같은
  Spring Security 정합이 실패 지점이라, 리뷰를 구현과 분리해 이 축을 전수 검증한다.

위임 규칙:
- 서브에이전트는 이 세션의 대화를 보지 못한다. 위임 프롬프트에 `todo-prompt.md` 해당 번호의
  코드블록 전문과 진행 메모(있으면)를 그대로 포함해 전달한다.
- 리뷰 서브에이전트는 구현한 에이전트(또는 세션)와 반드시 분리한다 — 구현 diff와 완료 기준만
  전달하고, 지적을 해소한 뒤 머지한다.
- 브랜치 머지·`todo.md` 상태 갱신·커밋은 항상 메인 세션이 수행한다.
