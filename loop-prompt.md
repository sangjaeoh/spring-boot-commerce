# todo 자동 진행 드라이버 프롬프트

한 이터레이션에 `todo.md`의 항목 **하나만** 수행한다. 위에서부터 진행한다. 아래 규칙을 순서대로 따른다.

1. **항목 선정**: `todo.md`에서 위에서부터 `상태: 대기`인 첫 항목 N을 찾는다.
   - 항목 제목에 `[결정]`이 있으면 "선행 결정" 섹션에 해당 결정이 확정 기록돼 있는지 확인한다. 기록이 없으면 그 항목을 `상태: 보류(결정 대기)`로 바꿔 커밋하고, 그다음 `대기` 항목으로 넘어간다.
   - `대기` 항목이 하나도 남지 않았으면(완료·보류만 남음) 정확히 `<promise>RALPH_ALL_DONE</promise>`을 출력하고 종료한다. 랄프 Stop 훅은 `<promise>` 태그 안의 문자열만 완료 신호로 인식하므로, 태그 없이 프로즈에 `RALPH_ALL_DONE`을 적어도 오종료되지 않는다.
2. **수행**: `todo-prompt.md`에서 같은 번호 N의 코드블록 프롬프트를 읽고 그 지시를 그대로 수행한다.
   - **프롬프트 조립 주의**: 5~10번 블록은 `[4번 프롬프트와 동일하되 아래만 교체]`로 시작한다. 이 경우 4번 블록 전문을 가져와 "작업 대상"·"스캔 대상"·"추가 주의"만 그 블록의 내용으로 바꾼 **완성된 프롬프트**를 만들어 전달한다. 서브에이전트에 `[4번과 동일]`이라고만 넘기지 않는다.
   - AGENTS.md·docs/ 규칙 로딩, 완료 기준 충족, 슬라이스 머지 전 독립 리뷰(구현 컨텍스트와 분리된 서브에이전트)를 모두 포함한다. 주석 규칙 정본은 `docs/coding-conventions.md`의 "### 코드 주석·Javadoc"이다.
   - **이 마이그레이션은 1차와 방향이 반대인 항목이 있다.** 메서드 요약은 "시그니처가 말하면 생략"에서 "전부 의무(면제 7종 제외)"로, 영속 필드·enum 상수 요약은 신설, 타입 doc은 "조감도 소유"에서 "그 타입이 무엇인지만"으로 바뀌었다. 서브에이전트가 규칙을 기억으로 적용하지 않고 정본을 로딩했는지 확인한다.
3. **검증**: 이 마이그레이션은 Javadoc·주석만 바꾸고 동작을 바꾸지 않는다.
   - `./gradlew build` 통과 — 회귀·포맷(Spotless) 확인용이다. **주석 규칙 준수를 증명하지 않는다**(Error Prone Javadoc 검사는 경고일 뿐 게이트 아님 → code-quality.md).
   - 규칙 준수의 **주 게이트는 독립 리뷰**다 — 해당 항목의 스캔 대상 전반에서 "카테고리 발산 0"을 확인한다. 관측 예시에 그치지 않고 전수 스캔했는지 본다.
   - `docs/architecture.md`의 "빌드가 강제하는 불변식" 자기검증 — 주석 변경은 이 축에 영향이 없어야 한다(영향이 있으면 동작을 건드린 것이다).
4. **마무리**: 작업 브랜치를 main에 **로컬 머지**하고, `todo.md`의 항목 N을 `상태: 완료`로 바꿔 커밋에 포함한다. 완료 메모에 삭제한 설계 근거의 소유처 확인 결과를 요약한다. **절대 push 하지 않는다** — 원격 반영 금지.
5. 항목 하나를 마치면 이터레이션을 종료한다. 다음 항목은 다음 이터레이션이 잡는다.
6. **미완 시 안전 규약**: 이번 이터레이션에서 슬라이스를 끝내지 못하면, 작업 트리를 깨끗이 정리(의미 있는 중간 상태는 작업 브랜치에 커밋, 나머지는 되돌리기)하고 `todo.md` 해당 항목에 `- 진행 메모: <현재 상태와 다음 단계>`를 남겨 커밋한 뒤 종료한다. 다음 이터레이션이 메모를 보고 같은 항목을 이어서 수행한다.

## 슬라이스 경계 주의

- 이 마이그레이션은 **동작·시그니처 무변경**이다 — Javadoc·주석만 손댄다. 로직·제어흐름·공개 API 형태·테스트 기대값을 바꾸면 슬라이스를 벗어난 것이다. `./gradlew build` green이 무변경의 회귀 증거다.
- 슬라이스는 모듈 경계로 갈라 서로 **독립적**이다(각자 Javadoc-only·빌드 green이라 절반 머지 위험이 없다). 순서를 바꿔도 무방하나 위에서부터 진행한다.
- 규칙에 이미 맞는 주석은 "개선"하거나 재작성하지 않는다(AGENTS.md 외과적 편집). **발산한 지점만** 고친다.
- **코드 리팩터로 넘어가지 않는다.** 주석 중복이 코드 중복의 증상인 경우(예: 슬라이스 1의 `restoreLineWithRetry`)가 있다. 관측 사실만 보고하고 코드는 건드리지 않는다.

## 삭제 규약 (이번 마이그레이션의 핵심 실패 지점)

타입 doc 축약이 이 마이그레이션의 큰 몫이라, **설계 근거를 소유처 확인 없이 지우는 것**이 가장 큰 위험이다.

- 잘라내기 전에 그 내용을 `DOMAIN_MODEL.md`·`REQUIREMENTS.md`·해당 메서드 doc 중 어디가 소유하는지 grep으로 확인하고, 파일·행 근거를 보고에 적는다.
- 소유처가 없으면 지우지 말고 적절한 위치(메서드 doc·조율 단계 주석)로 **옮긴다**.
- 독립 리뷰는 삭제분마다 소유처 근거가 제시됐는지, 그 근거가 실제로 그 내용을 담는지 표본 검증한다.

## 독립 리뷰 체크리스트

리뷰 서브에이전트는 아래를 항목별로 확인한다.

- 요약 의무 4종(타입·메서드·영속 필드·enum 상수) 누락 0 — 관측 예시가 아니라 대상 전 파일 전수인지.
- 면제 7종(오버라이드·필드 반환 접근자·생성자·파생 쿼리 메서드·테스트 메서드·설명 문자열 보유 enum·애노테이션 소유 타입/멤버)에 요약을 단 곳 0.
- 타입 doc에 상태 전이 규칙·동시성 메커니즘·설계 근거 잔존 0.
- 필드 doc에 그 값이 게이트하는 동작·전이 효과 잔존 0.
- 형식: 타입·메서드 요약은 3인칭 서술형, 필드·enum 상수 요약은 명사구.
- 주석 용어가 `DOMAIN_MODEL.md` 도메인 용어집과 일치하는지. 미등재 용어를 새로 썼으면 용어집 등재가 커밋에 포함됐는지.
- 삭제분마다 소유처 근거가 있는지(위 삭제 규약).
- 단계 번호 주석이 규칙 범위 안인지 — 조율 메서드이고 단계가 셋 이상이며 순서 자체가 정보일 때만. 둘 이하 메서드나 순서가 자유로운 나열에 붙었으면 발산이다.
- 동작이 정말 안 바뀌었는지(diff에 Javadoc·주석 밖 변경이 없는지).

## 모델 라우팅

항목의 `범위` 필드를 기준으로 구현·리뷰를 위임한다. 세션 모델은 opus을 전제하고, 위임 모델 하한은 **opus**다 — 어떤 작업도 opus 미만 모델에 위임하지 않는다.

- `범위: 소`(#1, #3, #4, #5) — 구현을 **opus** 서브에이전트에 위임하고, 독립 리뷰는 세션 모델이 직접 수행한다.
- `범위: 중`(#2, #6, #7, #8, #9, #10) — 구현과 독립 리뷰를 각각 별도 **opus** 서브에이전트에 위임한다. 이 슬라이스들은 엔티티 필드·enum 상수가 많아 전수 스캔 완전성이 실패 지점이라, 리뷰를 구현과 분리해 이 축을 검증한다.

위임 규칙:
- 서브에이전트는 이 세션의 대화를 보지 못한다. 위임 프롬프트에 `todo-prompt.md` 해당 번호의 **조립된 코드블록 전문**과 진행 메모(있으면)를 그대로 포함해 전달한다.
- 리뷰 서브에이전트는 구현한 에이전트(또는 세션)와 반드시 분리한다 — 구현 diff와 완료 기준만 전달하고, 지적을 해소한 뒤 머지한다.
- 브랜치 머지·`todo.md` 상태 갱신·커밋은 항상 메인 세션이 수행한다.
