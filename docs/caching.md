# Caching

## 언제

- 조회·계산 결과의 캐싱을 도입·변경·제거할 때.
- 캐시 무효화·TTL·버전을 설계할 때.
- 캐시 저장소(분산·로컬)를 결정할 때.
- 캐시 구성(캐시 매니저·직렬화·오류 처리)을 배선할 때.
- 캐시 테스트 시나리오를 판정할 때.

## 규칙

### 캐시 판정

- 판별은 flush 테스트로 한다.
  - 캐시 전체가 삭제됐다고 가정하고 판정한다.
  - 지연만 늘면 파생이며 투명 캐싱 대상이다.
  - 데이터·행동이 사라지면 상태이며 영속 판정 대상이다 → [architecture](architecture.md)의 구현 배치 판정.
- 투명 캐싱은 Spring Cache 추상화(`@Cacheable`)로 한다.
- JPA 2차 캐시·쿼리 캐시는 사용하지 않는다.

### 도입 게이트

- 캐시는 선제 도입하지 않는다.
- 도입 대상 조회·캐시 이름·TTL·로컬 채택 예외는 사람 합의 사항이다.
- 에이전트는 성능 추정만으로 도입하지 않는다.
- 합의된 캐시 목록(이름·대상 조회·TTL·로컬 채택 예외·사유)은 채택 프로젝트의 도메인 모델 문서가 표로 소유한다.
- 도메인 모델 문서가 없으면 표의 위치부터 사람에게 합의로 요청한다.
- 표에 없는 캐시 이름은 선언하지 않는다.
- 표 값의 변경(TTL 포함)도 도입과 같은 합의 사항이다.
- 합의 유지·철회의 재평가는 사람이 캐시 통계 메트릭을 근거로 수행한다.

### 허용 계층

- `@Cacheable`은 두 위치의 public 진입 메서드에만 허용한다.
  - 도메인 `application`의 `Reader` 계약 provided 구현.
  - query 모듈의 provided 구현.
- 파사드·앱·어댑터·`domain` 구역·쓰기 서비스에는 `@Cacheable`을 두지 않는다.
- 자기 호출은 프록시를 경유하지 않으므로 내부 메서드 캐싱은 성립하지 않는다.
- 쓰기 서비스(`Appender`·`Modifier`·`Remover`·`Processor`)와 `Validator`는 같은 모듈의 `@Cacheable` 메서드를 직접 호출하지 않는다.
- 쓰기·검증 경로는 캐시를 경유하지 않은 현재 상태를 읽는다.
- 타 도메인 provided 경유 호출은 이 금지의 대상이 아니다.
- 쓰기 판정이 타 도메인의 최신 상태를 요구하면 캐시 없는 provided 메서드 사용을 캐시 도입 합의에서 확정한다.

### 트랜잭션 분리

- `@Cacheable` 메서드는 `@Transactional`을 선언하지 않는다.
- 캐시·트랜잭션 인터셉터의 기본 순서는 미정의라 적중에도 트랜잭션이 열릴 수 있다.
- 금지는 메서드 단위로 판정한다.
- 클래스 레벨 `@Transactional`이 있는 클래스의 `@Cacheable` 메서드도 금지 대상이다.
- 미스 경로는 package-private 위임 서비스 `Transactional{계약명}`에 위임한다.
- 위임 서비스는 `@Cacheable` 구현과 같은 패키지에 둔다.
  - 도메인 모듈은 `application` 평탄 배치다.
  - query 모듈은 구현 패키지다.
- 위임 서비스는 provided 계약을 만들지 않는다.
- 위임 서비스는 트랜잭션 경계 규칙을 그대로 적용한다 → [architecture](architecture.md)의 트랜잭션 경계.

### 캐시 값

- 캐시 값은 경계 모델(`Info` record)과 그 컬렉션, 표준 스칼라(문자열·수치·불리언·UUID·시각)만 허용한다.
- JPA 매핑 타입의 비노출은 → [architecture](architecture.md)의 노출 규칙.
- `Page`·`Slice`는 캐시하지 않는다.
- 페이지 구현 타입은 역직렬화가 불안정하다.
- 페이지 조회를 캐시하는 provided 계약은 `Page` 대신 모듈 소유 `{대상}PageInfo(List<{대상}Info> content, int page, int size, long totalElements)` record를 반환한다.
- 앱 파사드는 `{대상}PageInfo`를 `Page`로 재구성해 기존 페이지 응답 변환에 연결한다.
- `@Cacheable` 메서드는 `@Nullable` 반환을 선언하지 않는다.
- null 반환의 소거는 → [code-quality](code-quality.md)의 NullAway + JSpecify가 소유한다.

### 이름·키·버전

- 캐시 이름은 `{도메인}:{대상}:v{n}` 형식으로 짓는다.
- query 모듈 캐시 이름은 `query:{모듈명}:{대상}:v{n}` 형식으로 짓는다.
- 캐시 이름은 소유 모듈의 이름 상수 클래스 `public static final String` 상수가 소유한다.
  - 도메인 모듈은 `{도메인}CacheNames`를 `application/provided`에 둔다.
  - query 모듈은 `{모듈명}CacheNames`를 provided 인터페이스와 같은 패키지에 둔다.
- 애노테이션과 앱 구성은 이 상수만 참조한다.
- 캐시 이름 상수 클래스는 모듈 밖 임포트 허용 목록에 든다 → [architecture](architecture.md)의 노출 규칙.
- 키는 SpEL로 명시한다.
- 키는 적중을 구분하는 스칼라 컴포넌트만 연결한다.
- 키 구분자는 `:`로 고정한다.
- `Pageable` 등 복합 객체를 키로 쓰지 않는다.
- 정렬이 결과에 영향을 주면 `속성-방향` 문자열로 평탄화해 키에 포함한다.
- 키에 담지 않는 파라미터는 결과에 영향을 주지 않는 것만 허용한다.
- 키 프리픽스는 기본값 `이름 + "::"`을 그대로 쓴다.
  - 최종 키는 `{도메인}:{대상}:v{n}::{키}` 형태다.
- 버전은 `v1`부터 시작한다.
- 값의 직렬화 형상이 달라지는 변경(`Info` 컴포넌트·중첩 컴포넌트·시리얼라이저)은 버전을 올린다.
- 키의 의미가 달라지는 변경(SpEL)은 버전을 올린다.
- TTL·evict 대상 변경은 버전 증가 대상이 아니다.
- 배포된 이름·버전 조합의 키 의미·값 형상은 바꾸지 않는다.
- 대상 조회 자체가 바뀌면 구 이름 제거와 신 이름 도입으로 처리한다.
- 버전 증가는 롤링 배포의 구·신 인스턴스를 네임스페이스로 분리한다.
- 구 버전 엔트리는 TTL로 소멸한다.
- 같은 모듈에 버전만 다른 같은 이름을 공존시키지 않는다.

### 무효화

- 무효화는 evict와 TTL 두 축으로 성립한다.
- 모든 캐시는 유한 TTL을 갖는다.
- TTL은 evict 누락·유실·잔여 stale 창의 최종 백스톱이다.
- TTL 값은 데이터의 허용 stale 창으로 판정하며 도입과 같은 사람 합의 사항이다.
- 같은 모듈 쓰기 서비스 4종(`Appender`·`Modifier`·`Remover`·`Processor`)이 선언적 `@CacheEvict`를 소유한다.
- `Processor`의 무트랜잭션 조율 메서드에는 `@CacheEvict`를 두지 않는다.
- 조율이 일으키는 변경의 evict는 분할된 애그리거트별 쓰기 서비스가 소유한다.
- 이벤트 소비자가 캐시 대상 상태를 변경할 때는 쓰기 서비스를 경유한다.
- 경유 없이 상태를 변경하면 evict가 무음 누락된다.
- 한 쓰기가 여러 엔트리에 영향을 주는 캐시는 `allEntries = true`로 evict한다.
- 키 지정 evict는 한 쓰기가 정확히 한 엔트리에 대응하고 쓰기 파라미터로 키를 재구성할 수 있는 캐시에만 허용한다.
- `beforeInvocation`을 쓰지 않는다.
- `@CachePut`·`condition`·`unless`를 쓰지 않는다.
- evict는 transaction-aware 구성으로 커밋 후 실행한다.
- 커밋 후 실행은 커밋 전 evict로 인한 구값 재적재 창을 제거한다.
- 동시 조회의 늦은 put이 남기는 잔여 stale 창은 evict 타이밍으로 막을 수 없다.
- query 모듈 캐시는 TTL-only 캐시다.
  - evict를 두지 않는다.
  - stale 허용 조회만 캐시한다.

### 캐시 저장소

- 기본 캐시 저장소는 분산(Redis)이다.
- 로컬(Caffeine)은 로컬 채택 예외로만 쓴다.
  - 예외 기준: 네트워크·직렬화 비용이 지배적이거나 저변경 데이터.
  - 예외 채택은 사람 합의 사항이다.
- 로컬 캐시의 evict는 자기 인스턴스에만 적용된다.
- 로컬 캐시의 인스턴스 간 stale 창은 TTL 전체다.
- 로컬 캐시는 stale 허용 데이터만 담는다.
- 로컬 캐시는 온힙 상한(`maximumSize`·`maximumWeight`)을 명시한다.
- 직렬화·command timeout 규칙은 분산 캐시에만 적용한다.

### 구성

- 캐시 매니저·직렬화·오류 처리·이름별 TTL 구성은 앱 구성 코드가 소유한다.
- 이름별 TTL은 프로퍼티가 아니라 캐시 매니저 구성 코드로만 성립한다.
- 공통 구성의 common 승격 시점은 → [architecture](architecture.md)의 common 모듈 구조를 따른다.
- 캐시 애노테이션이 있는 모듈을 컴포넌트 스캔 대상으로 임베드하는 실행 앱은 그 모듈의 캐시에 필요한 캐시 구성(`@EnableCaching` 포함)을 싣는다.
- 런타임 전용 의존으로만 싣는 앱(마이그레이션 실행기)은 대상이 아니다 → [architecture](architecture.md)의 앱 모듈 구조.
- 구성 없는 임베드는 evict를 무음 no-op으로 만든다.
- 값 직렬화는 캐시 이름별 타입 고정 JSON 시리얼라이저로 한다.
- 타입 정보 임베딩(default typing) 방식을 쓰지 않는다.
- record는 타입 정보 임베딩 역직렬화가 깨진다.
- 캐시 매니저는 이름 상수에 등록된 이름만 생성한다(미등록 이름의 기본 구성 즉석 생성 금지).
- 이름별 TTL 엔트리가 없는 등록을 금지한다.
- 캐시 통계를 활성한다.
  - 분산은 `enableStatistics()`로 활성한다.
  - 로컬은 `recordStats()`로 활성한다.
- 캐시 애노테이션 의존은 도메인·query 모듈에 허용한다.
- 캐시 저장소 스타터·클라이언트는 앱만 싣는다.
- 의존 허용 위치는 계층 컨벤션 플러그인이 강제한다 → [architecture](architecture.md)의 빌드가 강제하는 불변식.

### 장애

- `CacheErrorHandler`를 필수 구성한다.
- get 오류는 로그 후 미스로 강등한다.
- put 오류는 로그 후 무시한다.
- 캐시 장애를 조회 실패로 전파하지 않는다.
- transaction-aware 지연 evict는 에러 핸들러 범위 밖이다.
- 지연 evict의 오류 처리(로그·메트릭)는 transaction-aware 래핑 구성이 소유한다.
- evict 오류 카운터 이름은 `cache.evict.errors`로 고정한다.
- evict 오류 카운터 태그는 캐시 이름으로 고정한다.
- 캐시 저장소 클라이언트의 command timeout을 명시 구성한다.
- 타임아웃이 오류로 전환돼야 미스 강등이 동작한다.
- 저하된 캐시 저장소는 강등 전까지 조회마다 timeout만큼 지연을 더한다.
- timeout 값은 캐시 우회 시 감수할 추가 지연의 상한으로 판정한다.
  - 원본 조회(DB) 지연보다 짧게 잡는다.
- timeout 값 결정은 도입과 같은 사람 합의 사항이다.
- 캐시 전용 저장소의 헬스 인디케이터 readiness 제외는 → [observability](observability.md)의 actuator가 소유한다.
- 미스 강등 설계와 모순되는 DOWN 전파를 막는다.

### 제거

- 캐시 제거는 도입과 같은 사람 합의 사항이다.
- 같은 이름의 애노테이션 사용처·이름 상수·TTL 구성·시리얼라이저 등록·합의 표 행을 같은 변경에서 함께 제거한다.
- 위임 서비스는 인라인으로 복원한다.
- 트랜잭션 선언은 원래대로 되돌린다.
- 페이지 래핑 반환형의 원복은 계약 소비자 영향이 있으므로 사람 합의로 정한다.
- 앱이 임베드한 어느 모듈에도 캐시 애노테이션이 남지 않으면 그 앱의 캐시 구성·`@EnableCaching`·캐시 저장소 의존(테스트 포함)도 제거한다.
- common으로 승격된 캐시 구성의 강등·제거도 같은 합의 사항이다.
- 캐시 저장소의 구 엔트리는 TTL 소멸에 맡기고 수동 삭제 절차를 두지 않는다(운영 수동 무효화는 → 범위 밖).

### 테스트

- 무효화 경로는 통합 레벨 대표 시나리오로 검증한다.
- 슬라이스의 롤백 격리에서는 커밋 후 evict가 실행되지 않는다.
- evict 짝이 있는 합의된 캐시 이름마다 대표 쓰기 경로 하나로 무효화 시나리오를 파생한다.
- TTL-only 캐시는 무효화 시나리오를 파생하지 않는다.
- 시계 의존 TTL 만료 테스트를 두지 않는다.
- 테스트 캐시 저장소는 Testcontainers를 쓴다 → [testing](testing.md)의 도구.
- 빌드 게이트 강제 내용의 행동 테스트 반복 금지는 → [testing](testing.md)의 테스트하지 않는 것.

### 강제와 리뷰

- 아래는 아키텍처 테스트가 빌드에서 강제한다.
  - `@Cacheable` 위치 제한(허용 계층의 두 위치만).
  - `@CacheEvict` 위치 제한(같은 모듈 쓰기 서비스만).
  - 쓰기 서비스·`Validator`의 같은 모듈 `@Cacheable` 메서드 직접 호출 금지.
  - `@Cacheable`과 `@Transactional`의 메서드 단위 동시 선언 금지(클래스 레벨 포함).
  - `@Cacheable` 메서드의 `@Nullable` 반환 선언 금지.
  - `@CachePut`·`condition`·`unless`·`beforeInvocation` 사용 금지.
  - `@Cacheable` 캐시 이름마다 같은 모듈 동명 `@CacheEvict` 존재(TTL-only인 query 모듈 제외).
  - `@CacheEvict` 캐시 이름마다 같은 모듈 동명 `@Cacheable` 존재(고아 evict 금지).
  - 캐시 이름 형식(버전 세그먼트 포함).
  - 같은 모듈에서 버전만 다른 같은 이름의 공존 금지.
  - `@Cacheable`과 키 지정 `@CacheEvict`의 `key` 명시(`allEntries = true` evict는 제외).
  - 애노테이션 캐시 이름 값 전수와 이름 상수 집합의 일치.
  - `@Cacheable` 반환 타입이 캐시 값 허용 타입인지 검사.
- 캐시 애노테이션 모듈을 컴포넌트 스캔 대상으로 임베드하는 앱의 캐시 구성 동반(런타임 전용 의존 앱 제외)은 빌드 검증 테스트(의존 그래프 기반)가 강제한다.
- 강제 항목의 장치가 없으면 최초 캐시 도입과 같은 변경에서 추가한다 → [architecture](architecture.md)의 아키텍처 테스트 모듈.
- 아래는 리뷰로 검증한다.
  - TTL 값의 적정성.
  - evict 대상 선택의 의미 정확성.
  - 키에 담지 않은 파라미터의 무영향.
  - 값 형상·키 의미 변경 시 버전 증가.
  - TTL 구성·시리얼라이저의 고아 엔트리 부재.
  - 구성·장애 배선(에러 핸들러·timeout·통계·온힙 상한·헬스 그룹 제외)의 규칙 준수.

### 완료 판정

- 에이전트는 캐시 변경 후 이 목록으로 자기검증한다.
- 도입·변경·제거가 합의 표에 반영돼 있다.
- `@Cacheable` 선언·evict 짝·이름 상수·TTL 구성·시리얼라이저 등록·표 행이 같은 이름 집합으로 일치한다(TTL-only 캐시는 evict 짝 제외).
- 값 형상·키 의미가 바뀐 변경의 버전 증가가 반영돼 있다.
- evict 짝이 있는 캐시마다 무효화 시나리오가 합의된 시나리오 목록에 있다(→ [testing](testing.md)의 시나리오 소유).
- 구성·장애 배선이 구성·장애 규칙과 일치한다.
- 아키텍처 테스트를 포함한 빌드가 통과한다.

### 범위 밖

- 캐시 스탬피드·워밍·분산 락·`sync` 옵션: 실물 필요 시 추가한다.
- 운영 수동 무효화(키 패턴 삭제·관리 엔드포인트): 실물 필요 시 추가한다.
- 캐시 저장소 서버 구성(메모리 축출 정책·용량): 인프라 소유다.
  - 조기 축출은 미스로, put 실패는 오류 처리로 흡수되므로 캐시 규칙과 독립이다.
- 상태 저장소로서의 관리형 캐시: → [architecture](architecture.md)의 구현 배치 판정.
