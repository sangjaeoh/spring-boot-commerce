# 커머스 요구사항 (Requirements)

Spring Boot 기반 커머스 최소 구현. 회원이 상품을 담고 주문·결제하는 핵심 거래 흐름의 요구사항(*무엇을 만드는가*)을 정의한다. 실제 구현 모델(도메인·엔티티·필드·정책·불변식)은 [`domain_model.md`](./domain_model.md)가 소유한다.

- **대상 사용자**: 구매 회원 · 서비스 운영 관리자
- **기준 방향**: 모듈러 모놀리스 — 10개 도메인(회원·상품·재고·장바구니·쿠폰·주문·결제·위시리스트·리뷰·문의)을 도메인 모듈로 두되 도메인 간은 ID 값 참조 + 앱 계층(파사드) 조율로 잇는다(물리 FK 없음 — 모듈 구조·의존 규칙은 `docs/architecture.md` 소유). 배포 단위는 4개 앱(구매자 API·관리자·배치·마이그레이션)으로 나뉘되 도메인 모듈을 공유한다. 결제는 벤더 중립 포트로 격리한 **동기 승인 fake PG**(실패·응답 유실을 결정론적으로 시뮬레이션)이며, 응답 유실로 미확정에 남은 결제는 **주기 리컨실·웹훅 확정 경로**가 PG 거래 상태 조회로 확정한다. 도메인 이벤트는 **트랜잭션 아웃박스**를 경유한다(발행-커밋 원자, 배치 릴레이가 최소 1회 전달). 인증은 이메일+패스워드 로그인·JWT 액세스 토큰과 리프레시 토큰 발급, 구매자 셀프서비스 표면의 토큰 주체 강제·소유권 검사, 역할(BUYER/ADMIN) 기반 관리자 오퍼레이션 가드까지 포함한다.

## 범위

- **포함**: 회원(가입·정지·탈퇴·역할·주소록·본인 비밀번호 변경), 인증(가입 패스워드 자격증명·이메일+패스워드 로그인·JWT 액세스 토큰·리프레시 토큰·로그아웃·비밀번호 재설정·이메일 인증·구매자 셀프서비스의 토큰 주체 강제·소유권 검사·관리자 오퍼레이션 역할 가드), 상품 카탈로그·변형(SKU)·옵션·카테고리·이미지·검색·정렬, 재고(차감·복원·재입고·판매 재개 시 재입고 알림 이벤트), 장바구니, 쿠폰(발급·발급 한도·할인·사용·할인 미리보기·발급분 관리자 무효화), 위시리스트(찜)·재입고 알림, 리뷰, 상품 문의(비밀글·관리자 답변), 주문(체크아웃[전체·선택·바로구매]·미리보기·결제 상태·이행 상태·출고 시 택배사·운송장 번호 기록·취소·라인 부분 취소·반품 요청/승인/거절 워크플로·전체 반품 환불·관리자 주문 검색), 결제(승인·실패·환불·부분 환불·미확정 결제 리컨실·웹훅 확정), 체크아웃·취소·반품 환불 크로스 도메인 조율, 도메인 이벤트의 트랜잭션 아웃박스 전달.
- **제외**: 소셜 로그인, 세분화된 퍼미션·권한 매트릭스(역할은 BUYER/ADMIN 이분법), 관리자 계정 관리 API·역할 변경 오퍼레이션(관리자는 설정 기반 기동 시딩으로만 생성), 실 PG 연동(fake PG 교체), 실 메일 발송(메일 게이트웨이는 로그 스텁), 배송 추적·택배사 API 연동·운송장 수정(출고·배송완료 상태와 출고 시 택배사·운송장 번호 기록 자체는 포함), 분할 배송·부분 이행, 교환(exchange) 워크플로, 반품 회수 배송 추적, 사용된(USED) 쿠폰의 소급 회수·주문 금액 재계산(무효화는 미사용 발급분만 대상), 상품 변형 옵션의 구조 정규화, 상인용 SKU 코드, 사용자 개시 미결제 주문 취소 경로, 주소록의 체크아웃 연결(배송지 참조 방식 — 체크아웃은 매 요청 배송지 전체를 입력받는다).

## 용어

| 용어 | 정의 |
|---|---|
| 변형(SKU) | 상품의 판매·재고 단위. 판매가·옵션 조합을 소유 |
| 옵션 시그니처 / 라벨 | 변형 옵션 조합의 정규화 유니크 키 / 사람이 읽는 표시 |
| 자격 활성 회원 | 담기·주문·발급 자격이 있는 회원(활성 상태이고 미탈퇴) |
| 발급분 | 회원에게 발급된 쿠폰 1건 |
| 사용 기한 | 발급분의 사용 만료(발급시각 + 정책 사용창). 정책의 발급 가능 기간과 별개 |
| 이행(Fulfillment) | 주문의 물리 배송 진행 축(시작 전·준비·출고·배송완료·보류) |
| 스냅샷 | 주문 시점 값을 복사해 이후 원본 변경과 무관하게 보존 |
| 체크아웃 | 장바구니(전체 또는 선택 라인) 또는 직접 지정한 라인을 주문·결제로 전환하는 앱 계층 흐름 |
| 바로구매 | 장바구니를 거치지 않고 변형·수량을 직접 지정해 즉시 주문·결제하는 체크아웃 경로 |
| 주문 라인 | 주문 안의 변형 단위 항목. 부분 취소·반품의 처리 단위 |
| 찜(위시리스트) | 회원이 상품에 남기는 관심 표시. 찜 자체가 재입고 알림 수신 자격 |
| 재입고 알림 | 품절 재고가 판매 재개될 때 그 상품을 찜한 회원에게 발송되는 메일 알림(단순 수량 증량으로는 발송되지 않음) |
| 비밀글 | 본문·답변을 작성자 본인과 관리자만 열람할 수 있는 상품 문의 |
| 아웃박스 | 도메인 이벤트를 상태 변경과 같은 트랜잭션의 테이블 행으로 기록하고 별도 릴레이가 전달하는 메시징 방식 |
| 오버셀 | 재고보다 많이 판매되는 것 |
| 사가 앵커 | 체크아웃의 모든 부작용이 걸리는 미결제(생성 직후) 주문 |

## 기능 요구사항

### 회원

- 이메일·이름·패스워드로 가입하고, 가입 즉시 담기·주문할 수 있다(별도 활성화 절차 없음).
- 활성 회원 사이 이메일은 유일하되, 탈퇴한 이메일은 재가입할 수 있다.
- 표시 이름을 변경할 수 있다(이메일은 불변).
- 로그인한 회원은 현재 비밀번호를 대조해 새 비밀번호로 변경할 수 있다. 새 비밀번호는 가입과 같은 패스워드 정책을 따르고, 현재 비밀번호가 일치하지 않으면 변경을 거부한다.
- 배송지를 여러 건 등록해 주소록으로 관리할 수 있다. 배송지는 회원당 등록 한도까지 등록할 수 있다.
- 첫 배송지는 자동으로 기본 배송지가 되며, 이후 회원이 원하는 배송지를 기본으로 재지정할 수 있다. 기본 배송지는 항상 하나다.
- 배송지 정보(수령인·우편번호·도로명 주소·상세 주소·연락처)를 수정할 수 있다. 수정은 기본 지정 여부를 바꾸지 않는다.
- 배송지를 삭제할 수 있다. 기본 배송지를 삭제해도 다른 배송지가 자동으로 기본이 되지 않는다(회원이 다시 지정해야 한다).
- 타인의 배송지는 조회·수정·삭제·기본 지정 모두 미존재로 취급한다(존재 누출 방지).
- 주소록은 아직 체크아웃과 연결되지 않았다 — 체크아웃은 매 요청마다 배송지 전체를 입력받는다(배송지 참조 방식 미지원).
- 탈퇴(사유 기록)를 제공하고, 탈퇴 회원은 기본 조회에서 제외한다.
- 관리자가 회원을 정지·해제(사유 기록)할 수 있다. 정지는 탈퇴와 독립 축이라 정지된 채 탈퇴·탈퇴 없이 정지가 모두 가능하다.
- 담기·주문·쿠폰 발급은 자격 활성(활성 ∧ 미탈퇴) 회원만 가능하다.
- 정지·탈퇴는 회원의 장바구니·발급 쿠폰·미결제 주문을 연쇄 정리하지 않고, 다음 담기·체크아웃 자격 검증에서만 차단한다.
- 미배송 결제완료 주문이 있는 회원의 탈퇴는 거부한다.

### 인증

- 가입 시 패스워드를 설정한다. 패스워드는 정책이 정한 길이 범위(bcrypt 입력 한계 이내)를 따른다.
- 회원은 역할(`BUYER`·`ADMIN`) 하나를 갖는다. 가입은 항상 구매자를 만들고, 관리자는 설정(환경변수 `AUTH_ADMIN_EMAIL`·`AUTH_ADMIN_PASSWORD`·`AUTH_ADMIN_NAME`) 기반 기동 시딩으로만 생성한다(같은 이메일의 활성 회원이 있으면 건너뛰는 멱등 시딩, 관리자 계정 관리 API 없음).
- 이메일+패스워드 로그인이 성공하면 JWT 액세스 토큰과 불투명 리프레시 토큰을 함께 발급한다. 액세스 토큰은 주체(회원 ID)와 역할을 클레임으로 싣는다. 역할 변경은 새 토큰 발급부터 반영된다(가드는 토큰 클레임만 검사).
- 리프레시 토큰은 Redis에 주체(회원 ID)와 함께 유효기간 336시간(14일)으로 저장되고, 이 토큰으로 새 액세스 토큰을 재발급받을 수 있다. 재발급은 토큰이 가리키는 회원이 탈퇴하지 않았는지만 재확인하고 정지 여부는 확인하지 않는다(로그인과 같은 축).
- 리프레시 토큰은 회전하지 않는다 — 갱신 응답은 새 액세스 토큰만 담고, 로그아웃하거나 유효기간이 만료될 때까지 같은 리프레시 토큰을 계속 재사용한다.
- 로그아웃은 제출한 리프레시 토큰을 즉시 삭제해 무효화한다. 이미 무효한(미발급·만료·기로그아웃) 토큰의 로그아웃도 성공으로 처리한다(존재 누출 방지와 같은 결의 설계). 무효한 리프레시 토큰(위조·만료·로그아웃됨·가리키는 회원이 탈퇴함)으로 갱신을 시도하면 401로 거부한다.
- 로그인 실패(미존재 이메일·탈퇴 회원·패스워드 불일치)는 원인을 구분하지 않고 동일하게 거부한다(계정 존재 노출 방지).
- 정지 회원의 로그인은 허용한다 — 정지 차단은 담기·주문 자격 게이트가 담당한다(정지·탈퇴 비연쇄 원칙과 동일 축).
- 가입 이메일로 비밀번호 재설정을 요청할 수 있다. 이메일 존재 여부와 무관하게 항상 성공으로 응답한다(계정 존재 노출 방지 — 로그인 실패 균일 거부와 같은 원칙). 실제 존재하는 회원이면 1회용 재설정 토큰(유효기간 30분)을 발급해 메일 게이트웨이로 전송한다.
- 재설정 토큰과 새 비밀번호로 비밀번호를 교체할 수 있다. 토큰은 1회용이며(동시 소비 경합에서도 1회만 성공) 재사용·만료·위조 토큰은 401로 거부한다. 재설정은 현재 비밀번호 대조 없이 새 비밀번호(가입과 같은 정책)로 즉시 교체한다 — 본인 확인 수단이 이메일 소유(토큰 수신)이기 때문이며, 로그인 상태에서 현재 비밀번호를 대조하는 본인 비밀번호 변경과는 별개 흐름이다.
- 비밀번호 재설정 요청 표면에도 로그인·가입과 같은 IP당 레이트리밋이 걸려 있다(무인증 메일 발송 유발 표면 보호 — 키 설계·한도 근거는 decisions.md ADR-4).
- 가입이 성공하면 1회용 이메일 인증 토큰(유효기간 24시간)을 발급해 메일 게이트웨이로 전송한다. 인증 토큰을 제출하면 회원의 이메일 소유 인증 시각을 기록한다. 무효(위조·만료·기사용) 토큰은 401로 거부한다.
- 이메일 인증은 회원 자격에 어떤 영향도 주지 않는다 — 담기·주문·쿠폰 발급·로그인 등 모든 자격 검사가 인증 여부를 조건으로 쓰지 않는다(가입 즉시 사용 원칙 유지). 이메일 인증 시각은 본인 조회 응답에 참고 정보로만 노출된다.
- 엔드포인트 표면은 넷으로 분류한다 — 공개·본인(셀프서비스)·관리자·시스템(웹훅).

| 분류 | 판별 | 강제 | 대표 예시 |
|---|---|---|---|
| 공개 | 가입·로그인·카탈로그 조회 등 비로그인 쇼핑 표면 | 무인증 허용 | `POST /api/v1/auth/login`, `GET /api/v1/products` |
| 본인(셀프서비스) | 회원 자원(장바구니·주문·쿠폰·위시리스트·리뷰·문의·주소록 등)의 조회·변경 | 토큰 주체에서 회원 도출, 미인증 401, 미소유 리소스 404 | `POST /api/v1/orders`, `GET /api/v1/wishlists` |
| 관리자 | `/api/v1/admin/**` 전체(app-admin) | ADMIN 역할 토큰만, 미인증 401·비관리자 403 | `POST /api/v1/admin/orders/{id}/ship` |
| 시스템(웹훅) | PG 결제 확정 통지 수신(app-batch) | 토큰 대신 HMAC-SHA256 본문 서명 | `POST /api/v1/payments/webhook` |

엔드포인트 전수 목록은 각 앱의 swagger-ui(OpenAPI 표면)가 소유한다 — 이 문서는 분류 규칙만 소유한다.

- PG 결제 확정 통지 웹훅은 `X-Webhook-Signature` 헤더에 서명을 싣고 `PAYMENT_WEBHOOK_SECRET` 공유 시크릿으로 검증한다.
- 쿠폰 발급은 회원 셀프 클레임이다 — 대상 회원을 토큰 주체에서 도출하며, 관리자가 특정 회원에게 발급하는 형상은 두지 않았다.
- 타인 소유 리소스(주문·발급분) 접근은 존재 누출 방지로 미존재(404) 취급한다.

### 상품 (카탈로그)

- 상품을 등록하고(최초 숨김), 노출·숨김을 전환할 수 있다.
- 상품명·설명을 편집할 수 있으며, 이미 생성된 주문 내역에는 영향이 없다.
- 상품을 논리삭제할 수 있다.
- 카탈로그 노출은 판매중 ∧ 미삭제 ∧ 활성 변형 1개 이상일 때다.
- 활성 변형이 모두 소진돼도 상품은 계속 노출되며 품절로 표시된다.
- 상품 대표가는 활성 변형의 현재가를 반영한다.
- 관리자가 카테고리를 이름 하나로 생성할 수 있다. 계층(상위·하위 분류)은 두지 않는다 — 모든 카테고리는 단일 레벨이다. 카테고리 이름을 변경할 수 있고, 논리삭제할 수 있다. 삭제된 카테고리를 참조하던 상품은 관리자가 다시 지정하기 전까지 그 참조를 그대로 들고 있는다(연쇄 정리 없음).
- 상품 등록·편집 시 카테고리를 지정할 수 있다(선택, 상품당 최대 1개). 지정을 생략하거나 해제하면 미분류다.
- 관리자가 상품에 이미지를 업로드할 수 있다. jpeg·png·webp만 지원하고 장당 용량 상한을 넘을 수 없다.
- 업로드 순서가 곧 정렬 순서이며, 가장 먼저 업로드한 이미지가 대표 이미지다. 순서를 나중에 바꾸는 기능은 없다.
- 관리자가 이미지를 개별 삭제할 수 있다. 삭제하면 목록 표시와 실제 저장소에서 함께 제거된다.
- 상품 목록은 대표 이미지 1장을, 상품 상세는 전체 이미지를 정렬 순서대로 보여준다. 이미지가 없는 상품은 이미지 없이 조회된다.
- 카탈로그 목록을 상품명 키워드로 검색할 수 있다(부분 일치, 대소문자 무시). 생략하면 전체 노출 상품이 대상이다.
- 카탈로그 목록을 카테고리로 좁혀 조회할 수 있다. 생략하면 전체 카테고리가 대상이다.
- 카탈로그 목록을 최신 등록순(기본)·대표가 낮은순·대표가 높은순으로 정렬해 조회할 수 있다. 키워드·카테고리·정렬은 함께 조합할 수 있고, 정렬 기준이 페이지 크기·총 건수를 바꾸지 않는다(노출 집합은 동일).

### 상품 변형 (SKU)

- 변형을 판매가·옵션 조합으로 등록할 수 있다(최초 비활성). 판매가는 1원 이상이다(무료 변형 미지원).
- 변형 판매가를 변경할 수 있으며, 이미 생성된 주문 내역에는 영향이 없다.
- 변형을 활성·비활성·은퇴할 수 있고, 은퇴는 되돌릴 수 없는 종료다.
- 한 상품·한 옵션 조합에 비은퇴 변형은 최대 1개이며, 은퇴한 조합은 새 변형으로 재등록할 수 있다.
- 옵션 조합은 생성 시 확정·불변이며, 조합 변경은 새 변형이다.

### 재고

- 재고는 변형당 하나이며 초기 수량으로 시작한다.
- 재고를 차감(부족 시 거부)·복원(취소·결제 실패 보상)·재입고할 수 있다.
- 복원은 재고 상태(품절·단종)와 무관하게 항상 성공한다(취소·환불이 재고 상태에 막히지 않도록). 재입고는 단종 재고에는 거부한다.
- 운영자가 수동 품절·판매 재개, 단종(종료)할 수 있다.
- 운영자가 수동 품절 재고를 판매 재개하면 재입고 알림 발송을 위한 도메인 이벤트가 발행된다. 단순 수량 재입고(증량)만으로는 이 이벤트가 발행되지 않는다 — 알림 대상은 "품절이었다가 다시 살 수 있게 된" 변형이다.
- 동시 주문에도 재고보다 많이 팔리지 않는다(오버셀 없음).
- 소진(수량 0)·수동 품절·단종은 회복 경로가 달라(재입고·판매 재개·불가) 구분한다.

### 장바구니

- 회원당 장바구니 1개를 두고, 최초 담기 시 생성한다(회원 가입과 결합하지 않는다).
- 자격 활성 회원이 활성 변형(그 상품이 노출·미삭제)을 담을 수 있고, 동일 변형 재담기는 수량을 합산한다.
- 수량을 1 이상으로 변경할 수 있다 — 증량은 담기와 동일 자격 게이트, 감량·유지는 게이트 없이 허용한다(정지·탈퇴 회원, 비활성·삭제 라인도 정리 가능). 0으로 낮춰 제거하지는 않는다(제거는 별도 기능).
- 라인 개별 제거·전체 비우기를 제공한다.
- 재고는 담기 시점에 검증하지 않고 체크아웃 시점에 검증한다. 총액은 체크아웃 시 변형 현재가로 계산해 항상 현재가를 반영한다.
- 결제 완료 시 주문된 변형 라인을 장바구니에서 제거한다. 제거는 결제 완료 트랜잭션과 분리된 비동기 후처리다 — 체크아웃 응답 시점에는 아직 비워지지 않을 수 있고, 별도 배치 프로세스가 뒤이어 비운다. 유실 시 장바구니가 남을 뿐이라 재체크아웃으로 같은 상품을 다시 주문할 수 있다(사용자 개시·가시).

### 쿠폰

- 쿠폰 정책을 할인(정액/정률)·최소 주문 금액·발급 가능 기간·사용창·발급 한도(선택)로 생성한다. 한도가 없으면 무제한 발급이다.
- 정책의 발급 가능·중지를 전환할 수 있다.
- 발급 가능한(활성·발급 기간 내·한도 미소진) 쿠폰 목록을 최신 등록순 페이지로 조회할 수 있다.
- 본인에게 발급된 쿠폰 목록을 최신순으로 조회할 수 있다.
- 회원에게 발급한다 — 정책이 발급 가능·발급 기간 내·대상 회원이 자격 활성·회원당 1회·발급 한도 미소진일 때만. 발급 시 사용 기한을 확정(스냅샷)한다.
- 발급 한도가 소진되면 발급을 거부하고, 동시 발급 경합에서도 한도를 초과하지 않는다.
- 관리자가 미사용(ISSUED) 발급분을 사유와 함께 무효화(REVOKED)할 수 있다. 무효화된 발급분은 사용이 거부된다.
- 사용된(USED) 발급분은 무효화를 거부한다 — 사용된 쿠폰의 소급 회수·주문 금액 재계산은 범위 밖이다.
- 할인액을 계산한다 — 정액은 주문금액 상한, 정률은 퍼센트·상한(있으면)·주문금액으로 제한. 결과는 항상 주문금액 이하다.
- 적용은 (할인 전) 주문금액이 최소 주문 금액 이상·본인 소유·미사용·사용 기한 내·산출 할인 > 0일 때만 한다.
- 발급분이 주어진 주문금액에 적용 가능한지와 예상 할인액을 미리 조회할 수 있다(할인 미리보기).
- 주문 생성 후 사용을 확정하고, 주문 취소·결제 실패 시 복원한다.
- 발급 후 정책이 중지돼도 이미 발급된 쿠폰은 계속 사용할 수 있다(중지는 신규 발급만 막는다).
- 만료는 사용 시점 판정으로 거부한다(별도 만료 상태·배치 없음).
- 동시 두 주문이 같은 발급분을 사용하면 한쪽만 성공한다.

### 위시리스트·재입고 알림

- 회원은 상품을 찜하거나 찜을 해제할 수 있다. 이미 찜한 상품을 다시 찜해도, 찜하지 않은 상품을 다시 해제해도 그대로 성공한다(멱등).
- 찜 담기는 상품의 존재·노출·판매 상태를 검사하지 않는다 — 숨김·삭제 상품도 찜할 수 있다(장바구니 담기의 자격 게이트와 다르다).
- 회원은 본인의 찜 목록을 최신 찜 우선으로 조회할 수 있다. 각 항목은 상품 ID와 찜한 시각을 담는다.
- 찜은 회원별로 격리된다 — 다른 회원의 찜을 조회·해제할 수 없다.
- 같은 (회원, 상품) 조합은 유일하다. 동시에 같은 상품을 두 번 찜해도 한 건만 남는다.
- 회원이 별도로 알림을 신청하는 절차는 없다 — 상품을 찜하는 것 자체가 그 상품의 재입고 알림 자격이다.
- 관리자가 품절 재고를 판매 재개로 전환하면, 그 변형이 속한 상품을 찜한 회원 각각에게 재입고 알림 메일이 자동 발송된다. 재고 수량만 늘리고 품절 상태를 벗어나지 않으면 알림은 나가지 않는다.
- 발송 대상은 미탈퇴 회원이다(정지 회원 포함, 탈퇴 회원 제외).
- 같은 재입고 이벤트가 중복 전달되거나 일부 회원 발송 실패 후 재시도돼도, 이미 받은 회원에게는 다시 발송하지 않는다.
- 메일 발송은 벤더 중립 포트를 통하며, 현재 개발 구현은 실제 발송 없이 로그로 남기는 스텁이다(외부 연동 참고).

### 리뷰

- 배송 완료(구매확정)한 상품에 한해 별점·본문(정책 범위 내) 리뷰를 쓸 수 있다. 결제만 완료되고 미배송인 주문은 자격이 아니다.
- 상품당 한 건만 쓸 수 있다. 이미 쓴 상품에는 재작성이 거부된다.
- 본인이 쓴 리뷰의 별점·본문을 고쳐 쓸 수 있다. 타인 리뷰는 수정할 수 없다.
- 본인이 쓴 리뷰를 삭제할 수 있다. 타인 리뷰는 삭제할 수 없다. 본인 삭제 후에는 같은 상품에 다시 쓸 수 있다.
- 상품의 리뷰 목록을 최신 작성 순으로 페이지 조회할 수 있다. 로그인 없이도 열람할 수 있다.
- 관리자는 부적절한 리뷰를 사유와 함께 제거할 수 있다. 제거된 리뷰는 공개 목록에서 빠지고 사유는 보존되며, 회원은 해당 상품에 리뷰를 다시 쓸 수 없다.

### 문의

- 로그인한 회원 중 자격 활성(활성 ∧ 미탈퇴) 회원만 상품에 문의를 남길 수 있다(본문은 정책 범위 내, 공백 불가). 대상 상품은 실존(미삭제)해야 하며, 숨김 상품에도 문의할 수 있다(구매 후 숨김된 상품 대응). 구매 이력은 무관하다.
- 문의를 비밀글로 남길 수 있다 — 비밀글의 본문·답변은 작성자 본인과 관리자만 열람할 수 있다. 비밀글이 아니면 누구나 열람할 수 있다.
- 상품별 문의 목록을 인증 없이 최신 문의 우선 페이지로 조회할 수 있다. 비밀글은 목록에 존재(작성 시각·비밀글 여부)는 보이되, 열람 권한이 없는 조회자에게는 본문·답변이 가려진다.
- 관리자는 문의에 답변을 등록할 수 있다. 이미 답변이 있는 문의에 다시 등록하면 기존 답변을 덮어쓴다(이력은 남기지 않는다).
- 관리자는 답변 여부로 필터한 전체 문의 목록을 최신 문의 우선 페이지로 조회할 수 있다 — 비밀글도 항상 전문이 실린다.

### 주문

- 체크아웃으로 장바구니 전체 또는 선택한 라인(변형 지정)을 주문으로 전환한다. 선택한 변형 중 장바구니에 없는 것이 있으면 체크아웃 전체를 거부하고, 선택하지 않은 라인은 장바구니에 남는다.
- 장바구니를 거치지 않고 변형·수량을 직접 지정해 즉시 주문·결제(바로구매)할 수 있다. 바로구매는 장바구니 내용을 변경하지 않는다.
- 체크아웃을 실행하기 전에 장바구니 전체 또는 선택한 라인·배송비·쿠폰으로 결제 예정 금액(상품 합계·할인액·배송비·결제 예정액)을 미리 계산해 볼 수 있다. 미리보기는 체크아웃과 같은 주문 가능성 게이트(회원 자격·변형 활성·상품 판매중·재고 판매 가능·쿠폰 적용성)와 같은 금액 산식을 쓰되, 주문·재고 차감·쿠폰 사용·결제 요청 등 어떤 부작용도 남기지 않는다.
- 라인(변형·상품명·옵션·단가)과 배송지를 주문 시점 값으로 스냅샷 보관해, 이후 상품·변형·회원 변경이 내역을 바꾸지 않는다.
- 단가·쿠폰 할인·결제 대상 금액은 서버가 계산한다(클라이언트 전달값 불신). 결제 대상 금액 = 상품 라인 합계 − 쿠폰 할인 + 배송비이며 항상 0 이상이다.
- 배송비(0 가능)는 배송 정책이 범위 밖이라 체크아웃 요청이 제공하는 스냅샷을 사용한다.
- 결제 상태(미결제·완료·취소·환불)를 관리한다.
- 이행 상태(준비·출고·배송완료, 보류·해제)를 결제 상태와 독립 축으로 관리한다. 이행은 결제 전 '시작 전'이며, 결제 완료가 이행을 '준비'로 진입시킨다.
- 모든 이행 전진은 결제 완료 주문에서만 유효하다(취소·미결제 주문은 출고 불가). 취소 진행 중 — 전체 취소 개시가 기록됐거나 취소 진행 중인 라인이 있는 — 주문의 출고도 거부한다.
- 출고 시 택배사·운송장 번호를 필수로 기록하고 주문 상세에서 조회할 수 있다(출고 전에는 값이 없다). 택배사는 자유 문자열이며 택배사 API 연동·배송 추적·운송장 수정은 범위 밖이다.
- 결제완료 주문은 사용자가 취소할 수 있다(환불 수반). 출고 이후(출고·배송완료)는 취소를 거부한다.
- 결제 완료·출고 전(준비 중·보류) 주문은 라인 단위로 부분 취소할 수 있다. 취소한 라인만큼만 환불되고 그 라인의 재고만 복원된다. 마지막으로 남은 라인을 취소하면 주문 전체가 취소로 수렴하고, 이때 적용 쿠폰이 있으면 함께 복원된다.
- 라인 환불액은 라인 금액에서 쿠폰 할인을 라인 금액 비율로 안분(내림)한 몫을 뺀 금액이다. 다만 마지막으로 남는 라인은 안분 대신 결제 잔액 전액을 받아 내림으로 생기는 끝전과 배송비를 흡수한다.
- 주문은 부분 취소·부분 반품으로 환불된 금액의 누계를 보관한다. 이 누계가 0보다 크면(부분 취소·부분 반품 이력이 있으면) 전체 취소 개시·전체 반품 요청·전체 반품 환불을 거부한다 — 남은 라인은 라인 단위로만 처리해야 한다(이중 환불 방지).
- 배송 완료된 본인 주문(전체 또는 개별 라인)에 대해 회원이 반품을 사유와 함께 요청할 수 있다. 이미 반품 요청이 진행 중인 주문(또는 라인)에는 재요청할 수 없다. 다만 요청이 거절되면 같은 대상에 다시 요청할 수 있다.
- 관리자는 반품 요청을 승인해 환불을 완결하고 재고·쿠폰을 복원하거나, 거절해 주문(라인)을 원래 상태로 되돌릴 수 있다. 거절된 주문·라인은 결제 완료·배송 완료(라인은 주문됨) 상태로 남는다.
- 라인 반품이 승인되면 그 라인만 부분 환불·재고 복원된다. 전 라인이 종결(취소 또는 반품)되면 주문 전체가 반품 환불로 수렴한다.
- 관리자는 반품 요청을 거치지 않고도 배송 완료 주문을 즉시 전체 반품 환불(사유 기록) 처리할 수 있다(관리자 단일 액션 경로는 반품 요청/승인 워크플로와 별개로 유지된다). 출고됐으나 배송 완료 전인 주문은 취소도 환불도 안 되는 의도된 공백이다(배송 완료 확정 후 반품).
- 미결제 주문의 취소는 내부 보상 전이로만 일어난다 — 체크아웃 실패 시 동기 보상, 또는 payment 행 없이 유예가 지난 잔여를 주기 PENDING 스윕이 종결한다(이 범위에서 사용자 개시 미결제 취소는 없다).
- 관리자는 회원 이메일(정확 일치)과 선택적 결제 축 상태로 주문을 최신 주문 우선 페이지로 검색할 수 있다. 상태를 지정하지 않으면 전 상태를 반환한다. 탈퇴한 회원의 이메일로 검색하면 빈 페이지를 반환한다(존재 누출 방지).
- 사람이 읽는 주문번호를 부여한다.
- 생성 후 라인 구성은 불변이며, 변경은 취소 후 재주문이다.

### 결제

- 주문당 결제 1건을 두고, 결제 금액은 주문의 결제 대상 금액과 같다.
- 결제를 승인·실패(사유 기록)·취소(환불)할 수 있다.
- 결제를 부분 금액으로 취소(환불)할 수 있다. 부분 환불 누계가 결제 금액에 도달하면 결제는 전액 취소로 종결된다.
- 부분 환불은 재시도·중복 처리에 안전하다 — 같은 누계로의 재동기화는 이중 반영되지 않는다.
- 부분 환불 이력이 있는 결제는 전액 취소(환불) 요청을 거부한다. 남은 금액은 부분 환불로 마저 처리해야 한다.
- 결제 대상 금액이 0원(전액 할인이고 배송비도 0)이면 PG를 생략하고 자동 승인한다.
- 승인/실패가 동기 확정되는 수단만 지원한다(카드·간편결제·계좌이체). 가상계좌 등 입금 대기·비동기 통지 수단은 범위 밖이다.
- 승인 거래와 취소(환불) 거래를 각각 식별·추적할 수 있다. 부분 환불은 건별로 기록해 PG 취소 거래 ID로 추적한다.
- 이미 결제된 주문의 재결제 요청은 거부한다.
- 승인 응답이 유실돼 미확정(요청 상태)으로 남은 결제는 유예 후 주기 리컨실과 웹훅 통지가 PG 거래 상태 조회로 확정한다 — 승인이면 주문 결제완료, 거절·청구 미도달이면 실패 기록과 보상(재고·쿠폰 복원·주문 취소), 주문이 이미 취소된 지연 승인이면 고아 청구 환불(환불 선행 후 승인·취소를 한 커밋으로 기록해 중단 시 재시도 가능하게 유지). 결제 상태는 기록됐는데 주문측 종결이 남은 잔여(승인 기록 후 결제완료 전 중단, 거절 기록 후 보상 취소 실패)는 PENDING 주문 스윕이 발견해 같은 확정 경로가 종결한다. 확정은 반복 실행·중복 전달에 멱등하다.
- 웹훅 통지는 서명 검증을 거치되 페이로드의 결과를 신뢰하지 않는다 — 통지는 결제를 지목하는 트리거이고 확정 근거는 항상 PG 상태 조회다.

### 체크아웃·취소·반품 환불 (크로스 도메인)

- 체크아웃은 검증 → 주문 생성(미결제) → 재고 차감 → 쿠폰 확정 → 결제 → (성공) 결제완료·장바구니 정리 / (실패) 동기 보상 순으로 조율한다.
- 체크아웃은 장바구니 전체, 장바구니에서 선택한 일부 라인, 장바구니를 거치지 않는 즉시 주문(바로구매) 세 입력 경로를 지원한다. 세 경로 모두 같은 검증·조율·보상 흐름을 공유하고 차이는 주문 라인의 출처뿐이다. 미리보기는 같은 게이트·산식을 검증·산출까지만 실행하고 부작용을 남기지 않는다.
- 어느 라인이든 주문 불가·재고 부족이면 전체 체크아웃을 거부한다(선택된 라인 묶음 안의 부분 주문 없음). 문제 라인은 자동 제거하지 않는다.
- 결제 성공 전 모든 실패(재고·쿠폰·결제)는 그 자리에서 동기 보상(복원·주문 취소)한다.
- 결제 후 전체 취소는 환불을 선행하고, 성공 시 재고·쿠폰을 복원한다. 환불이 실패하면 주문은 결제완료로 남고 복원하지 않는다(재시도). 환불 앞에 취소 개시를 기록해 취소 진행 중 출고를 차단한다 — 취소와 출고가 겹쳐도 돈만 환불되고 상품이 출고되는 어긋남이 없다.
- 전체 반품 환불은 취소와 같은 순서(환불 선행 → 성공 시 재고·쿠폰 복원)로 조율하되, 재고 복원은 반품 상품의 회수·재판매를 가정하고 쿠폰 복원은 사용 기한 경과분이면 명목 복원이다(사용 시점 검증이 거부, 기한 연장·재발급 없음).
- 라인 단위 부분 취소·부분 반품도 같은 원칙을 따른다 — 라인 환불액을 확정한 뒤 결제 부분 환불을 거쳐 라인을 완결하고, 완결로 전 라인이 종결되면 주문 전체가 취소·반품 환불로 수렴한다.
- 상품 등록 → 첫 변형·재고 시딩 → 변형 활성화 → 상품 노출을 순차 조율한다.
- 취소·실패·반품 보상에서 재고·쿠폰 복원은 주문의 1회성 전이가 게이트해 중복 복원이 없다. 이 1회성 전이는 주문·결제의 낙관락이 직렬화해 동시 중복 취소·환불이 겹쳐도 한쪽만 커밋된다(진 쪽은 충돌 거부, 클라이언트 재시도). 라인 단위 처리도 라인 상태의 1회성 전이가 같은 원리로 중복 집행을 막고, 라인 환불 누계 가산이 주문 낙관락으로 직렬화된다. 전이 커밋과 복원 완료 사이 중단의 복원 유실은 잔여로 수용한다(운영 대사로 회복 — 이중 복원(오버셀)보다 복원 유실(팬텀 품절)이 안전). 이중 환불은 결제·주문의 1회성 전이가 구조적으로 거부한다.

## 비기능 요구사항

- **정합성**: 오버셀·쿠폰 이중 사용이 없고, 취소 시 재고·쿠폰이 정확히 한 번 복원되며, 동기 경로의 결제 이후 상태는 즉시 확정된다("결제됐는데 미결제" 창 없음). 응답 유실·프로세스 중단으로 미확정 결제가 남으면 결제 리컨실이 유예 + 스윕 주기 안에 PG 상태 조회로 수렴시킨다(돈과 주문 상태가 어긋난 채 방치되지 않음). 결제 요청 이전 중단으로 payment 행 없이 남은 PENDING 주문은 PG 조회 대상이 없어 주문 기준 PENDING 스윕이 유예 후 보상 종결(재고·쿠폰 복원·주문 취소)로 수렴시킨다. 결제는 종결 기록됐는데 주문측 종결이 남은 잔여(승인 × 미결제, 실패 × 미결제)는 PENDING 스윕이 발견해 결제 리컨실 확정 경로가 수렴시킨다(승인은 결제완료 완결, 실패는 보상 종결). 스윕·리컨실 보상의 재고 복원은 주문 단위 차감 완료 마커(체크아웃이 전 라인 차감 뒤 기록)가 게이트한다 — 증거 없는 잔여(차감 전·차감 중 중단)는 복원을 생략해 과복원(재고 증식→오버셀) 없이 과소복원(팬텀 품절)으로 남고 운영 대사로 회복한다. 결제 완료 후 장바구니 정리·재입고 알림 같은 이벤트 후처리는 트랜잭션 아웃박스로 발행돼 발행-커밋 사이 유실 창이 없고, 배치 릴레이가 최소 1회 전달하며 소비가 멱등이라 중복 전달을 흡수한다(전달 메커니즘·한계는 domain_model.md § 아웃박스 소유).
- **멱등(더블서밋 방어)**: unsafe 요청에 `Idempotency-Key` 헤더를 실으면 같은 키의 TTL 창 내 재요청을 HTTP 경계(멱등 필터)가 409로 거부한다 — 헤더 opt-in의 best-effort 방어이며 응답 재생은 없다.
- **이력 보존**: 주문은 라인·배송지·상품명·단가를 스냅샷으로 보관해 상품·변형·회원 변경과 무관하게 내역이 불변이다.
- **모듈성·확장성**: 도메인 간 물리 FK·객체 연관이 없고 컴파일 의존은 원칙적으로 두지 않으며(게이트된 선언 예외 하나 — 재입고 알림의 동기 조회 간선 — 는 빌드 검증이 비순환을 강제한다. 모듈 구조·의존 규칙은 `docs/architecture.md` 소유), 크로스 도메인은 파사드가 조율한다. 외부 PG는 벤더 중립 포트로 격리해 교체 가능하다.
- **테스트 용이성**: 결제 PG는 트리거 금액으로 실패·응답 유실을 결정론적으로 재현하는 fake, 이벤트 전달은 외부 브로커 없이 DB 아웃박스·배치 릴레이로 이뤄져 오프라인으로 거래 흐름을 검증한다.
- **동시성·재고 차감**: 동시 주문 경합(재고 차감·쿠폰 사용)과 동시 쿠폰 발급 경합(발급 한도), 동시 취소·환불 경합(복원 이중 실행)에서도 정합성이 깨지지 않는다. 재고는 주문 시 즉시 차감하며 예약(홀드) 모델이 아니다.
- **보안·자격**: 담기·주문·발급은 자격 활성 회원만, 주문·발급분 접근은 본인 소유만 허용한다(미소유는 존재 누출 방지로 미존재 취급). 구매자 셀프서비스는 토큰 주체로만 회원을 식별하고, 관리자 오퍼레이션은 관리자 역할 토큰만 허용한다. 패스워드는 bcrypt 해시로만 저장하고 평문·복호화 가능한 형태를 저장하지 않는다. 로그인은 클라이언트 IP당 창 한도를 넘는 시도를 429로 거부해 무제한 브루트포스·크리덴셜 스터핑을 막고, 모든 응답에 시큐리티 헤더 최소셋(nosniff·no-store)을 싣는다(레이트리밋 키·창·한도 근거는 decisions.md ADR-4, 헤더 보류 근거는 ADR-5).

## 외부 연동

- **결제 게이트웨이**: 벤더 중립 포트(`PaymentGateway`, approve·cancel·cancelPartially[부분 환불]·inquire[거래 상태 조회]). 구현은 외부 어댑터이며 개발은 실패·응답 유실을 시뮬레이션하는 fake. 상태 조회 키는 가맹점 참조(결제 ID)다 — 응답 유실 시 호출자가 PG 거래 ID를 모르기 때문.
- **메일 게이트웨이**: 비밀번호 재설정·이메일 인증·재입고 알림 메일은 벤더 중립 메일 게이트웨이 포트로 발송한다. 현재 구현은 실제 메일을 보내지 않고 수신자·내용을 로그로만 남기는 개발용 스텁이다(실 메일 발송기 연동은 향후 확장).

## 제약·전제

- 배포 단위는 앱 4개다 — app-api(구매자 셀프서비스 API)·app-admin(관리자 오퍼레이션·관리자 계정 기동 시딩)·app-batch(결제 리컨실·PENDING 스윕·아웃박스 릴레이·PG 웹훅 수신)·app-migration(전 도메인 스키마 마이그레이션 원샷). 네 앱은 도메인 모듈을 공유하되 프로세스를 분리한다(모듈 구조는 `docs/architecture.md` 소유).
- 미확정 결제 확정은 배치 앱(app-batch)의 `@Scheduled` 리컨실 스윕이 담당하고, PG 웹훅 수신도 같은 앱에서 같은 확정 경로를 공유한다. 확정 경로는 생성 후 유예(`payment.reconciliation.stale-after`)가 지난 결제만 손대 동기 체크아웃과 경합하지 않는다 — 유예 전 웹훅 통지는 무시되고 PG 재전달·주기 스윕이 수렴을 보장한다.
- 결제 요청 이전 중단으로 payment 행 없이 남은 PENDING 주문은 별도 `@Scheduled` 주문 기준 PENDING 스윕(app-batch)이 담당한다. 생성 후 유예(`order.reconciliation.stale-after`, 결제 리컨실 유예 이상)가 지난 주문 중 payment 행이 없는 주문만 직접 보상 종결한다(관할 판정 상세는 [`domain_model.md`](./domain_model.md) § PENDING 주문 스윕 소유).
- 스케줄 스윕 동시 실행 제어(ShedLock)·헬스체크·Readiness 설계·Redis 공유 인프라 채택 근거(멱등·레이트리밋·리프레시토큰·분산락)·시큐리티 응답 헤더·CORS·인증·인가 아키텍처(Spring Security 필터체인·JWT·관리자 가드·관리자 시딩)의 결정 배경과 기각한 대안은 [`decisions.md`](./decisions.md)가 소유한다.
- 통화 단일 가정·크로스 트랜잭션 부분 실패 복구 범위·아웃박스 발행 메커니즘은 [`domain_model.md`](./domain_model.md)가 소유한다.
- 관리자 쓰기·운영 조회 오퍼레이션(회원 지정 조회·이메일 검색·정지/해제, 상품·변형·이미지·카테고리 관리, 재고 관리, 쿠폰 정책·발급분 관리, 주문 출고·배송완료·보류/해제·반품 승인/거부·환불·상태 목록·검색, 문의 답변, 리뷰 삭제)은 전부 별도 Spring Boot 앱 app-admin의 `/api/v1/admin/**` 표면에 있고 도메인 Reader·Info 경계를 지킨다(강제 메커니즘은 decisions.md ADR-6 소유).

## 향후 확장

- 실 PG 연동(fake PG 어댑터 교체).
- 실 메일 발송기 연동(로그 스텁 메일 게이트웨이 교체).
- 상품 변형 옵션의 구조 정규화(옵션 엔티티·상품 단위 옵션 타입·글로벌 옵션 카탈로그·파셋 검색).
- 상품 대표가·가격대 집계 읽기모델, 상품당 총재고 집계.
