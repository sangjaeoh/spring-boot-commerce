# Architecture

## 언제

- 새 모듈·앱·도메인을 추가하거나 모듈 경계를 정할 때.
- 어떤 계층이 어떤 계층을 의존해도 되는지 판정이 필요할 때.
- 새 클래스를 어느 패키지(구역)에 둘지, 패키지 이름을 어떻게 지을지 정할 때.
- 계약(제공·요구)과 구현을 어디에 두는지 정할 때.
- 크로스 도메인 호출·조회와 이벤트 전파 방식을 정할 때.
- 리포지토리를 어디에 두고, 누가 조회·쓰기하는지를 정할 때.
- 트랜잭션 경계와 이벤트 발행·소비 시점을 정할 때.
- 웹 표면(컨트롤러·API 문서화·페이징·인가)을 만들 때.
- 도메인 모듈의 독립 서비스 추출을 판정할 때.

## 규칙

### 컨벤션 플러그인

- 모듈은 유형별 컨벤션 플러그인 하나로 규칙을 적용한다.
  - 규칙을 `build.gradle.kts`마다 직접 복붙하지 않는다.
  - 품질 도구(Spotless·NullAway·Error Prone)는 → [code-quality](code-quality.md)로 관리한다.
- 플러그인은 `build-logic/src/main/kotlin/convention.{name}.gradle.kts`에 둔다.

  | 플러그인                   | 적용 대상       | 소유 규칙                                                                          |
  | -------------------------- | --------------- | ---------------------------------------------------------------------------------- |
  | `convention.domain-module` | `domain-shared` 제외 `domain-{name}` | `domain-shared`·`common-{name}`·`event-{name}`, 명시 등록한 다른 `domain-{name}` 모듈만 의존 가능 |
  | `convention.domain-shared-module` | `domain-shared` | `common-{name}` 모듈만 의존 가능 |
  | `convention.app-module`    | `app-{name}`    | `domain-{name}`·`infra-{name}`·`common-{name}`·`event-{name}`·`query-{name}` 모듈만 의존 가능 |
  | `convention.infra-module`  | `infra-{name}`  | `common-{name}` 모듈만 의존 가능                                                  |
  | `convention.common-module` | `common-{name}` | `common-core` 방향으로만 단방향 의존 가능                                          |
  | `convention.event-module`  | `event-{name}`  | `common-event`만 의존 가능                                                        |
  | `convention.query-module`  | `query-{name}`  | `domain-{name}`·`common-{name}`, 전환 시 명시 등록한 `event-{name}` 모듈만 의존 가능 |

- 계층 컨벤션 플러그인은 첫 모듈 생성 시 함께 구현한다.
- `convention.java-base`와 `convention.java-common`을 적용해 품질 게이트를 함께 붙인다(도구 상세는 → [code-quality](code-quality.md)).
- 모듈 간 의존은 기본적으로 `implementation`을 쓴다.
- `api`는 공개 시그니처 재노출이 필요할 때만 사용한다.
- 새 모듈은 `module-{name}/{prefix}-{name}` 아래 `build.gradle.kts` 하나로 만들고 settings에 등록한다.
  - 예: `module-domains/domain-{name}/build.gradle.kts`에 `convention.domain-module`만 적용한다.

### 모듈 지도 (7계층)

- 모듈은 아래 계층 경계와 의존 방향을 지킨다.
- 의존은 한 방향으로만 흐른다.
  - 경계는 컴파일 의존성으로 강제한다.

  | 계층 | 역할 | 의존 가능 대상 |
  | ---- | ---- | -------------- |
  | `module-apps/` | 실행 애플리케이션 계층 | `domain-{name}`·`infra-{name}`·`common-{name}`·`event-{name}`·`query-{name}` |
  | `module-query/` | 조회 계층 — 합성이 성립하지 않는 크로스 도메인 조회 | `domain-{name}`·`common-{name}`·전환 시 명시 등록한 `event-{name}` |
  | `module-domains/` | 도메인 계층 — 계약·유스케이스·어댑터·도메인 모델을 모두 소유 | `domain-shared`·`common-{name}`·`event-{name}`·명시 등록한 다른 `domain-{name}` |
  | `module-infra/` | 인프라 계층 — common 소유 계약의 구현 | `common-{name}` |
  | `module-events/` | 이벤트 계약 계층 — 도메인이 공표하는 이벤트 record | `common-event` |
  | `module-common/` | 공용 계층 | `common-core`는 제로 의존, 나머지는 `common-core` 방향으로만 단방향 의존 |
  | `module-tests/` | 테스트 계층 | 모든 모듈 |

- 모듈 경계는 빌드가 컴파일·테스트 시점에 강제한다.
- 계층 표는 의존 방향 규칙이다.
- 모듈 단위 세부 화이트리스트는 컨벤션 플러그인 표가 소유한다.
- `module-tests/`는 이 의존 흐름 밖의 테스트 전용 계층이며, 플러그인 화이트리스트 대상이 아니다.

### 구현 배치 판정

- 새 계약과 구현의 위치는 아래 표로 판정한다.

  | 구현 대상 | 계약 소유 | 구현 위치 |
  | --------- | --------- | --------- |
  | 자기 애그리거트 상태의 영속 | 도메인 `application/required` | Spring Data는 인터페이스 자체로 종결, 그 외는 같은 도메인 `adapter/persistence` |
  | 도메인 의미가 없는 기술 기능 | `common-{name}` — 단일 소비자면 그 도메인 | `infra-{name}` — 도메인 소유면 그 도메인 `adapter` |
  | 외부 시스템 기능(도메인 요구) | 도메인 `application/required` | 같은 도메인 `adapter/integration` |
  | 크로스 도메인 이벤트 | 발행 도메인의 `event-{name}` | 발행은 발행 도메인 `application`, 소비는 소비 도메인 `application` |
  | 합성이 성립하지 않는 크로스 도메인 조회 | `query-{name}` | 같은 모듈 |

- 판정 기준은 호스팅 주체가 아니라 상태·기능의 주인이다.
- 남이 관리하는 저장소(관리형 캐시·스토리지)도 원본 상태를 담으면 영속으로 판정한다.
- 재계산 가능한 파생·조회 결과의 투명 캐싱은 영속 판정 대상이 아니다 → [caching](caching.md)의 캐시 판정.
- 판정 순서: 계약의 도메인 어휘 유무를 먼저 판정한다.
- 어휘가 없으면 기술 계약, 있으면 도메인 계약이다.
- 도메인 어휘 판정: 계약의 이름·시그니처에 그 도메인의 엔티티·값 객체·용어집 등재 용어가 나타나면 도메인 계약이다.
- 계약의 이름·시그니처가 벤더·프로토콜·범용 기술 명사(메일·SMS·파일·저장소)만 담으면 기술 계약이다.
- 예: `{Domain}Store`처럼 도메인 엔티티·값 객체 이름을 담은 계약은 도메인 계약이다.
- 예: `SmsSender`·`FileStorage`처럼 벤더·프로토콜·범용 기술 명사만 담은 계약은 기술 계약이다.
- 도메인 용어와 기술 명사가 섞인 계약은 도메인 계약으로 판정한다.
- 신규 도메인 용어는 같은 변경에서 용어집에 등재해 판정 근거로 삼는다.
- 기술 계약은 common 소유·infra 구현을 택한다.
- 기술 계약은 소비 도메인이 여럿이어도 소유 도메인 지정·이벤트 소비 방식으로 풀지 않는다.
- 소비자가 한 도메인뿐인 기술 계약은 그 도메인이 계약을 소유한다.
- common 승격은 같은 계약을 요구하는 두 번째 소비자가 실제로 나타난 때에만 한다.
- 도메인 계약은 common으로 올리지 않는다.
- 외부 시스템 기능 행과 소유 도메인 지정은 도메인 계약에만 적용한다.
- 여러 도메인이 같은 외부 기능을 도메인 계약으로 요구하면 그 기능을 소유할 도메인을 지정한다.
- 소유하지 않은 도메인은 결과를 이벤트로 소비한다.
- 응답 합성은 앱 파사드가 소유 도메인 provided 조합으로 한다.
- 동기 예외는 → 크로스 도메인 상호작용.

### 앱 구성

- 앱은 기본 4종이다.
- 최소 구성은 `app-api`와 `app-migration`이다.
- 각 앱은 하나의 실행 모듈(`app-{name}`)이다.

  | 앱              | 책임              | 데이터 접근                                   |
  | --------------- | ----------------- | --------------------------------------------- |
  | `app-api`       | API·파사드        | 도메인·query 모듈 provided 경유               |
  | `app-admin`     | 어드민·백오피스   | 도메인·query 모듈 provided 경유               |
  | `app-batch`     | 배치·정리 잡      | 도메인·query 모듈 provided 경유               |
  | `app-migration` | Flyway 독립 실행기 | 런타임 데이터 접근 없음                       |

- `app-migration`이 마이그레이션을 실행한다.
- 실행 앱은 부팅 시 Flyway를 실행하지 않는다.
- 실행 앱은 다음 불변식을 따른다.
  - `spring.jpa.open-in-view: false` — 경계 밖 지연로딩을 막는다.
  - `spring.jpa.hibernate.ddl-auto: validate` — 스키마는 Flyway가 관리한다.

### 앱 모듈 구조 (`app-{name}/`)

- 앱 모듈은 역할별로 패키지를 중심에 두고 구성한다.
- 공통 규칙은 앱 모듈 공통 섹션이 소유한다.

#### `app-api`
- 중심 패키지: `web`, `facade`
- `web/v{n}`은 API 버전과 리소스별 슬라이스를 패키지로 분리한다.
- `facade`는 도메인 provided를 조합해 응답을 만든다.

#### `app-admin`
- 중심 패키지: `web`
- `web/v{n}/admin/{리소스}` 슬라이스가 어드민 컨트롤러를 소유한다.
- 어드민 컨트롤러는 클래스 레벨 `@Admin`만 사용한다.
- 어드민 URL 네임스페이스는 게이트웨이와 방화벽 차단 축이다.

#### `app-batch`
- 배치 작업은 앱 고유 진입점 패키지가 소유한다.
- `web`은 배치 진입점이 필요할 때만 둔다.
- 스케줄 진입점(`@Scheduled`)은 앱 모듈만 소유한다.
- 도메인·infra 모듈에 `@Scheduled`를 두지 않는다.

#### `app-migration`
- Flyway 마이그레이션만 실행한다.
- SQL 소유는 → 도메인 모듈 공통.
- 도메인·infra·query 모듈은 런타임 전용 의존으로만 싣는다(마이그레이션 리소스 취득 목적).
- 런타임 데이터 접근을 하지 않는다.
- 도메인·인프라 빈을 인스턴스화하지 않는다.
- 컴포넌트 스캔 범위를 앱 자신으로 한정한다.

### 앱 모듈 공통
- `web`은 HTTP 전송(REST·WebSocket·SSE)을 담당한다.
- WebSocket·SSE 상세 규칙은 실물 필요 시 추가한다.
- 비-web 진입점은 형제 패키지로 둔다.
- `web/v{n}`은 버전을 바깥 축, 리소스를 안쪽 축으로 구성해 `/api/v{n}/{리소스}` URL을 미러링한다.
- `facade`는 도메인 조립과 크로스 도메인 조율을 담당한다.
- `*View` 응답은 `facade/view`가 소유한다.
- 앱은 도메인·query 모듈의 provided 인터페이스를 주입받는다.
- 구현 서비스를 직접 주입하지 않는다.
- 조회 경로(도메인 provided·파사드 합성·query 모듈)의 선택 기준은 모든 앱에 공통이며 → 크로스 도메인 조회가 소유한다.
- `event/listener`는 앱 고유 전달·관찰 관심사의 이벤트 구독만 소유한다 → 크로스 도메인 상호작용.
- `web/auth`는 앱 소유 메타 애노테이션 `@Admin`, `@Authenticated`, `@Anonymous`를 제공한다.
- `@Admin`은 클래스 레벨 전용이다.
- `@Authenticated`·`@Anonymous`는 필요 시 메서드 레벨에 쓸 수 있다.
- 앱 소유 마커는 제약만 더하고 공개 표면은 `permitAll` 공개 경로로 둔다.
- `SecurityConfig` 필터 체인과 메서드 시큐리티는 이중 게이트로 401/403을 구분한다.
- 경계 도달 예외의 HTTP 변환은 common-web 소유 전역 예외 핸들러가 담당한다.
- 동일 응답 DTO를 공유하는 경우 본인/공개 슬라이스가 소유하고 어드민이 참조한다.
- `web` 핸들러와 request/response는 `@Operation`, `@ApiResponse`, `@Parameter(description)`, `@Schema`로 명시한다.
- OpenAPI 구성은 인증 요구와 인증 주체 주입 파라미터의 문서 처리를 전역으로 선언한다.
- 핸들러마다 반복 선언하지 않는다.
- 페이징 GET은 common-web `PaginationRequest`(`@Valid @ParameterObject`)를 사용한다.
- 클라이언트 요청은 1-based 페이지를 받는다.
- 도메인 `Pageable`은 0-based를 유지한다.
- 페이지 응답은 common-web `PaginationResponse`의 `page` 컴포넌트로 싣는다.
- 응답의 페이지 번호는 도메인 0-based를 1-based로 보정한다.
- 요청 변환은 `PaginationRequest.zeroBasedPage()`가 소유한다.
- 응답 보정은 `PaginationResponse.from(Page)`가 소유한다.

### 도메인 모듈 구조 (`domain-{name}/`)

- 도메인 모듈은 세 구역으로 구성한다: `application`, `adapter`, `domain`.

  | 구역 | 책임 | 의존 가능 구역 |
  | ---- | ---- | -------------- |
  | `application` | 제공 계약·요구 계약·경계 모델·유스케이스 구현 | `domain` |
  | `adapter` | `required` 계약의 구현 | `application`·`domain` |
  | `domain` | 엔티티·값 객체·도메인 어휘 | 없음 |

#### `application`

- `provided` — 모듈이 제공하는 진입 계약 인터페이스.
  - 모듈 밖 호출은 provided 인터페이스로만 한다.
  - 캐시 이름 상수 클래스는 예외로 `provided`에 둔다 → [caching](caching.md)의 이름·키·버전.
- `required` — 모듈이 요구하는 계약 인터페이스.
  - Spring Data 리포지토리: 영속 계약 규칙은 → 리포지토리 접근 범위.
  - QueryDSL 프래그먼트 인터페이스는 required에 평탄하게 둔다.
  - QueryDSL 프래그먼트 구현은 `adapter/persistence`가 소유한다.
  - Spring Data로 표현할 수 없는 영속 계약(원자 연산·비관계형 저장소).
  - 외부 시스템 기능 계약: 계약명은 벤더 중립이어야 한다.
  - 타 도메인에 대한 요구는 required로 선언하지 않는다 → 크로스 도메인 상호작용.
- `info` — 경계용 불변 record.
- 이벤트 record는 두지 않는다.
- 발행 이벤트는 이벤트 모듈이 소유한다 → 이벤트 모듈 구조.
- 이벤트 소비자는 `application`이 소유하고 package-private로 둔다.
- 유스케이스 구현 서비스와 내부 협력 서비스는 `application`에 평탄하게 둔다.
  - provided 구현 서비스는 package-private로 선언한다.
  - 모듈 밖은 provided 인터페이스만 본다.
  - 모듈 내부 전용 협력 서비스는 provided 계약을 만들지 않는다.
  - 서비스 역할 어휘(Reader·Appender·Modifier·Remover·Processor·Validator)는 → [coding-conventions](coding-conventions.md)의 네이밍 규칙을 따른다.

#### `adapter`

- `required` 계약의 구현체를 소유한다.
- 모듈 밖에 노출하지 않는다.
- 하위 패키지는 실물 구현이 생길 때 계약 성격대로 만든다.
  - `persistence` — 자기 상태의 영속 구현(비-Spring-Data 저장소, QueryDSL 프래그먼트 구현).
  - `integration` — 외부 시스템 호출 구현.
- 모의 구현과 실벤더 구현은 같은 하위 패키지에 두고 구성(프로파일)으로 선택한다.
- `application`은 선택에 관여하지 않는다.
- 어댑터 전용 구성(클라이언트 설정 빈)은 해당 adapter 하위 패키지가 소유한다.

#### `domain`

- JPA 매핑 타입(엔티티·`@Embeddable`·`@MappedSuperclass`), `AttributeConverter`, 값 객체, enum, `{Name}ErrorCode`와 예외를 소유한다.
- 하위 패키지는 실물이 생길 때 성격대로 만든다.
  - `exception` — `{Name}ErrorCode`와 예외.
  - `converter` — `AttributeConverter`.
- 그 외 타입(엔티티·값 객체·enum)은 `domain` 루트에 평탄하게 둔다.
- JPA 매핑 클래스는 모듈 밖 노출 금지.
- 다른 구역을 의존하지 않는다.

#### 노출 규칙

- 모듈 밖에서 임포트 가능한 것: `application`의 provided·info, `domain`의 비-JPA 타입(enum·값 객체·예외).
- 캐시 이름 상수 클래스도 임포트 가능하다 → [caching](caching.md)의 이름·키·버전.
- required·`adapter` 전 구역·JPA 매핑 타입·`AttributeConverter`는 모듈 밖 비노출.
- 예외: JPA 매핑 타입은 query 모듈에 한해 가시다 → query 모듈 구조.
- provided·info 시그니처에 JPA 매핑 타입을 쓰지 않는다.

### 도메인 모듈 공통

- `domain-shared`는 두 도메인 이상이 쓰는 값 객체와 `AttributeConverter`만 소유한다.
- `domain-shared`는 구역 구조와 도메인 모듈 유형 규칙(간선·이벤트 의존)을 적용하지 않는다.
- `domain-shared`는 `convention.domain-shared-module`로 `common-{name}`만 의존한다.
- 하나의 도메인만 사용하는 값 객체는 해당 도메인의 `domain` 구역이 소유한다.
- 도메인별 독립 스키마를 소유한다.
- 스키마는 단일 데이터베이스 안의 스키마다.
- 스키마 생성은 도메인별 Flyway 인스턴스가 수행한다(`createSchemas: true`).
- 마이그레이션 SQL은 도메인 모듈 리소스가 소유한다.
- 마이그레이션 파일은 도메인별 `db/migration/{name}/`에 `V{n}__{설명}.sql` 형식으로 둔다.
- `app-migration`이 구성에 등록된 스키마마다 독립 Flyway 인스턴스로 실행한다.
- 적용된 마이그레이션 파일은 수정하지 않고 새 버전을 추가한다.
- 스키마 파괴 변경(컬럼·테이블 삭제, 리네임, 타입 축소, 기존 컬럼의 NOT NULL 추가)은 expand-contract 2단계로 나눈다.
  - expand 단계는 새 구조를 추가하고 코드는 신·구 구조를 병행 처리하며 백필한다.
  - 백필은 소량이면 마이그레이션 SQL이, 대형 테이블이면 배치 잡(`app-batch`)이 소유한다.
  - contract 단계는 구 구조 참조가 사라진 배포가 완료된 뒤 별도 마이그레이션으로 구 구조를 제거한다.
  - 근거는 롤링 배포 중 신·구 코드가 같은 스키마를 공유하는 것이다.
- 리네임은 추가·전환·제거로 수행한다.
- `RENAME` 문은 contract 파일에서도 쓰지 않는다.
- 기존 컬럼의 NOT NULL 추가는 백필 완료 후 별도 단계로 적용한다.
  - 대형 테이블은 `CHECK ... NOT VALID` 추가, `VALIDATE CONSTRAINT`, `SET NOT NULL` 순서로 적용해 풀스캔 잠금을 피한다.
- 신규 컬럼의 NOT NULL과 상수 DEFAULT 동시 추가는 파괴 변경이 아니다.
- 타입 축소 판정은 이전 스키마 상태가 필요하므로 리뷰로 검증한다.
- 파괴 문을 담는 contract 마이그레이션 파일은 첫 줄에 `-- contract` 주석을 둔다.
- 마이그레이션 DDL은 타 도메인 스키마를 참조하지 않는다(크로스 스키마 FK·뷰 금지).
- Flyway 인스턴스는 `createSchemas: true`와 `defaultSchema`를 지정한다.
- `createSchemas: false`는 히스토리 테이블 스키마 부재로 첫 migrate가 실패하므로 쓰지 않는다.
- 마이그레이션 파일은 스키마 생성 문 없이 테이블·객체만 다룬다.

### 크로스 도메인 상호작용

- 크로스 도메인 쓰기 전파는 도메인 이벤트로 한다.
  - 발행은 도메인 쓰기 서비스가 상태 변경과 같은 트랜잭션 안에서 수행한다.
  - 파사드·앱 등 조합 계층은 발행하지 않는다.
  - 발행은 전달 보장 계약(아웃박스)을 경유해 비즈니스 쓰기와 원자적으로 기록한다.
  - 전달 보장의 계약은 `common-event`가 소유한다.
  - 전달 보장의 구현(아웃박스 저장·재전달·dead letter)과 그 스키마는 infra가 소유한다.
  - 아웃박스는 데이터베이스마다 하나이며 도메인별로 두지 않는다.
  - 원자성은 같은 데이터베이스·같은 트랜잭션으로 성립한다.
  - 소비는 멱등하게 처리한다.
  - 소비 실패는 재시도와 dead letter로 격리한다.
  - 전달 보장 계약 밖의 이벤트 구독은 관찰 용도(로깅·메트릭)로만 쓴다.
- 이벤트 소비 배치는 반응의 소유자를 따른다.
  - 도메인 상태를 바꾸는 반응은 소비 도메인의 `application` 소비자가 소유한다.
  - 앱의 이벤트 구독은 앱 고유 전달·관찰 관심사에 한정한다.
  - 앱이 늘 때 복사해야 하는 소비자는 앱 소유가 아니다.
  - 소비 선언은 발행 도메인의 이벤트 모듈 의존으로 한다.
  - 이벤트 소비 등록은 동기 간선과 달리 요구되지 않는다 → 이벤트 모듈 구조.
  - 상호 소비는 정상이며 순환을 만들지 않는다 → 이벤트 모듈 구조.
  - 프로세스 내부 전달에서는 실행 앱이 발행 도메인의 이벤트를 소비하는 도메인을 함께 싣는다.
  - 싣지 않으면 그 소비자는 실행되지 않는다.
- 소비자의 값 처리는 값의 성격을 따른다.
  - 상태 투영·판정 재료 값의 미지 값은 시끄럽게 실패시켜 재시도·dead letter로 격리한다.
  - 무시·로깅 처리는 장식 값에만 허용한다.
  - 단일 값의 등호 분기는 이벤트 모듈 열거형 승격으로 충분하다 → 이벤트 모듈 구조.
  - 소비 도메인이 값 조합·이력 해석으로 발행 도메인의 판정을 재구성하면 발행 도메인의 세분화된 이벤트 타입으로 승격한다.
- 동기 호출은 상대 도메인의 판정 결과가 응답에 필요한 경우와 조회에 한해 사용한다.
- 기록 전파는 동기로 하지 않는다.
- 허용된 동기 호출은 상대 도메인의 provided 직접 의존으로 한다.
  - 도메인 간 의존 간선은 컨벤션 플러그인에 명시 등록한 것만 허용한다.
  - 간선은 비순환이어야 한다.
  - 간선 위에서도 임포트 범위는 노출 규칙을 따른다.
- 크로스 도메인 계약을 도메인 required로 선언하고 앱이 배선하는 방식은 사용하지 않는다.

### 크로스 도메인 조회

- 조회 요구는 축(필터·정렬·페이지네이션·집계의 기준 도메인)의 위치로 판정한다.
- 단일 도메인 조회는 그 도메인의 provided로 한다.
- 결과가 여러 도메인에 걸쳐도 축이 한 도메인이면 앱 파사드 합성으로 푼다.
  - 주 도메인 provided가 목록·페이지를 소유한다.
  - 타 도메인 데이터는 주 결과의 ID 목록으로 벌크 조회해 장식한다.
  - 반복 단건 조회는 금지한다.
- 축이 두 도메인 이상에 걸쳐 합성이 성립하지 않는 조회만 query 모듈이 소유한다 → query 모듈 구조.
  - 합성이 성립하는 조회는 query 모듈에 두지 않는다.
- 도메인이 판정 목적으로 이벤트를 소비해 유지하는 로컬 투영 위의 조회는 그 도메인 provided의 정상 경로다.

### query 모듈 구조 (`query-{name}`)

- query 모듈은 합성이 성립하지 않는 크로스 도메인 조회만 소유한다.
- 모듈은 소비 표면 단위로, 실물 요구가 나타난 때에만 만든다.
- 도메인 모듈들을 의존하고 엔티티를 조회 구현에 사용한다.
- JPA 매핑 타입 가시는 query 모듈 유형만의 특권이다.
- 조회는 모듈 자체의 쿼리 구현(QueryDSL·`EntityManager`)으로 한다.
- 도메인 리포지토리를 임포트하지 않는다.
- 도메인 상태를 변경하지 않는다.
- 쓰기는 진화 단계의 자체 투영 저장소 적재에 한정한다.
- 조회 구현은 읽기 전용 트랜잭션 안에서 경계 반환 변환까지 완료한다.
- 예외: 캐시하는 조회의 트랜잭션 분리는 → [caching](caching.md)의 트랜잭션 분리.
- 예외: 이름 상수 배치는 → [caching](caching.md)의 이름·키·버전.
- 소프트삭제 조회 기본은 → [coding-conventions](coding-conventions.md)의 메서드 네이밍을 따른다.
- 기능은 provided 인터페이스로 제공하고 구현은 package-private로 둔다.
- 도메인 구역 구조는 두지 않는다.
- provided·구현 네이밍은 도메인 모듈과 같은 규칙을 따른다 → [coding-conventions](coding-conventions.md)의 네이밍.
- 경계 반환은 모듈 소유 `Info` record로 한다.
- 엔티티·JPA 타입을 시그니처에 노출하지 않는다.
- 계약의 모양은 소비 표면이 주도하되, 소유와 노출은 모듈이 한다.
- 의존은 앱 → query 모듈 방향만 존재한다.
- 도메인·infra·이벤트 모듈은 query 모듈을 의존하지 않는다.
- 저장소 분리로 진화할 때 provided 계약을 유지한 채 구현을 교체한다.
  - 초기 적재는 읽기 전용 크로스 스키마 백필로 한다.
  - 증분은 이벤트 구독으로 전환한다.
  - 전환 시 event 모듈 의존을 플러그인 화이트리스트에 등록한다.
  - 자체 투영 저장소의 스키마·마이그레이션 SQL은 query 모듈 리소스가 소유한다.
  - 투영 저장소 마이그레이션도 `app-migration`이 실행한다.

### 이벤트 모듈 구조 (`event-{name}`)

- 이벤트 모듈은 도메인이 공표하는 이벤트 계약을 소유한다.
- 발행 도메인마다 이벤트 모듈 하나를 둔다.
- 도메인이 발행하는 이벤트 record 전수는 그 도메인의 이벤트 모듈에 둔다.
- 공통 이벤트 계약(이벤트 표식·발행·소비자 계약)은 `common-event`가 소유한다 → common 모듈 구조.
- 이벤트 모듈에는 그 모듈에서 태어난 타입만 둔다(계약 태생 어휘).
- 도메인 내부 타입(엔티티·엔티티 enum·JPA 타입)은 반입하지 않는다.
- 페이로드의 어휘 값은 문자열을 기본으로 한다.
- 식별자·시각 등 언어 표준 타입은 그대로 싣는다.
- 소비자가 값으로 분기하는 판정 재료만 이벤트 모듈 소유 열거형으로 승격한다.
- 값 조합·이력 해석이 필요한 분기는 열거형이 아니라 이벤트 타입 승격 대상이다 → 크로스 도메인 상호작용.
- 발행 도메인은 내부 값을 계약 열거형으로 망라 매핑(exhaustive switch)해 변환한다.
- 내부 값이 추가되면 매핑 누락이 컴파일 실패로 드러나야 한다.
- 이벤트 모듈은 프레임워크에 의존하지 않는다.
- 언어 표준·null 계약 애노테이션(JSpecify)·`common-event` 계약만 사용한다.
- 이벤트 식별자·발생 시각은 발행 측이 생성해 값으로 전달한다.
- 이벤트 모듈은 생성기를 소유하지 않는다.
- 의존 제한(`common-event`만 의존)의 강제는 → 빌드가 강제하는 불변식.

### common 모듈 구조

- 공용 코드는 역할에 따라 common 모듈에 둔다.
- 도메인 로직과 도메인 지식은 common에 두지 않는다.

#### common 모듈 공통 규칙

- common 모듈은 실제 필요가 생긴 때에만 만든다.
- 소비자 없는 모듈은 미리 만들지 않는다.
- 배치 후보가 여럿이면 의존 범위가 가장 좁은 모듈에 둔다.
- 여러 도메인에서 공통으로 쓰이는 책임만 공용 모듈로 분리한다.
- 앱 진입 표면에서만 소비되는 책임(앱 표면 관심사)의 모듈은 도메인 모듈이 의존하지 않는다.

| 모듈 | 책임 |
| --- | --- |
| `common-core` | 프레임워크 의존이 없는 기본 공용 코드 |
| `common-jpa` | 여러 모듈이 공유하는 영속성 지원 책임 |
| `common-event` | 이벤트 표식·발행·소비자 계약(프레임워크 무의존) |
| `common-auth` | 인증·인가 공통 책임 |
| `common-web` | 웹 공통 책임 |

### infra 모듈 구조

- infra 모듈은 common이 소유한 계약의 구현만 소유한다.
- 계약과 구현의 배치는 → 구현 배치 판정.
- 모듈은 기술 단위로 나눈다(`infra-{name}`).
- 도메인 계약의 구현을 두지 않는다.
- 도메인 계약의 구현은 그 도메인의 `adapter` 구역이 소유한다.
- 구현 실물이 생길 때만 모듈을 만들고, 구현이 모두 사라진 모듈은 제거한다.
- 구현 선택과 배선은 앱이 소유한다.
- 도메인은 common 계약만 의존한다.
- 구역(`application`·`adapter`·`domain`)을 두지 않는다.
- 모듈 전체가 구현이다.
- 하위 패키지는 구현하는 계약 단위로 나눈다.
- 한 패키지는 한 계약의 구현과 그 전용 헬퍼를 담는다.
- infra 타입은 모듈 밖에서 임포트하지 않는다.
- 소비는 common 계약으로만 한다.
- 구현 클래스는 package-private로 선언한다.
- 빈 등록은 컴포넌트 스캔에 맡긴다.
- infra가 스키마를 소유하면 그 마이그레이션 SQL도 해당 infra 모듈 리소스가 소유한다.
- `app-migration`이 런타임 전용 의존으로 싣고 실행한다.

### 공통 규칙

- 아키텍처 테스트는 모듈 경계·구역 경계와 의존 방향을 검증하는 품질 게이트다.
- 새 모듈이 누락되지 않도록 검증 대상은 모듈 목록에서 파생한다.

### 아키텍처 테스트 모듈

- 아키텍처 테스트는 `module-tests/test-architecture` 모듈이 소유한다.
- 앱이나 애플리케이션 모듈 안에 두지 않는다.
- 테스트는 전체 모듈 그래프를 기준으로 경계를 검증한다.
- `test-architecture`는 계층 규칙이 아니라 품질 게이트 플러그인(`convention.java-base`, `convention.java-common`)만 적용한다.
- 빌드 검증 테스트(의존 그래프 기반)도 `module-tests/` 하위 모듈이 소유한다.

### 패키지 네이밍·배치

- 모든 모듈의 베이스 패키지는 `{그룹}.{모듈 접두}.{모듈명}` 형식으로 짓는다.
- `{그룹}`은 프로젝트 전체가 공유하는 단일 그룹 접두다.
- `{모듈 접두}`와 `{모듈명}`은 모듈 아티팩트명 `{prefix}-{name}`의 두 요소를 하이픈을 점으로 치환해 그대로 쓴다.
- 예: `domain-{name}` 모듈의 베이스 패키지는 `{그룹}.domain.{name}`이다.
- 예: `app-{name}` 모듈의 베이스 패키지는 `{그룹}.app.{name}`이다.
- 베이스 패키지는 모듈 아티팩트명에서 기계적으로 파생한다.
- 베이스 패키지에 임의 세그먼트를 추가하지 않는다.
- 패키지명은 목적을 드러내는 간결한 소문자 단수형으로 짓는다.
- 최상위 패키지에 일반 클래스를 두지 말고 책임별 하위 패키지로 구성한다.
- 애플리케이션 진입점은 예외로 둔다.
- 각 모듈 베이스 패키지에 `package-info`를 두고 모듈 경계와 계약을 문서화한다.
- 예외·에러코드 같은 모듈 수준의 표현은 해당 모듈의 전용 패키지로 모아서 관리한다.
- 설정과 횡단 관심사는 별도 패키지로 분리해 책임을 명확히 한다.
- 테스트 코드는 대상의 패키지 구조를 그대로 반영한다.
- 테스트 전용 픽스처는 테스트 소스가 소유한다.

### 리포지토리 접근 범위

- 도메인 영속 계약은 Spring Data 인터페이스 자체다.
- 별도 포트 인터페이스를 만들지 않는다.
- Spring Data 인터페이스는 도메인 모듈의 `application/required`에 평탄하게 둔다.
- `custom/` 하위 패키지를 만들지 않는다.
- 엔티티를 반환하는 조회 메서드는 도메인 모듈 경계 밖으로 노출하지 않는다.
- 경계를 넘는 조회는 Info로 변환한다.
- 앱의 리포지토리 직접 접근은 금지한다.
- 애그리거트는 불변식·생명주기 일관성 단위로 묶는다.
- 애그리거트 외부 진입점은 루트다.
- 리포지토리는 애그리거트 루트마다 하나 둔다.
- 자식 엔티티 전용 리포지토리는 성능 예외로만 둔다.
- 성능 예외는 상한 없이 성장하는 자식 컬렉션에만 적용한다.
- 무한 성장 판정은 리뷰로 확정한다.
- 외부는 루트를 통해서만 내부 엔티티를 변경한다.
- 애그리거트 간 참조는 ID만 보관한다.
- 자식 엔티티 리포지토리의 성능 예외는 조회와 벌크에만 적용한다.
- 자식 엔티티는 부모 전체 로딩에 따른 OOM·N+1을 피하기 위해 리포지토리·QueryDSL로 직접 조회한다.
- 자식 리포지토리를 통한 벌크 `UPDATE`·`DELETE`·대용량 `INSERT`는 허용한다.
- 자식 단건 `INSERT`는 금지한다.
- 소프트삭제 엔티티는 base `JpaRepository` finder 직접 호출을 금지한다.
- 예외: `deletedAt`이 없는 엔티티는 그대로 사용한다.
- 파생 쿼리 메서드: 정적 단순 조건(등호, `And`/`Or`, 정렬, 활성 필터)에서 사용한다.
- `@Query`(JPQL): 파생 쿼리 조건이 4개 이상이거나 파생 파서가 처리하지 못할 때 사용한다.
- 조건 수는 WHERE 술어 수로 세고 정렬·소프트삭제 활성 필터는 세지 않는다.
- QueryDSL 프래그먼트: 런타임 조건 조합, 커서 페이지네이션, 복잡 동적 조인, 타입 안전 프로젝션이 필요할 때 사용한다.
- 네이티브 SQL: 방언 종속이 불가피할 때만 최후수단으로 사용한다.
- 메서드명 규칙은 → [coding-conventions](coding-conventions.md)의 메서드 네이밍을 따른다.

### 경계 원칙

- 내부 반환: 서비스와 리포지토리 간에는 엔티티를 사용한다.
- 경계 반환: 모듈 밖으로 나가는 조회는 도메인 소유 `info` record로 변환한다.
- 명령 결과: 명령은 ID 등 최소 결과만 반환한다.
- 페이지 변환: `Page<Entity>`는 경계에서 `Page<Info>`로 변환한다.
- 예외: 캐시하는 페이지 계약의 반환 형상은 → [caching](caching.md)의 캐시 값.
- OSIV off: 경계 밖 지연 로딩의 런타임 차단은 → 앱 구성.
- 예외: query 모듈의 엔티티 접근과 진화 규칙은 → query 모듈 구조.
- 진입 계약: 모듈 밖 호출은 provided 인터페이스를 경유한다.
- 구현 서비스는 모듈 밖에 노출하지 않는다.
- 요구 계약: 모듈이 요구하는 기능 계약은 `application/required`가 소유한다.
- 계약 구현: required 구현은 같은 모듈의 `adapter` 구역이 소유한다.
- 기술 계약(common 소유)의 구현은 infra가 소유한다.
- 역전 기준: 의존 역전은 구역 경계로 성립한다.
- `application`은 `adapter`를 의존하지 않는다.

### 트랜잭션 경계

- 쓰기 메서드(`Appender`·`Modifier`·`Remover`): 트랜잭션 경계를 소유한다.
- 쓰기 메서드에 `@Transactional`을 적용한다.
- 한 트랜잭션은 하나의 애그리거트만 변경한다.
- 전달 보장 기록(아웃박스)은 애그리거트 변경으로 세지 않는다.
- `Processor`의 다중 애그리거트 조율 메서드는 `@Transactional`을 선언하지 않는다.
- 다중 애그리거트 조율은 애그리거트별 쓰기 서비스 호출로 분할한다.
- 분할된 각 쓰기의 트랜잭션은 호출된 쓰기 서비스가 소유한다.
- 단일 애그리거트만 변경하는 `Processor` 메서드는 쓰기 메서드 규칙을 따른다.
- 취소 불가능한 부작용이 있는 쓰기: 부작용은 트랜잭션 밖에서 수행한다.
- 부작용의 결과 영속만 프로그램적 `TransactionTemplate`으로 감싼다.
- 프로그램적 트랜잭션 사용 조건: 커밋 실패 시 부작용이 롤백되지 않는 경우에만 사용한다.
- 프로그램적 트랜잭션 호출 조건: 호출자는 반드시 트랜잭션 밖에서 호출한다.
- 프로그램적 트랜잭션 제한: 영속은 자기 애그리거트만 변경한다.
- 이벤트 소비자의 프로그램적 트랜잭션 서비스 호출: 재전달 시 이중 실행 창이 생기므로 도메인 상태 기반 멱등 가드를 함께 둔다.
- facade: 트랜잭션을 열지 않고 도메인 provided 조립·조율만 수행한다.
- 여러 도메인 변경: 동기 트랜잭션 대신 도메인 이벤트 기반 최종일관성을 사용한다.
- 도메인 이벤트: 발행·소비 규칙은 → 크로스 도메인 상호작용.
- 조회(`Reader`): `@Transactional(readOnly = true)` 범위 안에서 Info 변환까지 완료한다.
- OSIV off: 엔티티 → Info 변환은 반드시 트랜잭션 안에서 완료한다.
- 예외: 캐시하는 조회 메서드는 트랜잭션을 위임 서비스로 분리한다 → [caching](caching.md)의 트랜잭션 분리.

### 모듈 추출 판정

- 도메인 모듈의 독립 서비스 추출은 아래 기준으로 판정한다.
- 이 섹션은 판정만 소유한다.
- 동기 결합: 컨벤션 플러그인 등록 표에서 추출한, 후보 도메인이 낀 등록 간선마다 전환 방식(원격 호출 유지·이벤트 대체·모듈 병합) 결정이 기록되어 있다.
- 앱 결합: 후보 provided를 조합하는 앱 파사드 합성에 대해 원격화 결정이 기록되어 있다.
- 결정 기록은 저장소 `docs/`의 추출 판정 문서가 소유한다.
- 이벤트: 발행·소비는 아웃박스의 브로커 전환으로 계약 불변 이식되므로 추출 차단 요소가 아니다.
- 스키마: 코드 경로는 리포지토리 접근 범위가 닫고 있음을 확인한다 → 도메인 모듈 공통.
- 스키마: 마이그레이션 DDL은 크로스 스키마 참조 금지가 닫고 있음을 확인한다 → 도메인 모듈 공통.
- query 모듈: 후보 도메인의 JPA 매핑 타입을 임포트하는 query 모듈이 없어야 한다.
  - 있으면 그 조회의 재설계가 추출에 선행한다.
- 캐시: evict 경로가 후보 모듈 안으로 닫혀 있음을 확인한다 → [caching](caching.md)의 무효화.

### 빌드가 강제하는 불변식

- 아래 규칙은 리뷰 대상이 아니라 빌드 차단 규칙이다.
- 에이전트는 코드 생성 후 이 목록으로 자기검증한다.
- 채택 팀은 각 규칙에 대응하는 강제 장치 구현 여부를 검증한다.
- 강제 장치(컨벤션 플러그인, 아키텍처 테스트, 빌드 검증 테스트, 마이그레이션 SQL 스캔 테스트)는 빌드 하네스가 소유한다.
- 이 목록에 강제 장치가 없는 본문 규칙은 리뷰로 검증한다.
- 계층 의존 방향(모듈 지도): 컨벤션 플러그인(컴파일 시점)
- 도메인 간 의존은 명시 등록한 간선만 허용: 컨벤션 플러그인(컴파일 시점)
- 도메인 간 의존 간선은 비순환: 컨벤션 플러그인(컴파일 시점)
- 도메인 모듈의 앱 표면 관심사 common 의존 금지: 아키텍처 테스트(테스트 시점)
- 도메인 모듈 구역 의존 방향(`domain`은 타 구역 비의존, `application`은 `adapter` 비의존): 아키텍처 테스트(테스트 시점)
- `adapter`의 모듈 밖 비노출: 아키텍처 테스트(테스트 시점)
- infra 타입의 모듈 밖 임포트 금지(소비는 common 계약 경유): 아키텍처 테스트(테스트 시점)
- 이벤트 모듈의 의존 제한(`common-event` 외 무의존)과 프레임워크 무의존: 컨벤션 플러그인(컴파일 시점)
- 이벤트 계약을 구현하는 record는 이벤트 모듈만 소유: 아키텍처 테스트(테스트 시점)
- 조합 계층(파사드·앱)의 이벤트 발행 금지(발행 계약 사용은 도메인 쓰기 서비스만): 아키텍처 테스트(테스트 시점)
- 실행 앱은 임베드한 도메인이 발행하는 이벤트의 소비 도메인을 함께 임베드한다(인프로세스 전달의 소비 커버리지): 빌드 검증 테스트(의존 그래프 기반, 테스트 시점)
- JPA 매핑 타입의 임포트는 소유 도메인과 query 모듈만 허용: 아키텍처 테스트(테스트 시점)
- query 모듈의 도메인 상태 변경 금지: 아키텍처 테스트(테스트 시점)
- query 모듈 시그니처의 엔티티 비노출: 아키텍처 테스트(테스트 시점)
- 모듈 밖 호출의 provided 경유(도메인 구현 서비스 주입 금지): 구현 서비스 package-private(컴파일 시점) + 아키텍처 테스트(테스트 시점)
- JPA 매핑 클래스의 모듈 밖 시그니처 노출 금지: 아키텍처 테스트(테스트 시점)
- JPA 매핑 클래스의 타입 위치 강제: 아키텍처 테스트(테스트 시점)
- apps의 리포지토리 직접 접근 금지: 아키텍처 테스트(테스트 시점)
- base `JpaRepository` finder 직접 호출 금지(소프트삭제 엔티티만): 아키텍처 테스트(테스트 시점)
- web 컨트롤러 핸들러의 `@Operation`, `@ApiResponse`, `@Parameter(description)`, request/response 타입과 컴포넌트의 `@Schema` 선언 필수: 아키텍처 테스트(테스트 시점)
- 예외: 에러 상태 집합의 정확성은 리뷰로 검증한다.
- web 컨트롤러 핸들러의 int·Integer `@RequestParam` 직접 선언 금지: 아키텍처 테스트(테스트 시점)
- 예외: 페이징 파라미터는 common-web `PaginationRequest` 사용.
- 어드민 표면은 admin 패키지, `*AdminController` 네이밍, 클래스 레벨 `@Admin`, `/api/v{n}/admin/` 프리픽스를 모두 충족해야 한다: 아키텍처 테스트(테스트 시점)
- 메서드 레벨 `@Admin` 사용 금지.
- 엔티티 영속 불변식: 아키텍처 테스트(테스트 시점)
- 엔티티 영속 불변식의 항목 목록은 → [entity-persistence](entity-persistence.md)의 강제와 리뷰가 소유한다.

- 캐시 불변식의 항목 목록은 → [caching](caching.md)의 강제와 리뷰가 소유한다.
- 캐시 저장소 스타터·클라이언트 의존은 앱 모듈만: 컨벤션 플러그인(컴파일 시점)
- 캐시 애노테이션 모듈을 임베드하는 앱의 캐시 구성 동반: 빌드 검증 테스트(의존 그래프 기반, 테스트 시점)
- 관측성 불변식: 아키텍처 테스트(테스트 시점)
- 관측성 불변식의 항목 목록은 → [observability](observability.md)의 강제와 리뷰가 소유한다.
- 외부 연동 불변식: 아키텍처 테스트(테스트 시점)
- 외부 연동 불변식의 항목 목록은 → [integration](integration.md)의 강제와 리뷰가 소유한다.
- 마이그레이션 파괴 문(`DROP`·`SET NOT NULL`)은 `-- contract` 헤더가 있는 파일만 허용: 마이그레이션 SQL 정적 스캔 테스트(테스트 시점)
- 마이그레이션 파일의 `RENAME`·크로스 스키마 참조·FK 제약(`FOREIGN KEY`·`REFERENCES`)은 전면 금지: 마이그레이션 SQL 정적 스캔 테스트(테스트 시점)
- null 계약, 포맷, 정적 분석 준수: Spotless, NullAway, Error Prone(컴파일 시점)
- 각 모듈 베이스 패키지 `package-info`의 `@NullMarked` 존재: 아키텍처 테스트(테스트 시점)
- 금지 의존성(Lombok, H2, spring-webflux) 사용 금지: 컨벤션 플러그인(컴파일 시점)

### 범위 밖

- 프로파일 체계·구성 파일 구조·시크릿 관리: 인프라·운영 구성이 소유한다.

- 보상 트랜잭션 등 코레오그래피 상세: 실물 필요 시 추가한다.
- 모듈 추출 절차(브로커 도입·이중 기록·트래픽 전환): 실물 필요 시 추가한다.
