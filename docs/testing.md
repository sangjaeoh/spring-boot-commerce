# Testing

## 언제

- 새 기능·버그 수정·리팩터의 완료 조건이 되는 테스트를 계획·작성·수정할 때.
- 검증 시나리오 목록을 만들고 합의받을 때.
- 시나리오를 어디까지 파생하고 테스트 추가를 어디서 멈출지 판정할 때.
- 테스트 레벨·목 허용 범위·픽스처 형태·테스트 도구를 결정할 때.
- 모듈 경계·의존 방향 검증은 → [architecture](architecture.md)의 아키텍처 테스트 모듈.
- 포맷·null·정적분석 게이트는 → [code-quality](code-quality.md).

## 규칙

### 시나리오 소유

- 검증 시나리오 목록의 선정은 사람이 소유한다.
- 행동을 추가·변경하는 작업은 구현 전에 시나리오 목록을 한국어 문장으로 제시해 합의받는다.
- 코드와 테스트를 같은 주체가 쓰면 의도 해석 오류가 양쪽에 상관된다. 시나리오 문장 합의가 이 상관을 끊는다.
- 목록은 필수 시나리오 축에서 파생되는 케이스를 기본 포함한다.
- 합의 후 구현과 통과는 에이전트가 소유한다.
- 합의된 문장은 `@DisplayName`과 1:1로 대응시킨다.
- 버그 수정은 재현 시나리오가 곧 목록이다.
- 리팩터는 새 시나리오를 만들지 않는다. 전후 테스트 통과가 기준이다.
- 합의 없이 시나리오를 축소·삭제하지 않는다.

### 작업 절차

- 새 행동·버그 수정은 합의된 시나리오의 테스트를 구현·수정 전에 작성한다.
- 컴파일에 필요한 최소 시그니처만 먼저 만든다. 구현 본문은 채우지 않는다.
- 테스트를 실행해 시나리오마다 기대한 단언 실패·예외로 실패함을 확인한다.
- 컴파일 오류·픽스처 오류·셋업 오류는 실패 확인으로 치지 않는다.
- 구현 전에 통과한 신규 시나리오 테스트는 단언이 행동을 구분하는지 재검토한다.
- 실패 확인 후 구현해 전체를 통과시킨다.
- 이미 존재하는 행동의 시나리오 보강 테스트는 즉시 통과가 정상이다. 사전 실패 확인을 요구하지 않는다.
- 리팩터는 이 절차를 따르지 않는다. 기준은 시나리오 소유의 리팩터 규칙을 따른다.

### 테스트 레벨

- 행동마다 검증 가능한 가장 낮은 레벨 하나를 고른다. 같은 행동을 상위 레벨에서 반복하지 않는다.

| 레벨 | 대상 | 환경 | 선택 기준 |
|---|---|---|---|
| 단위 | 엔티티 메서드·값 객체·`Validator`·순수 로직 | 컨텍스트 없는 JVM | 협력자 없이 검증 가능한 행동 |
| 영속 슬라이스 | 리포지토리·도메인 서비스·query 조회·이벤트 소비자 | `@DataJpaTest` + 실 DB | DB 왕복이 행동의 일부 |
| 웹 슬라이스 | 컨트롤러·예외 매핑·인가 게이트 | `@WebMvcTest` | HTTP 계약이 행동의 일부 |
| 통합 | 앱 경계 대표 시나리오 | `@SpringBootTest` + 실 DB | 계층 조립 자체가 검증 대상 |
| 아키텍처 | 모듈 경계·의존 방향 | → [architecture](architecture.md) | — |

- 통합 테스트는 대표 경로 소수만 둔다. 경계 케이스는 단위·슬라이스가 소유한다.
- 아키텍처 테스트는 행동 테스트와 섞지 않는다.

### 배치·네이밍

- 테스트는 대상 모듈의 `src/test`에 둔다. 패키지 미러링·픽스처 소유는 → [architecture](architecture.md)의 패키지 네이밍·배치.
- 통합 테스트는 해당 앱 모듈의 `src/test`가 소유한다. `module-tests/`에 통합 테스트 모듈을 만들지 않는다.
- 테스트 클래스는 `{대상}Test`로 이름 짓는다.
- 행동 그룹은 `@Nested` 클래스로 묶는다.
- `@DisplayName`은 조건과 결과를 담은 한국어 서술문으로 작성한다.
- 테스트 메서드명은 영어 camelCase 요약으로 짓는다. 명세는 `@DisplayName`이 소유한다.
- 도메인 용어는 용어집 표준어를 사용한다(→ [coding-conventions](coding-conventions.md)).

### 행동 명세

- 테스트는 사람이 코드 대신 읽는 행동 명세다.
- 테스트 하나는 행동 하나를 검증한다. 한 행동의 결과 단언 여러 개는 허용한다.
- 본문은 준비·실행·검증 3단을 빈 줄로 구분한다.
- given·when·then 라벨 주석은 두지 않는다. 라인 주석 규칙은 → [coding-conventions](coding-conventions.md).
- 단언 대상은 공개 계약(반환값·예외·영속 상태)이다.
- 예외는 타입과 에러코드로 단언한다. 메시지 문자열로 단언하지 않는다.
- 같은 행동의 값 경계 나열은 `@ParameterizedTest`로 표현한다.

### 필수 시나리오 축

- 아래 축에 해당하는 코드를 만들면 해당 시나리오를 목록에 포함한다.
- 축의 규칙 본문은 파생 원천 문서가 소유한다.

| 축 | 검증 시나리오 | 파생 원천 |
|---|---|---|
| 상태 전이 | 허용 전이의 성공, 전이 불가 시 도메인 예외와 에러코드 | → [entity-persistence](entity-persistence.md) |
| 생성 불변식 | `create()`·compact constructor의 검증 실패 예외 | → [coding-conventions](coding-conventions.md) |
| 값 객체 매핑 | `@Convert`·`@Embeddable` 값의 저장·재조회 왕복 일치 | → [entity-persistence](entity-persistence.md) |
| 애그리거트 일관성 | 루트 의도 메서드 경유 자식 변경의 결과, 부모 소프트삭제 시 자식 소프트삭제 전파 | → [entity-persistence](entity-persistence.md)·[architecture](architecture.md) |
| 소프트삭제 | 활성-only 조회의 삭제분 미포함, `IncludingDeleted` 조회의 포함 | → [coding-conventions](coding-conventions.md) |
| 낙관락 | `@Version` 엔티티의 경합 충돌과 409 응답 | → [entity-persistence](entity-persistence.md) |
| 에러 경로 | 경계 도달 예외의 에러코드·HTTP 상태 매핑 | → [coding-conventions](coding-conventions.md) |
| 페이지네이션 | 1-based 요청 수용, 0-based 도메인 변환, 1-based 응답 보정 | → [architecture](architecture.md) |
| 인가 | 미인증 401과 권한 부족 403의 구분 | → [architecture](architecture.md) |
| 입력 검증 | request DTO의 Bean Validation 거부 | → [coding-conventions](coding-conventions.md) |
| 이벤트 발행 | 쓰기 행동 성공 시 도메인 이벤트 발행과 페이로드, 검증 실패 시 미발행 | → [architecture](architecture.md) |
| 이벤트 소비 | 리스너의 멱등 소비(중복 이벤트에 1회 효과) | → [architecture](architecture.md) |
| 크로스 도메인 조회 | query provided 조회의 축(필터·정렬·페이지) 결과 | → [architecture](architecture.md) |

### 시나리오 충분성

- 시나리오 목록은 아래 파생 규칙으로 제안한다. 최종 선정은 시나리오 소유를 따른다.
- 행동마다 대표 성공 시나리오 하나를 포함한다.
- 행동의 계약된 실패 경로(던지는 에러코드·거부 응답)마다 실패 시나리오 하나를 포함한다.
- 계약된 실패 경로가 없는 행동은 성공 시나리오만 둔다.
- 코드가 유한하게 열거하는 계약은 전수 검증한다. 전수 열거는 `@ParameterizedTest` 하나로 묶는다.

| 유한 계약 | 전수 범위 | 레벨 |
|---|---|---|
| 상태 전이 | 의도 메서드마다 허용 상태 전부와 불가 상태 전부 | 단위 |
| 에러코드 | 행동마다 그 행동이 던지는 에러코드 전부 | 행동이 선택한 레벨 |
| Bean Validation | request DTO의 제약마다 위반 값 하나 | 단위(standalone `Validator`) |
| 인가 | 보호 핸들러마다 미인증 401·권한 부족 403, 공개 핸들러의 미인증 접근 성공 | 웹 슬라이스 |

- `@Valid` 배선은 웹 슬라이스에서 DTO당 위반 대표 하나로 검증한다.
- 어느 테스트에도 단언되지 않는 `ErrorCode` 상수는 시나리오 누락 또는 데드 코드로 보고한다.
- 열거 자체(상수 존재·값 나열)는 검증하지 않는다. 열거에 걸린 행동만 검증한다.
- 무한 입력은 계약이 구분하는 동등 클래스로 분할하고 클래스당 대표 하나를 검증한다.
- 인접 클래스의 경계는 경계값과 경계 밖 첫 값으로 검증한다. 경계값이 클래스 대표를 겸하면 별도 대표를 두지 않는다.
- 아래 표준 클래스는 계약이 해당 구분을 가질 때만 적용한다. null 계약이 소거한 클래스는 테스트하지 않는다(→ [code-quality](code-quality.md)의 NullAway).

| 입력 | 표준 클래스 |
|---|---|
| 문자열 | 유효, 빈 문자열, 공백, 길이 경계 |
| 수치·시각 | 범위 경계, 경계 밖 첫 값 |
| 컬렉션 | 빈, 1개, 복수 |
| 옵셔널(`@Nullable`) | 부재, 존재 |
| 기간·만료 | 경계 시각, 직후 |

- 시간 경계는 시각을 파라미터·주입으로 받는 경로에서 검증한다. `now()` 직접 호출 경로는 결정성의 구간 단언을 따른다.
- 조합은 픽스처 기본값에서 요인 하나만 바꾼다. 픽스처가 조합의 기준점이다.
- 요인을 동시에 바꾸는 조합 시나리오는 규칙 문서·합의 목록에 상호작용 규칙이 명시된 경우에만 둔다.
- 스레드 동시 실행 재현 테스트를 두지 않는다. 경합 계약은 낙관락 축이, 중복 계약은 이벤트 소비의 순차 중복 호출이 소유한다.
- 중지 규칙: 필수 시나리오 축·유한 계약·동등 클래스의 각 항목에 시나리오가 하나씩 대응하면 목록은 완결이다.
- 새 시나리오가 새 계약 항목·새 동등 클래스를 검증하지 않으면 추가하지 않는다.

### 영속 슬라이스

- 실 PostgreSQL 기준은 → [code-quality](code-quality.md)가 소유한다.
- 컨테이너는 싱글턴으로 재사용하고 `@ServiceConnection`으로 연결한다.
- 테스트 스키마는 Flyway 마이그레이션으로 생성한다. `ddl-auto`로 생성하지 않는다.
- 도메인 서비스는 `@DataJpaTest` + `@Import`로 실 리포지토리와 함께 검증한다.
- QueryDSL 설정은 `@Import`로 명시 로딩한다.
- required 계약 협력자(`Gateway`·`Store`·`Publisher`)는 `@MockitoBean`으로 대체한다.
- 슬라이스는 테스트 단위 트랜잭션 롤백으로 격리한다.
- 수정 행동은 flush 후 재조회로 검증한다. 관리 엔티티에 `save()`를 호출하지 않는다(→ [entity-persistence](entity-persistence.md)).
- 파생 쿼리 메서드도 이름 선언이 곧 구현이므로 실 DB로 검증한다.

### 웹 슬라이스

- `@WebMvcTest`로 컨트롤러·예외 매핑·인가 게이트를 검증한다.
- `SecurityConfig`와 메서드 시큐리티를 `@Import`해 401·403을 실제 설정으로 검증한다.
- 파사드와 주입된 provided는 `@MockitoBean`으로 대체한다.
- 응답은 상태 코드와 JSON 형상으로 단언한다.

### 목·스텁

- 목은 Mockito를 사용한다.
- 목 허용 경계는 두 곳이다.
  - required 계약(`Gateway`·`Store`·`Publisher`): 실 벤더는 테스트에서 실행 불가·비결정이다.
  - 웹 슬라이스의 파사드·주입된 provided: HTTP 계약 검증에서 조립·조회는 대상이 아니다.
- 리포지토리·도메인 서비스는 목으로 대체하지 않는다. 실 DB 슬라이스로 검증한다.
  - 도메인 내부 협력을 목으로 고정하면 테스트가 구현 세부에 결합된다.
- 소유하지 않은 타입(프레임워크·라이브러리 클래스)은 목하지 않는다.
- `verify` 상호작용 단언은 발행·전송처럼 부수효과 자체가 계약일 때만 사용한다.

### 픽스처

- `Fixture` 접미사는 → [coding-conventions](coding-conventions.md), 테스트 소스 소유는 → [architecture](architecture.md)를 따른다.
- 픽스처는 유효한 기본값 객체를 만드는 정적 팩토리다. 시나리오가 바꾸는 값만 파라미터로 받는다.
- 두 모듈 이상이 쓰는 픽스처만 소유 모듈이 `java-test-fixtures`로 방출하고, `testFixtures(...)` 의존으로 소비한다.
- 단일 모듈 픽스처는 방출하지 않는다.
- 여러 테스트가 공유하는 전역 seed 데이터를 두지 않는다. 각 테스트가 자기 데이터를 만든다.

### 결정성

- 테스트는 실행 순서에 독립이다. 다른 테스트가 만든 상태에 의존하지 않는다.
- 실패한 테스트를 재시도로 통과시키지 않는다. 비결정 원인을 제거한다.
- 시각 단언은 정확 일치 대신 구간·존재 단언을 사용한다.
- 생성 UUID는 형식과 존재만 단언한다.
- 비동기 결과는 `Thread.sleep` 대신 Awaitility로 대기한다.

### 테스트하지 않는 것

- 빌드 게이트가 강제하는 내용은 행동 테스트로 반복하지 않는다(→ [architecture](architecture.md)의 빌드가 강제하는 불변식, → [code-quality](code-quality.md)).
- 프레임워크 자체의 동작은 테스트하지 않는다. 프레임워크에 건 선언(검증 애노테이션·쿼리 메서드·매핑)이 의도와 일치하는지는 테스트한다.
- 필드를 그대로 반환하는 getter·단순 위임은 테스트하지 않는다.
- 커버리지 수치는 게이트가 아니다. 합의된 시나리오 충족이 기준이다.
- 테스트만을 위한 프로덕션 공개 표면 확장을 금지한다. 접근제한자 완화 금지는 → [coding-conventions](coding-conventions.md).
- 실패하는 테스트를 통과시키기 위해 단언을 약화하거나 삭제하지 않는다. 구현을 고친다.
- `@Disabled`는 `// TODO(#이슈번호)`와 함께만 둔다(→ [coding-conventions](coding-conventions.md)의 TODO 규칙).

### 도구

- 테스트 도구는 아래로 고정한다. 버전은 Spring Boot BOM과 `gradle/libs.versions.toml`이 소유한다(→ [code-quality](code-quality.md)의 버전).

| 역할 | 채택 |
|---|---|
| 러너 | JUnit 5 (Jupiter) |
| 단언 | AssertJ |
| 목 | Mockito |
| 테스트 DB 연결 | Testcontainers + `@ServiceConnection` |
| 비동기 대기 | Awaitility |

- Kotest·Spock은 사용하지 않는다. 빌드에 Kotlin·Groovy 툴체인이 추가된다.
- Hamcrest 단언은 작성하지 않는다. 스타터에 포함돼도 단언은 AssertJ로 통일한다.
- JUnit 4(vintage)는 사용하지 않는다.
- 커버리지 도구는 선제 도입하지 않는다. 도입 시 이 문서에서 고정한다.

### 완료 판정

- 에이전트는 테스트 작성 후 이 목록으로 자기검증한다.
- 합의된 시나리오마다 대응하는 `@DisplayName` 테스트가 있다.
- 새 행동·버그 재현 테스트는 작업 절차대로 구현·수정 전에 실패를 확인했다.
- 행동마다 성공 시나리오와 계약된 실패 경로 시나리오가 있다.
- 시나리오 충분성의 유한 계약·동등 클래스 항목마다 대응 테스트가 있다.
- `./gradlew build`가 통과한다(→ [code-quality](code-quality.md)).
- 테스트를 위한 프로덕션 변경이 없다.
- 합의 없이 축소·삭제·비활성화된 테스트가 없다.
