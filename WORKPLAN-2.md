# WORKPLAN 2차 — 기능 확장

랄프 루프가 이 파일을 진행 상태 저장소로 사용한다. 각 단계를 완료할 때마다 해당 체크박스를 `[x]`로 갱신하고 파일을 저장한다.

## 진행 방법

- 작업은 위에서부터 순서대로, 한 번에 하나만 진행한다.
- 한 이터레이션은 한 작업 안에서만 진행한다. 한 작업의 연속된 단계 여러 개를 끝내도 되지만, 작업 경계를 넘어 다음 작업을 시작하지 않는다.
- 각 작업은 아래 5단계를 순서대로 거친다.

| 단계 | 내용 | 산출물·검증 |
|---|---|---|
| 1. 설계 | `docs/architecture.md`, `docs/coding-conventions.md`, `docs/entity-persistence.md` 중 해당 문서를 로딩한다. 가정·해석 후보·트레이드오프를 명시하고, 완료 기준을 검증 가능한 테스트 목록으로 구체화한다. | `worklog/2-<번호>-<작업명>.md`에 설계 기록 |
| 2. 설계 리뷰 | 설계를 `AGENTS.md` 작업 원칙(단순함 우선, 외과적 수정, 목표 주도)과 `docs/` 규칙에 대조해 리뷰하고 지적 사항을 설계에 반영한다. "시니어 엔지니어가 이걸 과설계라 할까?"를 자문한다. | worklog 파일에 리뷰 결과·반영 내역 추가 |
| 3. 구현 | TDD로 진행한다. 완료 기준의 테스트를 먼저 작성해 실패를 확인하고, 통과하는 최소 코드를 구현한다. 기존 코드 스타일·패키지 구조를 따른다. | 테스트 전부 통과 |
| 4. 코드 리뷰 | 변경 diff를 검토한다. 바뀐 모든 줄이 이 작업으로 추적되는지, `docs/` 규칙 위반이 없는지, 고아 import·데드 코드가 없는지 확인하고 수정한다. | worklog 파일에 리뷰 결과 추가 |
| 5. 게이트·커밋 | `./gradlew build` 통과를 확인하고 작업 단위로 커밋한다(`feat: ...`). | build 성공, 커밋 완료 |

- 막히면 worklog 파일에 시도한 것과 막힌 지점을 기록하고 이터레이션을 종료한다. 같은 단계에서 3회 연속 실패하면 작업 상태에 `중단`을 표기하고 사유를 기록한 뒤 다음 작업으로 넘어간다.
- 설계 단계에서 결정이 필요한 항목은 worklog에 선택지와 결정·근거를 남긴다. 사용자에게 되묻지 않고 더 단순한 쪽을 택한다.

## 작업 목록

### 1. 토큰 갱신·로그아웃

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 리프레시 토큰 발급·갱신과 로그아웃(토큰 무효화)을 제공한다.
- 배경: 로그인만 있어 액세스 토큰 만료 시 재로그인해야 한다.
- 범위: 로그인 응답에 리프레시 토큰 추가, `POST /api/v1/auth/refresh`, `POST /api/v1/auth/logout`. 리프레시 토큰은 Redis에 TTL 저장하고 로그아웃 시 삭제한다. 갱신 시 회전(rotation) 여부, 회원당 단일/다중 세션, 액세스 토큰 블랙리스트 필요 여부는 설계에서 단순한 쪽으로 결정하고 근거를 남긴다.
- 완료 기준:
  - refresh로 새 액세스 토큰이 발급되는 테스트 통과.
  - 로그아웃 후 같은 리프레시 토큰의 refresh가 거부되는 테스트 통과.
  - 위조·만료 리프레시 토큰 거부 테스트 통과.
- 참고: `web/v1/member/AuthController.java`, common-auth `JwtTokenCodec`, infra-redis, `RateLimitConfig`(Redis 배선 참고).

### 2. 체크아웃 멱등키

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 주문 생성 계열 API(체크아웃·바로구매)에 `Idempotency-Key` 헤더를 지원해 재시도로 인한 중복 주문·중복 결제를 막는다.
- 배경: 네트워크 재시도·중복 클릭 시 같은 주문이 두 번 생성될 수 있다.
- 범위: Redis 기반으로 구현한다(DB 테이블 방식은 쓰지 않는다). 키 축은 회원+키, TTL·필수/선택 헤더 여부·같은 키 재요청 시 정책(최초 응답 재반환 vs 409)은 설계에서 결정하고 근거를 남긴다. 동시 중복 요청은 직렬화되어야 한다.
- 완료 기준:
  - 같은 키로 2회 요청 시 주문이 1건만 생성되는 테스트 통과.
  - 동시 중복 요청에서 1건만 생성되는 동시성 테스트 통과.
  - 키 없는 요청의 동작 계약(허용 또는 400 — 설계 결정대로) 테스트 통과.
- 참고: `web/v1/order/OrderController.java`(checkout·direct), infra-redis 기존 사용처.

### 3. 위시리스트

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 상품 찜 추가·삭제·목록 조회를 제공한다.
- 배경: 구매 전 보관 수단이 장바구니뿐이다.
- 범위: 새 `domain-wishlist` 모듈을 신설한다(엔티티·리포지토리·서비스·마이그레이션·settings 등록·`SchemaFlywayFactory` 등록 포함). 찜 단위는 상품(변형 아님)을 기본으로 하되 설계에서 재검토한다. 중복 찜은 멱등 처리한다. URL은 설계에서 확정한다.
- 완료 기준:
  - 추가·중복 추가(멱등)·삭제·목록 테스트 통과.
  - 새 모듈이 아키텍처 테스트(모듈 경계·의존 방향)를 통과.
- 참고: `docs/architecture.md` 새 모듈 절차, domain-cart 구조.

### 4. 상품 리뷰

- 진행: [ ] 설계 / [ ] 설계 리뷰 / [ ] 구현 / [ ] 코드 리뷰 / [ ] 게이트·커밋
- 목표: 구매확정한 상품에 별점·본문 리뷰 작성·수정·삭제·상품별 목록 조회를 제공한다.
- 배경: 구매 후 경험 표면이 없다.
- 범위: 새 `domain-review` 모듈을 신설한다. 작성 자격은 해당 상품을 구매확정한 회원으로 제한한다(주문 도메인의 구매확정 상태와 연동 — 교차 도메인 검증은 파사드에서 조율). 작성 단위(1주문라인 1리뷰 vs 1상품 1리뷰), 상품별 목록의 공개 여부, 평균 별점 노출 여부는 설계에서 결정한다.
- 완료 기준:
  - 구매확정 없는 회원의 작성이 거부되는 테스트 통과.
  - 작성 단위 중복 정책 테스트 통과.
  - 작성·수정·삭제·상품별 목록 테스트 통과.
- 참고: domain-order `FulfillmentStatus`, 작업 3에서 만든 새 모듈 절차.
